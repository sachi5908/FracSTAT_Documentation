Dynamic Rose Tab

Located at dynamic_analyzer/tabs/dynamic_rose_tab.py, the Rose Tab (implemented as the mixin class RoseTab) is one of six coordinated views inside the FracSTAT Dynamic Analyzer workspace (Map, Rose, Mohr, Stereonet, Table, and Method). Its function is to render an equal-area, strike-frequency rose diagram of the fracture population currently loaded into the workspace, with each angular petal coloured according to the resolved stress attribute selected elsewhere in the application (slip tendency, dilation tendency, fracture susceptibility, or a binary critical-stress classification).

Unlike the Static Analyzer's Rose Plot tab, which characterises fracture-network geometry alone (see static_rose_tab.py), the Dynamic Analyzer's Rose Tab is a stress-resolved diagram: it fuses the purely geometric distribution of strike orientations with the mechanical behaviour of each fracture under a user-specified in-situ stress tensor. This dual encoding — petal length for frequency (or cumulative length) and petal colour for mechanical response — allows an analyst to identify, at a glance, which strike orientations are both abundant in the dataset and mechanically favourable for slip, dilation, or Coulomb failure under the current stress field.

The module operates on the same segment population used throughout the Dynamic Analyzer: either straight-line segments digitised from an SVG trace map (with stresses resolved via calculator.py) or fracture-plane records imported from Excel/CSV (legacy Strike/Dip tables, Trend/Plunge pole tables, or focal-mechanism nodal-plane tables). The Rose Tab consumes the already-computed self.results DataFrame; it performs no independent stress calculation of its own.

1.2 Intended Audience

This guide is written for structural geologists, reservoir geomechanics engineers, and rock-mechanics researchers who use FracSTAT to evaluate the reactivation potential of natural fracture networks under an assumed or measured in-situ stress state. Familiarity with basic stress-tensor concepts (principal stresses, Mohr-Coulomb failure, Andersonian faulting theory) is assumed; the relevant formulae are nonetheless reproduced in Section 5 for reference.

2. Interface Overview

The Rose Tab occupies the full client area of the "Rose" pane in the Dynamic Analyzer's main CTkTabview. Its layout is a two-column CTkFrame grid: a fixed-width plot card on the left (column 0, expandable) containing the Matplotlib canvas, and a fixed-width, vertically scrollable settings panel on the right (column 1, 290 px).

2.1 Plot Card

  • Toolbar label — a muted caption reading "Equal-area Rose Plot" identifying the diagram type.
  • Save Image button — invokes self.save_plot_image("rose", self.rose_fig), writing a high-resolution PNG (DPI controlled by the application-wide export setting) to the project's FracSTAT_plot_exports folder.
  • Polar Matplotlib canvas — a single polar Axes (self.rose_ax) embedded via FigureCanvasTkAgg, on which the rose petals, reference circles, compass labels, σ1 line, and title are drawn.
  • Colour-bar axes — a dedicated, independently positioned Axes (self.rose_cbar_ax) reserved for the colour scale legend; its rectangle is recomputed on every redraw according to the chosen colour-bar position.

2.2 Settings Panel

The settings panel is a CTkScrollableFrame subdivided into six visually separated cards, each with a bold section header and a thin horizontal rule. Every control is wired directly to self._draw_rose(), so changes are reflected in the plot immediately (no separate "Apply" step), with the exception of colour-swatch pickers, which redraw as soon as a colour is confirmed in the picker dialog.

3. Data Input

The Rose Tab does not load, parse, or import any files itself. It is strictly a consumer of self.results, the DataFrame produced by core/calculator.py after a successful Calculate action in the sidebar, or refreshed automatically whenever the stress parameters, geometry, or dip overrides change. The columns the Rose Tab actually reads are summarised below.

ColumnTypeUsed for
StrikeDegfloat, 0–360°Base orientation of each segment/plane; folded to 0–180° before binning.
Length_mfloatOptional length weighting when Weight by = Length.
<attribute column>float or boolWhichever column self._selected_attribute() resolves to (SlipTendency[Norm], DilationTendency, FractureSusceptibility[Norm], or CriticalStressValue); used to colour each petal.

3.1 Data Provenance and Column Semantics

StrikeDeg is stored in self.results as a fully-directed 0–360° value (see models.py's wrap_360 helper) rather than pre-folded to the 0–180° fracture-orientation convention. This design choice is deliberate: preserving the full compass direction lets the Stereonet and Mohr tabs correctly place a fracture pole on the correct hemisphere once a per-segment dip override changes the plane from vertical to inclined. The Rose Tab performs its own fold (StrikeDeg % 180.0) at draw time, because a rose diagram is inherently an axial (undirected-line) statistic: a joint striking 045° and one striking 225° describe the same physical orientation and must contribute to the same petal.

Length_m originates from two different geometric pathways depending on the data source. For SVG-derived segments it is the true, scale-calibrated Euclidean segment length. For Excel-sourced data without genuine spatial coordinates (legacy fracture-plane tables, Trend/Plunge pole tables, or focal-mechanism nodal planes) Length_m is populated with zero for every row unless the source spreadsheet happened to supply an explicit Length column; see Section 3.2 for how the Rose Tab compensates for this.

3.2 Handling of Missing or Degenerate Length Data

When Weight by = Length is selected but the Length_m column is entirely absent, all-zero, non-finite, or negative, the module falls back transparently to an equal weight of 1.0 per segment — identical to the Static Analyzer's own Rose Plot tab fallback behaviour (see the code comment preserved in dynamic_rose_tab.py referencing static_angles.py). This prevents the diagram from silently rendering as blank when Excel-sourced orientation-only data (no true geometric length) is loaded, while still weighting by true length whenever that information genuinely exists.

3.3 Empty-State Handling

If self.results is an empty DataFrame (no segments loaded, or the most recent Calculate action failed validation), the Rose Tab clears both axes and colour-bar axes and renders the single caption "No data" in the workspace's muted text colour, then exits without attempting any further computation. No exception is raised and the settings panel remains fully interactive so a user can pre-configure display options before data is loaded.

5. Mathematical Background

This section defines, with explicit notation, every quantity computed or displayed by the Rose Tab. Variable names in the source code are given alongside standard geological/statistical symbols where they differ.

5.1 Strike Folding (Axial Data Convention)

A fracture strike is an axial quantity: strikes θ and θ+180° describe the identical line orientation. The module therefore folds every strike into the canonical 0–180° domain prior to binning:

θ_fold = θ_raw mod 180°

where θ_raw is the StrikeDeg column value as stored (0–360°, fully directed). When the diagram is drawn in full-rose mode (Half rose unchecked), the folded population is mirrored by adding 180° to obtain a second, duplicate copy, so that the resulting circular histogram visually fills all four quadrants while still representing purely axial statistics — the diagram is symmetric about the origin by construction, not because of any new physical information.

5.2 Weighted Angular Histogram

Let φᵢ be the strike (in radians, after folding and optional mirroring) of segment i, and wᵢ its associated weight (Section 6.2). For a chosen bin count n, the angular domain [0, Φ] — with Φ = π for a half rose or 2π for a full rose — is partitioned into n equal-width bins with edges

e_k = k·Φ/n, k = 0, 1, ..., n

The weighted count assigned to bin k is

C_k = Σ w_i, for all i such that e_k ≤ φ_i < e_(k+1)

This is computed via a single call to numpy.histogram with the weights argument set to the per-segment weight vector, which is the standard, numerically efficient implementation of a weighted circular histogram.

5.3 Equal-Area Petal Radius Transformation

A conventional rose diagram that plots petal radius directly proportional to bin frequency visually over-emphasises the dominant orientation, because the area of a circular sector grows with the square of its radius. FracSTAT follows the standard structural-geology convention of an equal-area (square-root) rose diagram, in which the plotted radius r_k of each petal is the square root of its weighted count:

r_k = √C_k

Under this transformation, the area of petal k (approximately (1/2)·r_k²·Δθ for narrow bins) becomes directly proportional to C_k, so equal areas on the page correspond to equal frequencies (or equal cumulative lengths, if length-weighted), giving a perceptually honest representation of the underlying distribution and matching the convention used by the Static Analyzer's Rose Plot tab and by widely used structural-geology software.

5.4 σ1 Azimuth Resolution

The Rose Tab supports two mutually exclusive stress-orientation conventions, mirrored from the sidebar's Orientation Mode selector (see the Method tab and dynamic_app.py's _general_stress_axes_or_none):

  • Andersonian (simplified) mode — σ1 is horizontal, its trend given directly by the Azimuth field; σ3 is horizontal and orthogonal to σ1; σ2 is vertical. The Rose Tab reads current_stress['azimuth'] (degrees from North) directly as the σ1 line orientation.
  • General 3D mode — σ1 is built from an independent trend/plunge pair. The Rose Tab reads current_stress['sigma1_trend'] as the azimuth of the σ1 line's horizontal projection; the true (non-horizontal) σ1 vector is not depicted on this two-dimensional strike-frequency diagram.

In both cases the resulting azimuth az (0–360°) is drawn as a full diametral line (both az and az+180°) through the polar origin, since a stress axis, like a fracture strike, is an undirected quantity when projected onto the rose diagram's horizontal plane.

5.5 Colour-Mapped Attribute Aggregation

Each petal's fill colour represents the mean value of the selected mechanical attribute A (one of Tₛ, Td, Sf, or the binary critical-stress flag) among all segments falling in that bin:

Ā_k = nanmean{ A_i : e_k ≤ φ_i < e_(k+1) }

If a bin contains no segments, Ā_k defaults to 0.0 rather than NaN, which prevents matplotlib colour-mapping errors for empty angular sectors. The colour scale and normalisation strategy applied to Ā_k are delegated to self.attribute_colormap_and_norm(), the same routine used by the Map and Mohr tabs, ensuring colour consistency across the whole Dynamic Analyzer workspace (see Section 5.6).

5.6 Normalisation Strategy

Continuous attributes (SlipTendency, SlipTendencyNorm, DilationTendency, FractureSusceptibility, FractureSusceptibilityNorm) are mapped through a standard linear Normalize instance. Columns whose name ends in "Norm", together with DilationTendency and CriticalStressValue, are fixed to the domain [0, 1]; all other continuous columns use the finite data-driven minimum and maximum, with a small symmetric padding applied when the data range collapses to a single value (to avoid a degenerate colour scale). The binary CriticalStressValue attribute instead uses a two-colour BoundaryNorm over {muted, failure} at the boundaries [-0.5, 0.5, 1.5], guaranteeing that 0 ("Stable") and 1 ("Critical") map unambiguously to two flat colour swatches rather than a continuous gradient, and the colour bar in this special case is manually re-ticked to read "Stable" / "Critical" instead of numeric values.

6. Algorithms

6.1 Bin Count Resolution and Sanitisation

The bin-count text is parsed with a defensive try/except around int(); a parse failure falls back silently to 36. The resulting integer is clamped to the closed interval [4, 360] via max(4, min(360, n)), guarding against pathological inputs (e.g., a bin count of 0 that would crash numpy.histogram, or an excessively large value that would render an unreadably dense diagram).

6.2 Weight Vector Construction

ModeWeight constructionFallback condition
Lengthw_i = Length_m_i (metres, or the active display unit)If Length_m is absent, or every value is non-finite/≤0, w_i = 1.0 for all i (equal weighting).
Countw_i = 1.0None — always uniform.
Equalw_i = 1.0None — identical numerical result to Count in the current implementation.

6.3 Mirroring for Full-Rose Display

When Half rose is unchecked, the strike array and its weight array are each duplicated: the mirrored strike array is np.concatenate([strikes, strikes + 180]), and the mirrored weight array is np.tile(weights, 2). The colour-attribute array is likewise duplicated with np.tile so that every mirrored strike sample still carries its correct source attribute value into the per-bin colour-averaging step.

6.4 Petal Geometry Assembly

  1. Compute bin edges over [0, Φ] for n bins; derive bin centres and widths.
  2. Compute the weighted histogram C_k over the (possibly mirrored) strike population.
  3. Compute equal-area radii r_k = √C_k.
  4. For each bin, gather the subset of attribute values whose strike falls in [e_k, e_(k+1)) and take the mean (0.0 if empty) to obtain the per-petal colour value Ā_k.
  5. Resolve (colormap, norm, is_critical) from self.attribute_colormap_and_norm(Ā, value_column).
  6. Map Ā through a ScalarMappable to obtain an RGBA colour array, one entry per bin.
  7. Draw all bins in a single ax.bar(...) call with theta = bin centres, height = r_k, width = (bin width) × petal-gap fraction, and per-bin RGBA facecolor.
python 
strikes_deg = self.results["StrikeDeg"].to_numpy(dtype=float) % 180.0
if half:
    strikes_full = strikes_deg
    weights = raw_weights
else:
    strikes_full = np.concatenate([strikes_deg, strikes_deg + 180.0])
    weights = np.tile(raw_weights, 2)

strikes = np.deg2rad(strikes_full)
bins = np.linspace(0.0, full_range, num_bins + 1)
counts, edges = np.histogram(strikes, bins=bins, weights=weights)
centers = (edges[:-1] + edges[1:]) / 2.0
widths = np.diff(edges)

# per-bin mean attribute value -> petal colour
color_values = []
for low, high in zip(edges[:-1], edges[1:]):
    mask = (strikes >= low) & (strikes < high)
    color_values.append(
        float(np.nanmean(attribute_values[mask])) if mask.any() else 0.0
    )
color_values = np.array(color_values)

Simplified excerpt of _draw_rose() showing the strike-folding, mirroring, weighted-histogram, and per-bin colour-aggregation steps.

7. Internal Workflow

This section summarises the end-to-end control flow triggered whenever the Rose Tab must be (re)drawn — on initial tab construction, whenever a setting changes, whenever new stress results are calculated, and whenever a fracture segment is selected/deselected elsewhere in the workspace.

7.1 Construction Sequence

  1. SlipTendencyApp._build_layout() calls self._build_rose_tab() once, during workspace initialisation.
  2. _build_rose_tab() calls self._rose_init_vars() to lazily create every CTk variable (using hasattr guards so repeated calls are idempotent).
  3. The plot card, toolbar, Matplotlib Figure/Axes/canvas, and dedicated colour-bar Axes are instantiated and gridded.
  4. self._build_rose_settings_panel(tab) constructs the six-card scrollable settings panel and wires every widget's command/binding to self._draw_rose (or, for text-entry fields, to the widget's <Return> and <FocusOut> events).
  5. No explicit initial draw call occurs inside _build_rose_tab(); the first paint happens the moment the shared refresh_plots()/calculate() pipeline (in dynamic_app.py) subsequently invokes self._draw_rose() once data becomes available, or via the workspace's periodic full-refresh path.

7.2 Redraw Sequence (_draw_rose)

  1. Guard clause: if self.rose_ax has not yet been created, return immediately (defensive check for calls that may race construction).
  2. Read the four background/edge/circle/σ1 colour values from their StringVars.
  3. Clear both the polar Axes and the colour-bar Axes; hide the colour-bar Axes pending recomputation; apply the outer and inner background colours to the Figure and Axes respectively.
  4. If self.results is empty, render the "No data" caption and return.
  5. Parse and clamp the bin count.
  6. Resolve the half/full-rose flag and the corresponding angular domain Φ.
  7. Build the weight vector per Section 6.2.
  8. Fold StrikeDeg into 0–180°; mirror strikes/weights into 0–360° if a full rose is requested.
  9. Compute the weighted histogram and equal-area radii.
  10. Resolve the active colour-by attribute via self._selected_attribute(); tile the attribute array to match any mirroring; compute per-bin mean attribute values.
  11. Resolve (cmap, norm, is_critical) via self.attribute_colormap_and_norm() and map to an RGBA array.
  12. Parse and clamp the petal-gap fraction; draw all petals in one ax.bar() call.
  13. Configure polar orientation (theta zero at North, clockwise direction, half/full theta limits).
  14. Parse the circle line width; compute and apply angular tick positions/labels per the selected unit, or clear tick labels if disabled.
  15. Apply boundary spine visibility/colour/width; apply internal grid visibility/colour/width; hide radial (r-axis) tick labels unconditionally.
  16. If compass labels are enabled and the rose is a full rose, annotate N/E/S/W at 1.10× the maximum petal radius.
  17. If the σ1 line is enabled and self.current_stress is non-empty, resolve the azimuth per Section 5.4 and draw the diametral reference line plus its text label.
  18. If the title is enabled, compute the auto title string (or use the custom override) and apply it, sized from the current global font-size rcParam.
  19. Parse and clamp the colour-bar length; select and apply the layout preset from Section 6.6, or hide the colour bar and expand the polar Axes to fill the card if Position = None.
  20. Call self.rose_canvas.draw_idle() to schedule an asynchronous repaint.

7.3 Cross-Tab Interactions

The Rose Tab is a passive consumer within the broader workspace: it exposes no segment-selection or picking behaviour of its own (unlike the Map, Mohr, and Stereonet tabs, which support click-to-select and cross-tab highlighting). It is, however, refreshed automatically whenever refresh_plots() is invoked following a Calculate action, a colour-scale change, a stress-parameter edit, a dip-override load, or a global font/appearance change (via _apply_workspace_font, which forces a redraw of every plot so that title and label font sizes stay in sync with the header's Font Size control).

8. Statistical Methods

8.1 Circular (Directional) Statistics Context

The rose diagram is fundamentally a graphical circular-statistics device for axial data (Mardia and Jupp, 2000; standard structural-geology practice, e.g. Davis, 2002). Because fracture strikes are axial (mod 180°) rather than directional (mod 360°) quantities, the folding and mirroring steps described in Section 5.1 are essential; failing to fold strikes before binning would artificially split a genuinely unimodal fracture set into two half-populations 180° apart.

8.2 Weighted Frequency vs. Length-Weighted Intensity

Selecting Weight by = Length converts the diagram from a simple strike-frequency histogram (each fracture contributes equally regardless of size) into a length-weighted orientation histogram, which more closely approximates the areal or volumetric fracture intensity contributed by each strike set — a metric of direct relevance to fluid-flow anisotropy and fracture-network connectivity assessments. Selecting Count or Equal instead answers a purely frequency-based question ("how many fractures strike in this direction"), which is less sensitive to a small number of unusually long fractures dominating the diagram.

8.3 Per-Bin Attribute Aggregation as a Simple Statistical Estimator

The colour value assigned to each petal, Ā_k (Section 5.5), is an unweighted arithmetic mean of the selected mechanical attribute over all segments in that bin — it is not weighted by the same length/count weighting used for the petal radius. Analysts should therefore interpret petal colour as "the typical stress-attribute value among fractures at this orientation" and petal size as "how much of the population (by count or length) occurs at this orientation"; the two encodings answer distinct statistical questions and are not meant to be multiplied together to recover a single combined score.

8.4 Handling of Empty Bins

Bins containing zero segments are assigned Ā_k = 0.0, and will typically render as either the coolest end of the active colormap or, for the binary critical-stress mode, the "Stable" colour swatch. Users comparing colour intensity across bins with very different populations (e.g., a dominant fracture set with hundreds of members versus a minor set with two or three) should cross-reference petal length before drawing mechanical conclusions from colour alone, since a small-n bin's mean attribute value is a comparatively unstable estimator.

9. Visualization Components

9.1 Polar Coordinate Convention

The Axes is configured with theta_zero_location = 'N' (0° at the top of the plot, matching geographic North) and theta_direction = -1 (clockwise increasing angle), reproducing the standard geological compass-bearing convention rather than matplotlib's mathematical default (0° at east, counter-clockwise).

9.2 Petal Rendering

All petals are drawn with a single vectorised ax.bar(theta, radius, width, bottom=0, color=..., edgecolor=..., linewidth=0.6) call rather than one bar() call per bin, which is significantly faster for large bin counts and produces a single collection of matplotlib artists that is easier to manage across repeated redraws.

9.3 Reference Geometry

  • Outer boundary spine — the polar plot's circular frame, colour/width controlled by the Background Colours card, visibility by the Reference Circles card's "Show reference circles" toggle.
  • Internal angular/radial grid — matplotlib's native polar grid, colour/width from the Reference Circles card, visibility from "Show radial grid".
  • Compass annotations — bold N/E/S/W text placed at 1.10× the maximum bar radius, shown only for full (not half) roses.
  • σ1 azimuth line — a diametral line through the origin (both az and az+180° drawn explicitly, since matplotlib does not auto-mirror a single-point plot), annotated with a "σ1" label positioned at 1.05× the maximum radius along the azimuth.

9.4 Colour Bar

The colour bar is a standard matplotlib Colorbar object bound to the ScalarMappable created from the resolved colormap and norm. For the binary CriticalStressValue metric, its ticks are explicitly overridden to the two labelled positions [0, 1] with text labels "Stable" and "Critical" rather than a numeric scale, which is essential because the underlying BoundaryNorm would otherwise present as a continuous-looking gradient despite representing a strictly binary classification.

9.5 Title

When enabled, the title reads either the user-supplied custom string, or the automatically generated string "Equal-area rose coloured by {value_title}", where value_title is the third element of the tuple returned by self._selected_attribute() (e.g., "Slip tendency", "Dilation tendency"). Title font size is drawn from matplotlib's live axes.titlesize rcParam, which is itself kept synchronised with the workspace header's Font Size control by _apply_workspace_font, so all diagrams across the Dynamic Analyzer scale their text together.

10. Advanced Features

10.1 Synchronised Colour Semantics Across Tabs

Because self.attribute_colormap_and_norm() and self._selected_attribute() are shared, application-level methods rather than Rose-Tab-local logic, changing the Map Display metric or the active colormap in the sidebar simultaneously updates the Map tab, the Rose tab, and (for the continuous metrics) the Mohr tab's colour field, guaranteeing that a given colour always represents the same numeric value everywhere in the workspace during a single analysis session.

10.2 Unit-Agnostic Length Weighting

The Length_m column already reflects the workspace's active display unit conversion pipeline (the profile width/height and unit selector in the sidebar), so switching the display unit (m, cm, mm, km) does not require any additional handling inside the Rose Tab — the weighted histogram automatically reflects lengths in whichever unit is currently active, though the colour-bar and axis labels for the rose diagram itself carry no unit text since the y-axis (radius) has no numeric meaning beyond relative area.

10.3 Interactive Colour Pickers

Every colour field in the settings panel uses the shared _StaticColorButton widget (ui_widgets.py), which opens a 40-swatch preset palette (5 shade rows × 8 hues) plus a free-form hex-code entry with a live preview swatch. Selecting or confirming a colour immediately triggers self._draw_rose() via the widget's callback, and the corresponding StringVar is kept in sync via a trace_add('write', ...) binding so the swatch button visually reflects the underlying variable's current value even if it is changed programmatically (e.g., by Reset to Defaults).

10.4 Dynamic Font Scaling

Font sizes for the title, tick labels, and colour-bar text are never hard-coded; they are read live from matplotlib's global rcParams (axes.titlesize, xtick.labelsize) at draw time, which are themselves driven by the workspace header's Font, Font Size, Bold, and Colour controls. This design keeps every plot in the Dynamic Analyzer — Map, Rose, Mohr, Stereonet — visually consistent without requiring the Rose Tab to know about the header's implementation.

12. Practical Examples

12.1 Example 1 — Identifying the Dominant, Critically-Stressed Fracture Set

  1. Load an SVG trace map or Excel orientation table and click Calculate with a representative in-situ stress tensor.
  2. In the sidebar, set Map Display to "Critical fractures".
  3. Switch to the Rose tab; set Weight by = Length and Bins = 36 for a balanced angular resolution.
  4. Observe which petal(s) are both large (abundant/long fractures at that strike) and coloured with the "Critical" swatch — these represent the strike orientations most likely to be actively slipping or on the verge of Coulomb failure under the assumed stress state.
  5. Enable Show σ1 line to visually confirm the expected ~30°–45° angular offset between the critically stressed set and the σ1 azimuth, consistent with Mohr-Coulomb theory for a typical friction coefficient of μ ≈ 0.6–0.85.

12.2 Example 2 — Comparing Frequency- vs. Length-Weighted Orientation Distributions

  1. Draw the rose with Weight by = Count to see which strike bins contain the most individual fracture segments.
  2. Redraw with Weight by = Length; note any change in the dominant petal — a strike bin with few but very long fractures may become dominant under length weighting even though it was subordinate under count weighting.
  3. Use this comparison to decide whether a proposed fracture set is genuinely volumetrically significant or is merely numerically prevalent due to digitisation of many short segments.

12.3 Example 3 — Publication-Quality Export

  1. Set the desired custom title via the Custom title field (e.g., "Well A-1 — Fracture Rose Coloured by Normalized Slip Tendency").
  2. Adjust Outer/Inner background colours, boundary width, and petal edge colour to match the target journal's figure style.
  3. Set Colour Bar → Position to Right and Length to approximately 0.6 for a compact vertical legend suitable for a two-column layout.
  4. Click Save Image; the resulting PNG is written at the application-configured export DPI to the project's FracSTAT_plot_exports directory, ready for direct inclusion in a manuscript or report.

13. Geological Interpretation

The Rose Tab's combined orientation/mechanics encoding directly supports several core geomechanical workflows in fracture-network characterisation and reservoir/rock-slope stability analysis.

13.1 Reactivation-Prone Set Identification

Under Andersonian faulting theory, fractures oriented at the Coulomb-optimal angle to σ1 (approximately 2θ_opt = arctan(1/μ) from σ1, i.e. roughly 25°–35° for typical friction coefficients) exhibit the highest ratio of shear to effective normal stress and are therefore the most prone to frictional reactivation. A petal that is both prominent (high frequency or cumulative length) and coloured toward high slip tendency or the "Critical" classification at approximately this angular offset from the displayed σ1 line is a strong candidate for active or potentially active slip under the assumed stress state.

13.2 Dilation and Fluid-Flow Pathways

When Map Display is set to Dilation tendency, petals coloured toward high Td highlight strike sets that are mechanically favoured to open (dilate) rather than shear under the current stress field — these orientations are of particular interest for identifying preferential fluid-flow pathways or high-permeability fracture corridors in naturally fractured reservoirs, since dilated fractures typically exhibit enhanced hydraulic aperture relative to shear-dominated sets.

13.3 Fracture Susceptibility as a Composite Risk Indicator

Fracture susceptibility Sf combines the normal-stress clamping effect, pore pressure, cohesion, and friction into a single margin-to-failure metric (see Section 5 of the Method tab documentation for the full formula). Colouring the rose by susceptibility allows an analyst to rank strike sets by their overall geomechanical risk in a single view, complementing the strictly binary Critical fractures classification with a continuous gradation useful for sensitivity or what-if stress-scenario comparisons.

13.4 Comparing Geometric vs. Mechanical Fabric

Because the Static Analyzer's own Rose Plot tab depicts the same fracture population's purely geometric strike distribution (uncoloured by any stress attribute), a useful diagnostic workflow is to compare the Static Analyzer's rose against the Dynamic Analyzer's stress-coloured rose for the same dataset: strike sets that are geometrically minor but mechanically critical may represent an under-sampled but geomechanically important fracture population warranting additional field verification or targeted sampling.

14. Implementation Notes

14.1 Class and File Organisation

RoseTab is implemented as a Python mixin class (no independent __init__ or base class of its own) intended solely to be combined via multiple inheritance into SlipTendencyApp alongside MapTab, MohrTab, TableTab, MethodTab, and StereonetTab. This mixin pattern is used consistently throughout FracSTAT's Dynamic and Static Analyzer workspaces to keep each tab's construction and drawing logic in its own source file while sharing a single top-level application-state object.

14.2 Variable Naming Convention

Every CustomTkinter variable owned by this tab is prefixed rose_ (e.g., rose_bins_var, rose_show_circles_var) to avoid collisions with the many other tabs' variables that coexist on the same SlipTendencyApp instance.

14.3 Defensive Parsing

All free-text numeric entries (bin count, petal gap, circle line width, σ1 line width, colour-bar length, boundary width) are parsed with a try/except ValueError pattern that falls back to a hard-coded sane default rather than raising an exception into the Tkinter event loop, ensuring the application never crashes from a malformed or empty text entry.

14.4 Unused Helper Function

The module defines a small utility, _hex_to_rgb01(hex_color), which converts a '#RRGGBB' string into a normalised (r, g, b) tuple in the [0, 1] range. This helper is not currently invoked anywhere within the drawing pipeline (all colour handling is delegated to matplotlib's native hex-colour support) and is retained in the source as a utility available for any future feature requiring raw RGB tuples.

15. Limitations

LimitationDescription
No significance testingThe Rose Tab performs no independent statistical significance testing of preferred orientations (e.g., no Rayleigh test for circular uniformity, no von Mises distribution fitting); it is a purely descriptive/exploratory visualisation tool.
Unweighted colour aggregationPer-bin colour aggregation uses a simple arithmetic mean with no weighting by segment length or count, which can make small-population bins visually as prominent in colour intensity as large, statistically robust bins; users must cross-reference petal size when interpreting colour.
σ1 line is a horizontal projection onlyThe σ1 azimuth line, in General 3D orientation mode, displays only the horizontal-projection trend of σ1 and does not indicate its plunge; for steeply plunging principal stresses the two-dimensional rose diagram cannot fully represent the three-dimensional stress geometry.
Fixed bin-count presetsBin count is restricted to a fixed dropdown list (10 through 180) rather than free numeric entry, limiting fine angular-resolution tuning beyond the provided presets, though the underlying clamp logic would technically accept any value from 4 to 360 if set programmatically.
No numeric export of binned dataNo support exists for exporting the underlying per-bin numeric table (counts, weighted sums, mean attribute values) directly from the Rose Tab; users wishing to analyse the binned data numerically must currently do so via the Table tab's raw per-segment export or by replicating the binning logic externally.
Compass labels tied to Half rose toggleThe compass N/E/S/W annotations are suppressed for half-rose diagrams, which is an intentional design choice (the half rose only spans 0–180° and W/S annotations would be geometrically meaningless in that mode), but is not configurable independently of the Half rose toggle.
Fixed binary colour paletteColour mapping for the binary Critical fractures metric always uses a fixed two-colour (muted/failure) palette regardless of the user's selected continuous colormap in the sidebar, which is intentional but may surprise users expecting their colormap choice to apply universally.
