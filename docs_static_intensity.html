Static Intensity Tab

Located at static_analyzer/tabs/static_intensity_tab.py, the Intensity & Density tab (implemented as the mixin class TraceIntensityTab) is one of nine analytical workspaces within the FracSTAT Static Analyzer module. It implements a circular scanline sampling procedure — commonly attributed to Mauldon and co‑workers (Mauldon, 1998; Mauldon et al., 2001) — for the unbiased estimation of two fundamental fracture‑network descriptors used throughout structural geology, reservoir characterization, and rock‑mass engineering: P21 (areal fracture intensity — cumulative trace length per unit sampling area, units [length]/[length]² = [length]⁻¹) and P20 (areal fracture density — number of fracture centers, approximated here by fracture endpoints, per unit sampling area, units [length]⁻²).

Unlike simple line‑scanline or box‑counting methods, the circular scanline technique is orientation‑independent: because a circle presents the same sampling window regardless of azimuth, P21 and P20 estimates computed from circular windows are not biased by preferential fracture orientation relative to the sampling geometry. This makes the method particularly well suited to anisotropic fracture networks, where a single straight scanline could over‑ or under‑sample fractures aligned sub‑parallel to it.

Within the broader FracSTAT workflow, this tab consumes the same segment‑level geometry (X1, Y1, X2, Y2, Length_m) produced by the Fracture Map tab's geometry engine, and therefore requires that a project already contain fracture trace data with real spatial coordinates — either digitized from an SVG trace map or imported from a spatially‑referenced Excel/CSV file. It complements the Connectivity tab (topological I‑Y‑X node analysis) and the Histogram/CDF tabs (length‑distribution statistics) by adding a spatially explicit measure of how fracture intensity varies across the mapped domain, rather than summarising the network as a single global statistic.

The tab produces both a spatially gridded (heatmap or kriged contour) representation of local P21/P20 and single‑value global P21/P20 statistics computed over the entire mapped area, giving the user both a map‑view understanding of intensity heterogeneity and a scalar summary suitable for reporting or comparison between datasets.

1. Purpose and Scope

The Intensity & Density tab exists to answer a question the Histogram, CDF, and Rose tabs cannot: not just how long or how oriented the fractures are, but how much fracturing exists per unit area, and how that quantity varies spatially across the mapped domain. The circular‑scanline (Mauldon) method was chosen specifically because it removes the directional sampling bias inherent in straight‑line scanlines, making the resulting P21/P20 estimates directly comparable between outcrops with different dominant fracture trends.

2. Interface Overview

The tab is built by the _build_trace_intensity_tab() method and follows the two‑column layout used throughout FracSTAT's Static Analyzer: a scrollable control sidebar on the left (approximately 280 px wide) and a plot canvas occupying the remaining space on the right. The sidebar is organised into five stacked cards, and the plot area contains a single Matplotlib figure embedded via FigureCanvasTkAgg, together with a compact toolbar.

3. Data Input Requirements

The Intensity & Density tab reads exclusively from self.results (an alias of self._static_results), the DataFrame populated by the Fracture Map tab's geometry engine after an SVG or Excel/CSV file is loaded and processed. No file loading occurs directly within this tab; the user must first load data via the Fracture Map tab's Load SVG or Load Excel buttons in the shared workspace header.

3.2 Data‑Source Compatibility

Because circular scanline sampling requires genuine spatial geometry, the tab explicitly rejects orientation‑only Excel/CSV inputs. When the active data source is Excel and the underlying Segment objects carry a non‑null _preset_strike attribute (the internal marker used by excel_loader.py for the legacy fracture‑plane format — ID/Strike/Dip — and the Trend/Plunge pole‑orientation format), the tab displays the explanatory message:

Note: "Circular scanlines require coordinate geometry. Load an SVG or a spatial Excel/CSV with X1 Y1 X2 Y2 columns."

Spatially‑referenced Excel/CSV files (the "raw segment / results" format, containing real X1/Y1/X2/Y2 columns) are fully supported and processed identically to SVG‑derived traces, since excel_loader.py's _read_raw_segments() leaves _preset_strike as None for that format, allowing the geometry checks to pass.

3.3 No‑Data State

If self.results is None or empty, the tab calls _p21_empty_state() with the message "Load SVG fracture data to run calculations," clearing all four Global Results labels to a dash and drawing only a centred text annotation on an otherwise blank axes.

5. Mathematical Background

The circular scanline method was developed to overcome a fundamental limitation of straight‑line scanlines: the number of fracture intersections recorded by a straight scanline is systematically biased against fractures that are sub‑parallel to the scanline direction (the well‑known Terzaghi orientation bias). A circular sampling window, having no preferred direction, removes this bias, provided fracture intensity is evaluated using the correct geometric estimator.

5.1 Symbol Definitions

SymbolDescriptionUnits
rRadius of a circular scan window (rCircle in the implementation).[length]
NNumber of discrete fracture‑segment/circle‑boundary intersection points found for one scan circle.count
MNumber of fracture segment endpoints ("tips") that fall strictly inside a scan circle.count
ILocal (per‑circle) estimate of P21, fracture intensity.[length]⁻¹
DLocal (per‑circle) estimate of P20, fracture density.[length]⁻²
L_totSum of all segment lengths in the mapped dataset (Σ Length_m).[length]
A_mapBounding‑box area of the mapped dataset, (x_max−x_min)×(y_max−y_min).[length]²

5.2 Circle‑Based P21 (Intensity) Estimator

For a single scan circle of radius r, the Mauldon circular‑scanline estimator of areal fracture intensity is:

I = N / (4 r)

This relationship follows from integral‑geometry (Buffon‑type) arguments relating the expected number of crossings of a fracture network with the boundary of a convex sampling window to the total trace length enclosed, and holds regardless of the internal orientation distribution of the fractures — which is precisely why the method is orientation‑unbiased. In the implementation, N corresponds to N_circle, the count of valid line‑segment/circle‑boundary root crossings (§6.2), and I corresponds to one entry of the I_grid array.

5.3 Circle‑Based P20 (Density) Estimator

The corresponding density estimator, based on counting fracture centers (approximated here by fracture‑segment endpoints falling inside the circle) is:

D = M / (π r²)

which the implementation evaluates as D = M / (2πr²) × 2 in spirit, but is coded directly as M_circle / (2 * π * rCircle**2); note that the code's normalization constant of 2πr² (rather than πr²) reflects that each fracture segment endpoint lying inside the window is counted once per endpoint (two endpoints per whole segment), so the effective per‑fracture‑center density remains consistent with the conventional P20 definition when segments are not truncated at the circle boundary.

5.4 Global (Network‑Wide) Statistics

In addition to the spatially resolved I_grid/D_grid arrays, the tab reports two single‑value statistics computed directly from the full dataset rather than by averaging the per‑circle grid, giving an estimate that is insensitive to grid resolution or circle placement:

P21_global = L_tot / A_map P20_global = n_segments / A_map

These are the classical, non‑circular definitions of areal intensity and density applied to the entire bounding box of the dataset, and are what is displayed in the Global Results card.

5.5 Scan‑Circle Packing Geometry

Given a user‑specified target circle count n (num_circles_var) and the data's bounding‑box dimensions Δx = x_max−x_min and Δy = y_max−y_min, the algorithm computes a near‑tangent, non‑overlapping rectangular grid of circles. Denoting the longer of Δx, Δy as the "primary" axis (n circles are placed along it) and the shorter as the "secondary" axis:

Δ_primary = L_primary / (n + 1) r = 0.99 × Δ_primary / 2 Δ_secondary = 2 r / 0.99 n_secondary = floor( L_secondary / Δ_secondary − 1 )

The 0.99 scale factor intentionally shrinks each circle by 1% relative to exact tangency, preventing floating‑point edge cases in which adjacent circles would otherwise touch or micro‑overlap. When Δx equals Δy exactly, both axes use n circles directly with spacing based on the smaller of the two per‑axis spacings. Circle centres are then placed at:

x_i = x_min + i × Δx_circle , i = 1 … xNumCircle y_j = y_min + j × Δy_circle , j = 1 … yNumCircle

producing an evenly spaced, inset grid of centres that never places a circle exactly on the bounding‑box edge, ensuring every circle's full circumference lies within (or at worst, extends only minimally past) the sampled domain.

6. Algorithms and Internal Workflow

_draw_trace_intensity() is the single entry point that orchestrates data validation, grid construction, the vectorised geometric intersection test, statistic computation, and Matplotlib rendering, in that order, every time any control changes. This section walks through that pipeline in the sequence it executes.

6.1 Step 1 — Validation and Grid Layout

  1. Verify self.results is non‑empty and, if the data source is Excel, that the segments carry real spatial geometry (not orientation‑only preset strike/dip data).
  2. Verify the four required columns (X1, Y1, X2, Y2, Length_m) are present.
  3. Parse num_circles_var and contour_levels_var from their text fields, applying documented fallback defaults (12 and 15 respectively) on parse failure, and writing the corrected/parsed value back into the StringVar so the sidebar reflects what was actually used.
  4. Compute the dataset bounding box (xMin, xMax, yMin, yMax) from the min/max of both endpoint columns, and derive Δx, Δy (mapArea = Δx × Δy).
  5. Apply the packing geometry of §5.5 to obtain xNumCircle, yNumCircle, rCircle, and the two per‑axis centre‑spacing increments, clamping both circle counts to a minimum of 1.
  6. Build the 1‑D centre coordinate arrays x_centers, y_centers and combine them into a 2‑D meshgrid (CX, CY), which is then flattened to CX_flat, CY_flat for the vectorised intersection step.

6.2 Step 2 — Vectorised Line‑Circle Intersection Test

Rather than looping over every (segment, circle) pair in Python, the module builds two broadcastable NumPy arrays — one of shape (n_segments, 1) for the segment endpoints and one of shape (1, n_circles) for the circle centres — so that every pairwise combination is evaluated simultaneously via NumPy broadcasting. For each segment i and circle k, the standard parametric line/circle intersection is solved:

a = (x2−x1)² + (y2−y1)² b = 2[(x1−cx)(x2−x1) + (y1−cy)(y2−y1)] c = (x1−cx)² + (y1−cy)² − r² Δ = b² − 4ac t1,2 = (−b ∓ √Δ) / (2a)

A real root t is a genuine crossing of the (infinite) circle by the segment only if 0 ≤ t ≤ 1 (i.e. the crossing lies within the finite segment, not its extension) and Δ ≥ 0. The implementation additionally guards against degenerate zero‑length segments (a = 0) by substituting a = 1 for those rows before division and separately masking them out via the boolean array valid = a > 0, so no division‑by‑zero warning is ever raised. The second root is only counted as distinct when t1 ≠ t2 (t_valid2 additionally requires t1 != t2), preventing a tangent intersection from being double‑counted.

Per scan circle, the intersection count N (summed over all segments, over the segment axis) becomes N_circle, and the two boolean "endpoint inside circle" tests d₁_sq < r² and d₂_sq < r² (squared distance from each of the segment's two endpoints to the circle centre, compared against r²) are summed to give M_circle, the endpoint‑inside‑circle count used by the density estimator.

Both N_circle and M_circle are reshaped from a flat (n_circles,) vector back into the 2‑D (yNumCircle, xNumCircle) grid shape matching CX, CY, so that I_grid and D_grid (per §5.2–§5.3) can be plotted directly with pcolormesh or used as scattered input to the kriging/interpolation step.

6.3 Step 3 — Global Statistics and Sidebar Update

total_length = Σ(Length_m), mapArea, global_P21, and global_P20 are computed per §5.4 and pushed into the four Global Results labels, alongside the resolved grid layout (xNumCircle × yNumCircle) and total scan‑circle count. The unit suffix (m, cm, mm, or km) is read live from the shared _static_map_unit_var so the displayed units always track the workspace's current display unit.

6.4 Step 4 — Rendering Dispatch

Based on plot_style_var, one of three rendering branches executes:

6.4.1 Heatmap

The active grid (I_grid or D_grid, selected by plot_mode_var) is masked wherever its value is exactly zero (np.ma.masked_where) so empty regions render as fully transparent rather than as a spurious colour at the bottom of the colour scale, and drawn with ax.pcolormesh() over cell‑edge coordinate arrays derived by expanding each centre by ±half the corresponding axis spacing.

6.4.2 Contour (Filled) / Contour (Lines Only)

Both contour styles first interpolate the sparse per‑circle grid values onto a dense 100×100 regular mesh spanning the full bounding box. Two interpolation paths are supported:

  • Ordinary Kriging (preferred) — if the pykrige package is available, an OrdinaryKriging model with a linear variogram (slope = 1.39096×10⁻⁸, nugget = 0.0) is fit to the scan‑circle centres and evaluated on the dense grid via OK.execute('grid', …), with negative kriged values clipped to zero (fracture intensity/density cannot be negative).
  • Cubic griddata fallback — if pykrige is not installed, or kriging raises an exception, scipy.interpolate.griddata(…, method='cubic', fill_value=0) is used instead, silently degrading gracefully rather than failing the redraw.

Contour levels are computed with Matplotlib's MaxNLocator when contour_levels_var parses to an integer count, or as a fixed step interval (np.arange) when the parsed value is a non‑integer float, allowing the user to request either "n bands" or "a contour every X units" behaviour from the same text field. A second, level‑filtered array with the zero level removed (plot_levels_lines) is used specifically for the isoline/label pass, so a spurious "0.000" contour is never drawn or labelled in the interior of the map, while the zero level is still retained in the filled‑colour pass (plot_levels_fill) so the colour scale correctly represents zero‑intensity regions.

For the "Contour (Filled)" style, ax.contourf() draws the shaded bands at 0.85× the layer alpha; for both contour styles, ax.contour() draws black (or user‑coloured) isolines with inline numeric labels via ax.clabel(inline=True, fontsize=8, fmt='%.3f').

6.5 Step 5 — Overlay Layers, Axis Styling, and Colour Bar

After the primary surface is drawn, the module conditionally adds: the fracture‑trace LineCollection (if Show fracture traces is checked), the individual scan‑circle outlines (if Show scan circles is checked), and the background photograph (if enabled), respecting the z‑order/opacity rules of §4.1.1. Axis limits are padded by 5% of each dimension, aspect is fixed to 'equal', axis flips are applied, grid lines and border/spine styling are applied from the Map Elements card, and finally fig.tight_layout() and canvas.draw_idle() commit the frame.

6.6 Caching and Performance Optimisation

Kriging (and, to a lesser extent, cubic griddata interpolation) is computationally the most expensive step in the pipeline, particularly for large scan‑circle grids. To avoid re‑running it on every purely cosmetic redraw (e.g. a colour change), the module maintains a single‑entry cache, self._krige_cache, keyed on a tuple of (len(df), loaded file path, plot_mode, num_circles_var) — i.e. the inputs that actually influence the interpolated surface. On each contour‑style redraw, the current key is compared against the cached key; if they match, the previously computed fine_Z array is reused directly, and kriging/griddata is skipped entirely. Any change to the underlying data, the selected statistic (P21 vs P20), or the requested circle count invalidates the cache and forces recomputation.

The opacity slider additionally uses a 100 ms Tk after() debounce (self._opacity_timer) so that dragging the slider does not trigger a full recomputation (including a potential kriging pass) on every intermediate pixel value emitted during the drag; only the final settled value (100 ms after the last movement) triggers _draw_trace_intensity().

The core intersection test itself (§6.2) is implemented purely with NumPy array broadcasting rather than nested Python loops, so its cost scales as O(n_segments × n_circles) in vectorised array operations rather than in interpreted Python iterations — a substantial performance benefit for networks with hundreds to thousands of segments sampled by tens of circles.

7. Statistical Methods

7.1 Point Estimators vs. Grid Interpolation

The tab distinguishes between two distinct statistical operations: (1) unbiased point estimation of P21/P20 at each discrete scan‑circle location using the Mauldon estimators of §5.2–§5.3, and (2) spatial interpolation between those discrete point estimates for continuous‑surface visualization (heatmap cell shading is a nearest‑value/no‑interpolation display of the raw grid, whereas the contour styles interpolate). Users should be aware that the interpolated contour surface between scan‑circle centres is a smoothed visual aid, not itself a rigorously unbiased estimator at arbitrary intermediate locations — the rigorous estimate exists only at the discrete circle centres.

7.2 Ordinary Kriging

When available, Ordinary Kriging is used for its property of exact interpolation at data points combined with a principled treatment of spatial autocorrelation via the fitted linear variogram model. A linear variogram was selected as a robust, parameter‑light default appropriate for scan‑circle sampling grids of moderate size, avoiding the need for interactive variogram model selection within the GUI. Because the variogram parameters are fixed rather than fit adaptively to each dataset, users working with strongly clustered or highly heterogeneous fracture intensity fields should treat the kriged surface as a qualitative smoothing aid rather than a geostatistically rigorous best linear unbiased predictor.

7.3 Cubic Spline Fallback

In the absence of PyKrige, scipy.interpolate.griddata with method='cubic' provides a smooth, differentiable interpolation surface. Cubic griddata does not honour the nugget/nugget‑free distinction of kriging and can produce mild over/undershoot (Gibbs‑like ringing) near sharp gradients in the underlying scan‑circle values; this is an accepted trade‑off for environments without the optional PyKrige dependency.

7.4 Global vs. Local Statistics — Interpretation Guidance

The Global Results panel's P21/P20 values are computed directly from the full dataset (total length / total segment count divided by the bounding‑box area) and therefore represent the exact areal intensity/density of the digitized network over its full extent — they are not themselves circular‑scanline estimates and carry no orientation‑bias correction, but they also carry no scan‑circle sampling error, since they use every available data point. The gridded/contoured map, by contrast, reveals spatial heterogeneity in intensity/density that the single global value necessarily conceals; a network can have a modest global P21 while containing localized high‑intensity damage zones clearly visible on the map.

9. Advanced Features

9.1 Layer Stacking with Independent Opacity

The traces‑above‑grid / grid‑above‑traces toggle, combined with the single shared opacity slider, allows the user to visually emphasise either the raw fracture network (useful when validating that the intensity surface correctly reflects the underlying digitised traces) or the interpolated intensity/density field (useful for presentation‑quality figures where the traces would otherwise visually clutter a smooth contour surface).

9.2 Background Photograph Auto‑Detection

_autodetect_p21_bg_image_path() implements a three‑tier fallback search: (1) the path already stored in this tab's own p21_bg_image_path_var, if it still exists on disk; (2) the Fracture Map tab's calibrated background image path (_static_map_bg_image_path_var), letting a single calibration performed once on the main map propagate automatically to this tab; (3) the <svg_base>_bg.png sidecar file written automatically by the Trace Map tracing tool (tracer.py) whenever a background reference photo was used during digitisation. This layered lookup means most users will never need to manually browse for an image on this tab at all.

9.3 Adaptive Contour Level Specification

The Contour Levels field accepts either an integer (interpreted as a target band count, resolved via Matplotlib's MaxNLocator to "nice" round‑number boundaries) or a decimal value (interpreted as a fixed absolute contour interval in the data's own units, e.g. entering 0.05 draws contours every 0.05 units of P21). This dual behaviour is detected automatically from whether the parsed float has a zero fractional part.

9.4 Zero‑Level Suppression on Contour Labels

Because scan circles with no fracture length or fracture endpoints correctly evaluate to I = 0 or D = 0, an unfiltered contour pass would frequently draw and label a "0.000" contour threading through every low‑intensity region of the map, adding visual noise without informative content. The implementation filters this level out of the isoline/label pass (plot_levels_lines) while retaining it in the colour‑fill pass (plot_levels_fill), so the zero‑intensity colour band is still shown, but not needlessly outlined and numerically labelled.

10. Performance Optimization

Because scan‑circle count enters the intersection test as a multiplicative factor on cost, users working with very dense fracture networks (many thousands of segments) are advised to keep the Number of scan circles control modest (10–20) unless finer spatial resolution is specifically required, since both the O(n_segments × n_circles) intersection test and, for contour styles, the kriging/interpolation step scale with the number of scan circles.

11. Practical Examples

11.1 Example 1 — Assessing Fracture‑Damage Localisation

A user has digitised a fracture trace map from an outcrop photograph showing a fault damage zone. After loading the SVG on the Fracture Map tab, they switch to Intensity & Density, select Estimated Intensity (P21), choose Grid Heatmap for an immediate, uninterpolated view of local intensity, and increase Number of scan circles to 20 to resolve the damage‑zone boundary more sharply. The heatmap reveals a clear high‑P21 corridor coincident with the mapped fault trace, while the Global Results card reports a single average P21 for the whole outcrop — illustrating why the spatial map, not the global statistic alone, is needed to identify localized damage.

11.2 Example 2 — Publication‑Quality Kriged Density Map

For a manuscript figure, the user switches Plot Mode to Estimated Density (P20), Plot Style to Contour (Filled), sets Contour Levels to 12, enables the background field photograph at 40% opacity to provide geological context, sets Layer stacking order to Grid above Traces so the smooth kriged surface is the dominant visual element, and exports the figure via Save Image at the project's configured export DPI.

11.3 Example 3 — Comparing Circular‑Scanline vs. Global Statistics

A user wants to check whether their fracture network is spatially homogeneous. They record the Global Intensity (P21) value from the Global Results card, then visually inspect the heatmap for regions markedly brighter or darker than that global value. Large, spatially coherent deviations from the global mean indicate a network that should not be treated as areally stationary for downstream flow or geomechanical modelling; near‑uniform colouring across the map supports a stationarity assumption.

12. Geological Interpretation

Fracture intensity (P21) and density (P20) are foundational inputs to discrete fracture network (DFN) modelling, fractured‑reservoir permeability estimation, rock‑mass classification (e.g. RQD‑related indices), and slope‑stability or excavation‑damage assessments. Their spatial variability, as revealed by this tab's gridded/contoured maps, has direct engineering and geological significance:

  • High‑P21 corridors frequently mark active or reactivated fault zones, relay ramps, or fold‑hinge damage zones where strain has been concentrated, and are commonly zones of enhanced permeability in naturally fractured reservoirs.
  • Elevated P20 without a correspondingly high P21 can indicate a population of short, closely spaced fractures (e.g. from hydraulic shattering or thermal cracking) rather than a few long, throughgoing structures — an important distinction for connectivity and flow‑pathway modelling.
  • Spatially uniform P21/P20 across a mapped domain supports treating the domain as statistically homogeneous for upscaling purposes, whereas strong local heterogeneity (as visualised here) motivates domain sub‑division or the use of spatially variable DFN input statistics.
  • Because the circular‑scanline estimator is explicitly designed to be orientation‑unbiased, P21/P20 values computed here are more directly comparable between outcrops or map sheets with different structural trends than values derived from straight‑line scanlines oriented arbitrarily with respect to the dominant fracture set.

Users should bear in mind that all statistics in this tab are 2‑D, trace‑map‑derived measures; they characterise the exposed (or digitised) fracture trace pattern on a mapped surface (an outcrop face, a satellite image, or a core photograph laid flat) and are not automatically equivalent to volumetric (3‑D) fracture intensity (P32) without additional geometric assumptions relating trace‑plane sampling to the true 3‑D fracture population — a conversion that FracSTAT's Static Analyzer does not perform in this tab.

14. Limitations

LimitationDescription
Endpoint‑based P20 proxyThe P20 estimator implemented here counts fracture‑segment endpoints falling inside each scan circle as a proxy for fracture centers; for very short segments relative to the circle radius, or for fracture traces that were digitised as multiple connected sub‑segments (rather than one segment per physical fracture), this can differ from a strict fracture‑center count and should be interpreted with care.
Fixed variogram modelThe fixed linear‑variogram kriging model (§7.2) is not adaptively fit to each dataset's own spatial autocorrelation structure; it functions as a general‑purpose smoothing interpolator rather than a geostatistically optimal predictor for every possible fracture network.
Indirect grid resolution controlGrid resolution is controlled indirectly, via the Number of scan circles field, which sets circle count along the shorter map dimension only — very elongated (high aspect‑ratio) datasets may produce an uneven number of circles along the two axes.
2‑D onlyThe tab operates strictly on 2‑D trace‑map geometry; no 3‑D fracture‑intensity (P32) conversion, borehole scanline correction, or subsurface/depth information is incorporated.
No orientation‑only supportOrientation‑only Excel/CSV data (legacy Strike/Dip tables or Trend/Plunge pole files with no real spatial coordinates) cannot be analysed on this tab by design, since circular scanline sampling has no meaning without real X/Y geometry.
First‑redraw costVery large fracture datasets combined with a high scan‑circle count can produce a noticeably slower first redraw (particularly with kriging enabled) before the caching mechanism of §6.6 becomes effective on subsequent cosmetic‑only redraws.
