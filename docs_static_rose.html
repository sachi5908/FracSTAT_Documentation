Static Rose Tab

Located at static_analyzer/tabs/static_rose_tab.py, the Rose Plot tab (implemented as the mixin class StaticAnglesTab) renders an equal-area, strike-frequency rose diagram — a polar histogram in which each fracture segment is binned by its trend (strike) azimuth, and the resulting frequency, optionally weighted by segment length, is displayed as a radially scaled petal. Rose diagrams have been a cornerstone visualisation in structural geology for over a century, providing an immediate quantitative impression of preferred fracture orientations, orientation clustering versus dispersion, and the presence of multiple fracture sets. Within FracSTAT, the Rose Plot tab operates purely on the geometric results table produced by the Fracture Map tab and performs no stress resolution, making it a Static Analyzer (geometry-only) tool distinct from the stress-dependent Rose tab found in the Dynamic Analyzer workspace.

1. Interface Overview

The Rose Plot tab uses a two-column layout: a fixed-width (290 px), vertically scrollable settings sidebar on the left, and an expandable polar plot canvas on the right hosting a Matplotlib polar Axes embedded via FigureCanvasTkAgg. The Figure is sized 9 × 7 inches at 100 DPI and hosts one polar Axes (static_rose_ax) plus one dedicated colorbar Axes (static_rose_cbar_ax), whose position and visibility are recomputed on every redraw according to the selected colorbar placement.

2. Data Input

2.1 Upstream Data Sources

The Rose Plot tab does not implement its own file-loading logic. It reads directly from self.results, the shared results DataFrame maintained by the Static Analyzer workspace, which is refreshed whenever the user loads a new SVG fracture-trace file, a new Excel/CSV orientation table, or changes the profile calibration. The tab automatically reflects any such change without requiring user interaction through the workspace's refresh_all_tabs() chain.

2.2 Required Columns

ColumnTypeRole in Rose Plot
StrikeDegfloat, degrees, 0–360 (fully directed)Source orientation value; folded to the 0–180° axial convention via strikes % 180.0 before binning.
Length_mfloat, project display unitsOptional per-segment length used as the histogram weight when "Weight petals by" is set to Length.

If StrikeDeg is absent, the tab shows an explanatory empty-state message on the polar Axes. If StrikeDeg is present but every value is non-finite after validity filtering, a separate "No valid strike angles found" message is shown. Both conditions are handled gracefully without exceptions.

2.3 Compatible Upstream Data Sources

All four upstream data formats supported by the Static Analyzer flow into the Rose Plot tab. SVG fracture-trace maps supply strike computed geometrically from the digitised trace endpoints, corrected for the Y-axis-from-North azimuth. Excel/CSV segment-coordinate tables carry pre-computed StrikeDeg values through unchanged. Legacy fracture-plane tables (ID, Strike, Dip) supply Strike directly, with the optional Length column honoured for length weighting. Trend/Plunge pole tables are converted internally to an equivalent Strike/Dip pair before reaching the Rose Plot tab.

Note: because StrikeDeg is stored as a fully-directed 0°–360° azimuth throughout the rest of the application (to preserve dip-direction information needed by the Stereonet and Dynamic Analyzer tabs), the Rose Plot module performs its own modulo-180° folding at draw time rather than relying on a pre-folded column.

4. Mathematical Background

4.1 Axial (Bidirectional) Orientation Data

Fracture strike is an axial quantity: a fracture trending 040° is geometrically identical to one trending 220°, since both describe the same line in the horizontal plane. The Rose Plot module respects this by folding every strike value into the half-open interval [0°, 180°) prior to any statistical operation:

strikes = StrikeDeg mod 180°

This folding is applied identically in both the FracSTAT-mode and Classic-mode drawing routines and is a prerequisite for the axial circular-mean formula in Section 6.1.

4.2 Weighting Scheme

Each observation i contributes a weight wi to its assigned azimuth bin. Two modes are supported:

w_i = L_i (Weight petals by = "Length") w_i = 1 (Weight petals by = "Count")

where Li is the segment's real-world length (Length_m). When length weighting is requested but Length_m is missing, non-finite, or non-positive for a given row, a fallback weight of 1.0 is substituted for that row rather than discarding it.

4.3 Histogram Binning

Given B bins (clamped to [4, 180]), the bin edges form a uniform partition of the 0°–180° axial domain:

e_k = k · (180° / B), k = 0, 1, …, B

Each observation's folded strike is assigned to exactly one bin via NumPy's weighted histogram, producing bin totals and midpoints:

V_k = Σ_{i : e_k ≤ strike_i < e_{k+1}} w_i m_k = (e_k + e_{k+1}) / 2

4.4 Equal-Area (Square-Root) Radius Transform

A properly constructed rose diagram requires that the visual area of each petal — not its radial length — be proportional to the quantity it represents. Because the area of a circular sector scales with the square of its radius, FracSTAT computes the plotted radius as the square root of the weighted bin total:

r_k = √V_k

This equal-area convention prevents bins with high raw frequency from visually dominating the diagram disproportionately, and ensures that comparisons of petal area between diagrams are quantitatively meaningful.

4.5 Full-Circle Mirroring

Because strike is axial, a complete rose diagram must display the same frequency distribution twice — once in the 0°–180° half and once, by reflection, in the 180°–360° half. When Half rose is unchecked, the module concatenates bin midpoints and radii with a 180° angular offset:

θ_full = { m_k } ∪ { m_k + 180° }, r_full = { r_k } ∪ { r_k }

This produces a visually symmetric diagram about the origin, matching the universal convention for axial rose diagrams in the structural-geology literature.

4.6 Colour Mapping (FracSTAT Mode)

In strike colour mode, each petal is coloured according to its bin midpoint using a fixed linear normalisation over the full axial domain, independent of the actual data range:

c_k = Colormap( m_k / 180° ), Norm: [0°, 180°] → [0, 1]

Using a fixed 0°–180° normalisation (rather than normalising to the observed min/max strike) ensures the same azimuth is always rendered in the same colour across different datasets and sessions — important when visually comparing rose diagrams from multiple outcrops or wells.

4.7 Percentage Reference Circles

When enabled, three dashed reference circles are drawn at radii that are simple fractions of the maximum plotted radius rmax:

r_k = r_max · (k / 3), k = 1, 2, 3

Because the plotted radius already embeds the square-root equal-area transform, the fraction of the maximum bin value represented by each circle is the square of the radial fraction:

Percentage_k = round( 100 · (k/3)² )

The three default circles are therefore labelled approximately 11%, 44%, and 100% of the peak weighted bin value, giving a calibrated visual scale for judging petal size without reading exact numbers from an axis.

5. Algorithms

5.1 Mode Dispatch

The top-level redraw entry point _draw_static_rose() is a two-way dispatcher that inspects static_rose_mode_var and routes to one of two independent drawing algorithms. If the Style string begins with "Classic", control passes to _draw_classic_rose(); otherwise it passes to _draw_fracSTAT_rose(). Both routines share an identical outer skeleton — clear axes, validate data, compute the weighted histogram, apply the equal-area transform, render petals, format the polar grid, set the title, configure the colorbar, update the statistics panel, and request a canvas redraw — but differ in how petal colour is derived.

5.2 FracSTAT-Mode Rendering Algorithm

StepOperation
1Clear the polar Axes and colorbar Axes; apply outer/inner background colours.
2Validate self.results is non-empty and contains StrikeDeg; otherwise show an empty-state message and abort.
3Fold strikes to [0°, 180°) via modulo 180; discard non-finite values; abort with a message if none remain.
4Resolve weights from Length_m (with 1.0 fallback for invalid entries) or uniform 1.0, depending on "Weight petals by".
5Compute bin count B (clamped [4, 180]); build B+1 uniform edges over [0°, 180°); compute weighted histogram values[] and bin midpoints.
6Abort with an empty-state message if the histogram sum is zero.
7Resolve the active Matplotlib colormap; map every bin midpoint through a fixed Normalize(0, 180) to obtain one RGBA colour per bin.
8If Half rose is enabled, use bins as-is; otherwise mirror bins, weights, and colours into the 180°–360° half.
9Apply the equal-area square-root transform; draw all petals in one ax.bar() call with per-petal colour array and zero petal gap (full angular width).
10Apply shared polar-grid formatting via _format_polar_grid.
11Set the plot title (custom or auto-generated "Segment angles (equal area), n=…").
12Build a ScalarMappable from the same colormap/normalisation and delegate colorbar placement to _setup_rose_colorbar().
13Compute and push Segments / Mean Strike / Dominant Bin / Total Length into the Statistics card via _rose_update_stats().
14Request an idle canvas redraw via draw_idle().

5.3 Classic-Mode Rendering Algorithm

The Classic-mode algorithm mirrors steps 1–6 and 8–14 of the table above exactly, but omits the colormap resolution and per-bin colour-array step. Every petal is rendered with the single solid colour held in static_rose_color_var, a 90% opacity, and a petal angular width of 0.92 × the full bin width to leave a visible gap between adjacent petals — a more traditional discretised rose-diagram appearance. Because Classic mode carries no data-dependent colour mapping, the colorbar Axes is always suppressed regardless of the selected Colorbar position, and the polar Axes expands to fill the reclaimed space.

5.4 Shared Polar Grid Formatting

StepOperation
1Fix the polar orientation convention: 0° at geographic North (set_theta_zero_location("N")), angles increasing clockwise (set_theta_direction(-1)) — the standard geological azimuth convention.
2If Half rose is active, restrict the visible angular sector to [0°, 180°] via set_thetamin/thetamax.
3Generate angular tick positions every 30°; substitute N/E/S/W labels at 0°/90°/180°/270° when cardinal labels are enabled.
4Hide radial tick labels (petal length is read qualitatively or via the optional percentage circles).
5Style the polar spine using Border Color and Border Width settings.
6If crosshairs are enabled, draw faint radial spokes at the cardinal directions from the origin to 1.03 × the maximum petal radius.
7Apply or suppress the angular/radial grid according to Show grid lines.
8If % reference circles are enabled, delegate to the module-level helper _pct_reference_circles().

5.5 Colorbar Placement Algorithm

Given the selected Colorbar position, _setup_rose_colorbar() computes two rectangles in Figure-fraction coordinates: one for the colorbar Axes and one for the repositioned polar Axes, so that the polar plot never overlaps the legend. Horizontal placements (Bottom/Top) receive tick marks every 20°; vertical placements (Left/Right) receive tick marks every 30°, reflecting the differing available label space. Selecting "None", or being in Classic mode, hides the colorbar Axes entirely and expands the polar Axes to an enlarged default footprint.

python 
def _pct_reference_circles(ax, max_r, palette, n_circles=3):
    """Draw dashed percentage reference circles on a polar axis."""
    if max_r <= 0:
        return
    theta_ring = np.linspace(0, 2 * np.pi, 361)
    for k in range(1, n_circles + 1):
        r_k = max_r * k / n_circles
        pct = int(round(100.0 * (k / n_circles) ** 2))
        ax.plot(theta_ring, np.full_like(theta_ring, r_k),
                color=palette["spine"], linewidth=0.7, linestyle="--",
                zorder=1, alpha=0.55)
        ax.text(np.deg2rad(180), r_k, f"{pct}%",
                color=palette["muted"], fontsize=8,
                ha="center", va="bottom", zorder=2)

Module-level helper function _pct_reference_circles() defined at module scope outside the class, so it requires no instance state and remains independently testable.

6. Statistical Methods

6.1 Weighted Axial (Circular) Mean

Ordinary arithmetic averaging is invalid for orientation data because of wrap-around discontinuities — for example, the arithmetic mean of 179° and 1° is 90°, which is geometrically nonsensical for near-parallel fractures straddling the 0°/180° boundary. FracSTAT instead uses the doubled-angle circular-statistics method, implemented in _static_axial_mean(strikes, weights):

S = Σ w_i · sin(2 · strike_i) C = Σ w_i · cos(2 · strike_i) strike_mean = ½ · atan2(S, C) (mod 180°)

Doubling each angle maps the 180°-periodic axial domain onto the standard 360°-periodic circular domain, where conventional vector-resultant averaging is valid; halving the resultant direction maps the answer back onto the physically meaningful 0°–180° range. The mean is always computed from the raw, unbinned observations — not the discretised histogram — so its value is independent of the chosen Number of bins.

6.2 Dominant Bin

The dominant (modal) orientation is reported as the midpoint of whichever histogram bin carries the largest weighted total: argmax_k(V_k). Unlike the axial mean, this statistic is sensitive to the chosen bin count and edge alignment, and should be interpreted as a discretised estimate of the peak of the underlying orientation distribution rather than an exact value.

7. Visualization & Figure Components

7.1 Polar Petal Bars

Petals are rendered with a single vectorised ax.bar() call in polar coordinates. Bar centre angles equal the (possibly mirrored) bin midpoints in radians; bar heights equal the equal-area radii rk; bar bottoms are fixed at zero. In FracSTAT mode the bar width equals the full bin angular width so adjacent petals visually abut one another (as in a continuous histogram). In Classic mode a 0.92 width factor introduces a visible inter-petal gap for a more traditional, discretised rose-diagram appearance.

7.2 Colour Legend / Colour Bar

In FracSTAT mode, a ScalarMappable sharing the same colormap and fixed 0°–180° Normalize instance used for the petals is passed to Matplotlib's fig.colorbar(), producing a legend labelled "Segment strike, degrees". Its position (Bottom / Top / Left / Right / None) is fully reconfigurable, and the polar Axes' bounding box is recomputed on every change to avoid overlap. The colour bar is intentionally omitted in Classic mode, since a single solid petal colour carries no scale to legend.

7.3 Cardinal and Angular Labelling

Angular tick marks are placed every 30° around the full or half circle. When cardinal labelling is enabled, the four principal directions are rendered as N / E / S / W in place of 0° / 90° / 180° / 270°, while all remaining ticks continue to show numeric degree values — a hybrid convention familiar from paper-based structural fabric diagrams.

7.4 Reference Circles and Crosshairs

Two independent optional overlays aid quantitative reading of the diagram: the percentage reference circles (Section 4.7), which calibrate petal size against the peak bin value, and the N-S/E-W crosshair spokes, which provide a lightweight visual reference grid without the density of a full polar grid. Both can be enabled or disabled independently via their respective checkboxes in the Options card.

7.5 Empty-State Handling

Three distinct conditions each trigger a dedicated styled empty-state message rendered directly on the polar Axes via _static_rose_empty_state(), rather than an uncaught exception or a blank canvas: no data loaded / StrikeDeg column absent ("Load SVG or Excel fracture data"); all strike values non-finite ("No valid strike angles found"); histogram sum is zero, for example all weights filtered out ("No plottable rose values"). In all three cases the Statistics card fields are reset to a placeholder dash.

8. Step-by-Step Example Workflows

8.1 Example 1 — Quick-Look Orientation Screening

  1. Load an SVG trace map (or Excel/CSV) via the workspace header's Load SVG or Load Excel button; the Fracture Map, Rose Plot, and all other tabs populate automatically.
  2. Open the Rose Plot tab. With default settings (18 bins, Length weighting, FracSTAT strike-colour mode, full circle) the diagram immediately reveals one or more dominant petal directions.
  3. Read the Statistics card: Mean Strike gives the length-weighted average trend; Dominant Bin gives the single most frequent azimuth class, useful for checking whether the mean is being pulled between two conjugate sets.
  4. Toggle Show % reference circles to gauge, at a glance, what fraction of the dominant petal's weighted length the secondary petals represent.

8.2 Example 2 — Comparing Count-Based and Length-Weighted Fabrics

Switching Weight petals by from Length to Count on the same loaded dataset can reveal a different dominant orientation. Count weighting treats every fracture equally regardless of length; Length weighting emphasises the orientations carrying the greatest total trace length. Running both modes and comparing the resulting Dominant Bin and Mean Strike values is a standard diagnostic for distinguishing "many small" versus "few large" fracture populations — a meaningful distinction for flow-model permeability anisotropy inputs.

8.3 Example 3 — Publication-Quality Figure Preparation

  1. Set Colorbar position to "Bottom" and choose a colour-blind-friendly colormap such as Viridis or Cividis.
  2. Disable Show grid lines for a cleaner appearance, but enable Show cardinal labels and Show % reference circles for quantitative legibility.
  3. Enter a descriptive Custom title (e.g., "Joint set orientations, Outcrop A, n = 214").
  4. Increase Border Width slightly (e.g., to 1.4) for improved visibility at reduced print size.
  5. Click Save Image to export a high-resolution PNG to the project's plot-exports folder.

9. Geological Interpretation

9.1 Reading a Rose Diagram

A single sharply peaked petal (or symmetric petal pair in the full-circle presentation) indicates a strongly unimodal fracture fabric, consistent with a single dominant regional stress orientation acting during fracture formation — such as joints formed perpendicular to a uniaxial extension direction or systematic joints related to a single tectonic episode. Two petals separated by an acute angle (commonly 40°–70°) are the classic signature of a conjugate shear-fracture pair, and their angular bisector approximates the orientation of the maximum principal compressive stress (σ₁) at the time of formation, following the Andersonian/Coulomb faulting framework used in FracSTAT's Dynamic Analyzer. Multiple broadly distributed petals of comparable magnitude suggest a polymodal or polyphase fracture history.

9.2 Significance of Length Weighting

Length-weighted rose diagrams better represent the orientation distribution of throughgoing, hydraulically or mechanically significant fractures, since long fractures typically dominate bulk rock-mass permeability and mechanical anisotropy. Count-based diagrams better represent the orientation distribution of fracture initiation events and can be more sensitive to sampling bias toward short, easily digitised features. The two diagrams together provide a complementary picture of the fracture population.

9.3 The Mean Strike as a Design Parameter

The reported Mean Strike is frequently used as an input for downstream engineering workflows — for example, orienting a horizontal wellbore trajectory to intersect the dominant fracture set at a favourable angle for stimulation, or orienting a slope-stability kinematic analysis with respect to the governing discontinuity set. Because the statistic uses the doubled-angle axial-mean method, it remains geometrically valid even when the underlying distribution straddles the 0°/180° boundary — a common pitfall when analysts instead average raw azimuths arithmetically.

9.4 Caution Regarding Sampling Bias

As with any orientation-frequency analysis derived from a two-dimensional trace map or outcrop pavement, the rose diagram is subject to the well-documented Terzaghi (1965) sampling-line bias: fractures oriented sub-parallel to the mapping direction are systematically under-sampled relative to fractures oriented perpendicular to it. Users should consult the Static Analyzer's Connectivity and Intensity & Density tabs, and consider the geometry of the original mapping traverse or outcrop exposure, when assessing whether an apparently dominant or subordinate petal reflects a genuine geological fabric or a sampling artefact.

10. Performance Optimization

10.1 Vectorised Histogram and Colour Computation

All per-observation operations — modulo folding, weight resolution, and the weighted histogram itself — are performed with NumPy array operations (np.histogram, boolean masking, np.where) rather than per-row Python loops, giving effectively linear-time performance in the number of fracture segments, which routinely numbers in the thousands to tens of thousands for digitised trace maps.

10.2 Colormap Resolution Caching

Resolving a user-facing colormap display name (e.g. "Red-Yellow-Blue") to its underlying Matplotlib identifier ("RdYlBu") is performed once inside _on_rose_cmap_change() and the result is cached in the instance attribute static_rose_cmap_resolved. Subsequent redraws triggered by unrelated setting changes read the cached identifier directly rather than re-resolving the dropdown's display string on every frame.

10.3 Idle-Draw Scheduling

The canvas is repainted via FigureCanvasTkAgg.draw_idle() rather than the blocking draw(), deferring pixel-level rendering to the next Tkinter idle cycle. This keeps rapid successive setting changes (e.g. dragging through bin-count values or a fast sequence of colour-picker changes) from stalling the UI thread with redundant full redraws.

10.4 Shared Geometry Computation

Because the Rose Plot tab performs no independent geometry recomputation — it consumes the DataFrame already computed once by the Fracture Map tab's geometry engine — reloading data or changing the profile scale triggers only a single shared O(n) geometry pass, not a duplicated per-tab recalculation. The Rose Plot tab's own per-redraw cost is limited to the histogram, colour-mapping, and rendering steps.

11. Advanced Features

11.1 Colormap Library

The Colormap dropdown draws from the same 23-entry, application-wide colour-scale catalogue used throughout FracSTAT, spanning perceptually uniform scientific colormaps (Viridis, Plasma, Inferno, Magma, Cividis), classic scales (Turbo, Jet, Parula — a bundled MATLAB-style approximation), diverging scales (Coolwarm, Spectral, Red-Yellow-Blue), cyclic scales (HSV, Twilight) well suited to angular data, and several thematic/sequential scales (Hot, Ocean, Terrain, Earth, and others).

11.2 Half-Circle vs Full-Circle Presentation

The Half rose toggle allows analysts to choose between the compact 0°–180° semicircular presentation (useful when screen or print space is limited, or when comparing several diagrams side-by-side) and the traditional full-circle mirrored presentation expected in most published structural-geology figures. Both modes use identical statistical inputs; only the angular display range and the mirroring of bins differ.

11.3 Dynamic Colorbar Repositioning

Selecting a new Colorbar position does not merely relocate the legend graphic; the polar Axes' own position rectangle is simultaneously recalculated via _setup_rose_colorbar() so that the two never overlap and the plot remains centred and appropriately sized — Bottom/Top placements reserve a horizontal strip and shrink the plot's vertical extent, while Left/Right placements reserve a vertical strip and shrink its horizontal extent.

11.4 Reset to Defaults

A Reset to Defaults button (_rose_reset_defaults) restores every sidebar control — bins, weighting mode, colorbar position, all display toggles, petal styling, and colour choices — to their factory defaults in a single action, then immediately triggers a redraw. This provides a fast recovery path after extensive manual styling experimentation.

11.5 Relationship to the Dynamic Analyzer's Rose Tab

A structurally analogous but functionally distinct Rose tab exists in the Dynamic Analyzer workspace (dynamic_rose_tab.py). That implementation supports colouring petals by a resolved stress attribute (slip tendency, dilation tendency, fracture susceptibility, or critical-stress state) rather than by strike itself, and additionally supports overlaying the σ₁ azimuth as a reference line. The two modules share the equal-area square-root radius convention and the general polar-histogram construction, but the Static Rose tab documented here performs no stress resolution and requires no stress-tensor input.

12. Limitations

LimitationDescription
Axial data onlyThe module treats fracture strike strictly as axial (180°-periodic) data. It cannot represent or distinguish directional (0°–360°, polarity-sensitive) information such as dip direction or fault polarity. Users requiring dip-direction-sensitive visualisation should consult the Static Stereonet tab.
Dominant bin sensitivityThe Dominant Bin statistic is a discretised estimate dependent on the chosen Number of bins and on the fixed bin-edge alignment starting at 0°. It is not a continuous-distribution mode estimate (e.g. from kernel density estimation) and can shift between adjacent bins for small changes in bin count on noisy or sparse datasets.
No circular dispersion measuresNo formal circular-statistics dispersion measure (e.g. circular variance, Rayleigh test for uniformity, von Mises concentration parameter) is computed or reported. Only the first-order mean direction and modal bin are provided.
Fixed percentage circle spacingThe percentage reference circles are calibrated to simple thirds of the maximum plotted radius and are not adjustable in count or fractional spacing through the current interface.
Narrow-range colour contrastColour mapping in FracSTAT mode uses a fixed 0°–180° normalisation. Datasets whose entire fabric is confined to a narrow angular range will be rendered across only a narrow slice of the chosen colormap, which can reduce visual contrast between adjacent bins.
No Terzaghi correctionThe module performs no automatic sampling-bias correction (e.g. Terzaghi weighting for scan-line orientation bias). Any such correction must be performed upstream, prior to import, if required.
Main-thread renderingAll rendering runs on the main UI thread since Matplotlib's TkAgg backend is not safe to drive concurrently. For very large datasets the redraw cost is bounded by Matplotlib's polar bar-collection rendering performance rather than by the NumPy histogram step.
