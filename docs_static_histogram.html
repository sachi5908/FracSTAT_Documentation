Static Histogram Tab

Located at static_analyzer/tabs/static_histogram_tab.py, the Histogram Tab (implemented as the class StaticLengthsTab) is the second workspace in the Static Analyzer, sitting between the Fracture Map tab — which supplies the underlying per‑segment geometry table — and the CDF tab, which performs power‑law and other distributional fitting on the same length data. The Histogram Tab is dedicated to the statistical characterisation of scalar fracture attributes: segment length, trace length, and segment orientation (strike angle), together with an integrated bivariate Cross Plot mode for examining relationships between any two derived attributes (for example, length versus aperture, or length versus strike). This guide documents every control, the maths running underneath, and how to read and interpret the resulting plots geologically.

1. Interface Overview

The Histogram Tab uses the same two‑column layout as every other Static Analyzer tab: a fixed‑width (290 px), vertically scrollable control sidebar on the left, and an expandable plotting card on the right. The sidebar is organised into a stack of titled cards — Data Settings, Options, Plot Appearance, Element Styling, Text & Axis Styling, Statistics, and, only when Cross Plot mode is active, Cross Plot Settings. The plot card contains a small toolbar with a "Save Image" button above a single embedded Matplotlib canvas.

1.2 Plot Canvas

The plot canvas is a Matplotlib Figure created with constrained_layout=True, which automatically reserves space for axis labels, titles, and legends on every redraw rather than relying on a single tight_layout() snapshot computed at build time. This matters because the user can change the plot's pixel dimensions, its legend position, or add/remove the right‑hand secondary axis (Frequency %) at any time, and the geometry needs to be recalculated each time so labels are never clipped or overlapping.

In Histogram mode the canvas hosts two overlapping axes that share the same x‑axis: a primary (left) axis showing absolute frequency (bar counts) and a secondary (right, twinned) axis showing frequency as a percentage of the total sample. In Cross Plot mode the canvas hosts a single axis showing a scatter plot of two chosen attributes, with an optional colour‑mapped third attribute and an optional ordinary‑least‑squares trend line.

2. Data Input

2.1 Upstream Data Source

The Histogram Tab does not load files itself. It reads from two attributes maintained by the parent StaticAnalyzerApp: self._static_results (preferred) and, as a fallback, self.results. Both are pandas DataFrames populated by the Fracture Map tab's geometry pipeline whenever an SVG trace map or an Excel/CSV orientation table is loaded and the profile scale (width, length, and display unit) is applied.

Two broad data modalities are supported, distinguished internally by whether the loaded segments carry a pre‑set orientation rather than pixel geometry:

  • SVG mode — each fracture trace is digitised as one or more straight segments with real pixel endpoints (X1, Y1, X2, Y2). Segment length and strike angle are derived directly from this geometry after pixel‑to‑real‑world scaling.
  • Excel/CSV mode — fracture planes are supplied as tabular Strike/Dip (optionally with Length and Aperture columns) or as Trend/Plunge pole orientations converted to Strike/Dip. Because no true spatial geometry exists in this format, segment endpoints are synthetic (placed on a unit circle purely to preserve orientation for downstream calculations), and any Length/Aperture values come directly from optional spreadsheet columns rather than being derived from geometry.

2.3 Handling of Missing or Empty Data

If no data has been loaded (both self._static_results and self.results are empty or None), both drawing routines display a centred, muted placeholder message directly on the axes — "Load SVG or Excel data to plot distributions" or "Load data first" — instead of raising an exception, and every entry in the Statistics card is reset to an em‑dash. This defensive placeholder pattern is used throughout FracSTAT so that an empty project never presents a blank or crashed plotting surface.

Note: a second guard applies once data does exist — after selecting the requested attribute and removing non‑finite values, if zero rows remain (for example, a focal‑mechanism dataset with no Segment Length/Aperture information while the default Cross Plot axes are still selected), the module shows "No valid data for the selected X/Y axes" and clears every statistics label rather than attempting to compute a minimum/maximum on an empty array.

4. Mathematical Background

4.1 Segment Length and Segment Angle Derivation

For SVG‑sourced segments, each fracture segment i has real‑world endpoint coordinates (x1, y1) and (x2, y2) after pixel‑to‑unit scaling. Two scalar attributes are used throughout the Histogram Tab:

Segment length: Li = sqrt[ (x2 − x1)² + (y2 − y1)² ] Segment angle (image-ref): θi = atan2( x2 − x1 , −(y2 − y1) ) (mod 180°)

The negation of the y‑component accounts for the fact that SVG/screen coordinate systems increase downward, whereas the geological azimuth convention (0° = up‑page / north‑referenced) increases in the opposite sense. The resulting angle is reduced modulo 180° because fracture strike is an axial (undirected line) quantity — a segment trending 040° is geometrically identical to one trending 220°.

Note: when the pre‑computed StrikeDeg column already exists in the results table (the normal case, since this angle is computed once by the Fracture Map tab's geometry pipeline and reused by every downstream tab), the Cross Plot "Segment Angle (°)" option simply reads and folds that column rather than recomputing it from raw coordinates; the raw‑geometry computation above is used only as a fallback for legacy or partially restored sessions in which the endpoint columns exist but the derived StrikeDeg column does not.

4.2 Histogram Binning

Given the filtered, positive sample {L₁, ..., Lₙ} and a user‑selected bin count nBins, the module constructs equal‑width bins spanning the sample range:

step = (Lmax − Lmin) / nBins bin centres: ck = Lmin + k·step, k = 0, 1, ..., nBins − 1 bin edges: ek = ck − step/2 (plus one trailing edge at the final centre + step/2)

Bin counts Ck are then obtained with a standard histogram accumulation (numpy.histogram) over these edges. In the degenerate case Lmin = Lmax (every observation identical, or n = 1), the module falls back to a single bin of unit width centred on that value, avoiding a division by zero in the step calculation.

Two vertical axes are always displayed together: the left axis shows the raw count Ck, and the right (twin) axis shows the same information rescaled to percentage of the total sample:

Percentage axis limit = ( max(Ck) / n ) × 100 × 1.05

The 1.05 multiplier provides a 5% headroom margin above the tallest bar so that the bar outline is never clipped against the top of the plotting frame. Because both axes are tied to the same underlying bar heights, they are always mutually consistent — the percentage axis is a pure linear rescaling of the count axis, not an independently binned distribution.

python 
n = lengths.size
nBins = self._length_bins()
lmin, lmax = lengths.min(), lengths.max()

if lmin == lmax:
    bin_edges = [lmin - 0.5, lmax + 0.5]
    counts, centres, widths = [len(lengths)], [lmin], [1.0]
else:
    step = (lmax - lmin) / nBins
    centres = np.arange(lmin, lmax + step/2, step)
    bin_edges = np.concatenate([[centres[0] - step/2], centres + step/2])
    counts, _ = np.histogram(lengths, bins=bin_edges)
    widths = np.diff(bin_edges)

ax_left.bar(centres, counts, width=widths, align="center",
            color=hist_col, edgecolor=b_edge_col, linewidth=b_edge_lw,
            label="Frequency")

ax_right = ax_left.twinx()
count_max = counts.max() if counts.max() > 0 else 1
ax_right.set_ylim(0, count_max / n * 100.0 * 1.05)

Simplified excerpt of the binning and dual‑axis logic in _draw_length_histogram(), showing the degenerate single‑bin fallback and the percentage‑axis headroom calculation described above.

4.3 Cross‑Plot Ordinary Least‑Squares Trend Line

When "Show trend line (OLS)" is enabled, the module fits a first‑degree polynomial (a straight line) to the plotted (x, y) pairs using numpy.polyfit, which solves the ordinary least‑squares normal equations for slope m and intercept b:

y ≈ m·x + b, minimising Σi ( yi − (m·xi + b) )² R² = corr(x, y)² (squared Pearson correlation coefficient)

The fitted line is drawn across the full observed x‑range [min(x), max(x)] and its legend entry reports the equation and R² to three decimal places: "OLS y = m·x + b, R² = value". Because the OLS fit and the R² statistic both assume a linear relationship and are sensitive to outliers and heteroscedasticity, they should be interpreted as a descriptive summary of linear association rather than a formal hypothesis test unless the regression assumptions (residual normality, homoscedasticity, absence of leverage points) have been independently verified.

4.4 Angle Wrapping and Axial Statistics

All angular quantities handled by the Histogram Tab (StrikeDeg, Strike (°), Trace Angle (°), Segment Angle (°)) are folded to the axial 0–180° domain via the modulo operator (value % 180.0) at the point of column resolution, consistent with the fracture‑orientation convention used throughout the Static Analyzer. No axial (circular) mean is computed by the Histogram Tab itself for angle data; users seeking a length‑weighted axial mean strike should consult the Rose Plot tab, which implements the appropriate circular‑statistics formula (double‑angle vector summation) for that purpose.

5. Algorithms

5.1 Data‑Source Selection and Attribute Resolution

The private helper _cross_col(df, axis_label) is the central attribute‑resolution routine used by Cross Plot mode. Given a human‑readable axis label (e.g. "Segment Length", "Strike (°)"), it performs the following steps:

  1. Normalise the label by stripping any parenthetical unit suffix (e.g. "(°)") and collapsing whitespace, then case‑fold it for comparison.
  2. Attempt to resolve the corresponding DataFrame column directly (e.g. StrikeDeg, Length_m, Aperture, DipDeg, ID, Trace), tolerating either the modern FracSTAT export naming or common synonyms.
  3. For "Segment Angle" and "Segment Length" specifically, fall back to reconstructing the value from raw SVG endpoint columns (X1, Y1, X2, Y2) if the derived column is absent — this occurs for older or partially restored project sessions that predate a schema change.
  4. For "Trace Angle" and "Trace Length", aggregate per‑trace values using a pandas groupby on the Trace column (mean strike per trace, or summed length per trace) rather than reading a per‑segment value directly.
  5. Return None if no resolution path succeeds, allowing the caller to attempt a further SVG‑geometry fallback or, failing that, present the "axis not available" empty state.

5.2 Edge‑Censoring Algorithm

Real fracture‑trace datasets digitised from a finite map or photograph are subject to a well‑known sampling bias: traces truncated by the edge of the mapped area have an artificially shortened observed length compared to their true (unknown) extent. The "Censor edge traces" option implements a geometric detection of this condition:

  1. Compute the bounding box of the entire loaded dataset: xmin, xmax from the union of X1/X2, and ymin, ymax from the union of Y1/Y2.
  2. Define a tolerance proportional to the extent of each axis: tol_x = max((xmax − xmin) × 10⁻⁶, 10⁻⁶) and likewise for tol_y. This scales the numerical comparison to the physical size of the map rather than using a fixed absolute tolerance, which would be either too strict (missing genuine boundary touches after floating‑point rounding) or too permissive (falsely censoring interior traces).
  3. For every segment, test whether any of its four coordinate values (X1, X2, Y1, Y2) lies within tolerance of the corresponding bounding‑box extreme, using numpy.isclose with the computed absolute tolerance.
  4. Identify the set of distinct Trace identifiers that contain at least one such boundary‑touching segment.
  5. Remove every segment belonging to any flagged trace from the plotted sample (not just the touching segment), and report the number of removed segments (Histogram modes) or removed traces (Trace‑length mode) as the "Censored" statistic.
Why a tolerance, not exact equality? An earlier exact floating‑point equality test intermittently failed to detect genuine boundary‑touching traces whenever the same nominal coordinate had been produced by two different internal code paths (the asynchronous SVG geometry worker versus the synchronous profile‑change recompute), which can round the same physical value to slightly different floats. The proportional tolerance corrects this.

5.3 SVG Geometry Reconstruction for Cross Plots

The helper _svg_cross_dataframe() rebuilds a minimal geometry table directly from the in‑memory Segment objects (bypassing the asynchronous results table) whenever that table is unavailable or incomplete for the requested axis pair. Because raw SVG segment coordinates are expressed in arbitrary drawing‑tool pixel units, this routine re‑applies the same pixel‑to‑real‑world scaling used by the main geometry pipeline:

scale_x = profile_width / (x_max − x_min) scale_y = profile_length / (y_max − y_min) X'i = (Xi − x_min) × scale_x, Y'i = (Yi − y_min) × scale_y

profile_width and profile_length are read from the Fracture Map tab's own sidebar fields (defaulting to 260 and 100 respectively if those widgets are not yet initialised). This ensures that a Cross Plot "Segment Length" computed via this fallback path is expressed in the same physical scale as the Length_m column used elsewhere in the application, rather than in unscaled pixel units.

5.4 Canvas Rebuild Algorithm (Custom Plot Sizing)

Because Matplotlib's FigureCanvasTkAgg does not support changing a live canvas's backing pixel dimensions in place without artefacts, the "Plot Area Size" controls operate through a full tear‑down‑and‑rebuild sequence, implemented in _len_rebuild_canvas(w_in, h_in, dpi):

  1. Destroy the existing Tk canvas widget.
  2. Close the existing Matplotlib Figure object via matplotlib.pyplot.close() to release its internal renderer resources and avoid a slow memory leak across repeated resizes.
  3. Construct a new Figure at the requested size in inches (converted from the requested pixel dimensions at the fixed 100 dpi used throughout FracSTAT) with constrained_layout=True, and a new single Axes.
  4. Construct a new FigureCanvasTkAgg bound to the new Figure, re‑attach the button‑release event handler used for legend‑position persistence, and grid it into the existing plot card.
  5. Invoke the drawing dispatcher to repopulate the new canvas with the current data and settings.

The width and height entry fields also drive an aspect‑ratio lock (ax.set_box_aspect(h_px / w_px)) applied after each redraw, which keeps the plotted box's aspect ratio consistent with the requested pixel dimensions even though constrained_layout is otherwise free to adjust axis position.

5.5 Draggable Legend / Trend‑Line Legend Position Persistence

When FracSTAT's global "Move Elements" editing mode is active, the Matplotlib legend on the histogram or cross‑plot axes becomes draggable. To ensure a manually dragged legend does not snap back to its default corner on the next redraw, the module records the legend's bounding‑box position — in axes‑fraction coordinates — on every mouse‑button‑release event, and reuses that stored anchor point (bbox_to_anchor with loc='lower left') on all subsequent redraws until the legend position preset in the Plot Appearance card is explicitly changed, at which point the stored anchor is cleared and the newly selected preset position takes over.

6.1 Relationship to the CDF Tab

The Histogram Tab intentionally limits itself to elementary descriptive statistics and equal‑width binning. Rigorous distributional inference — maximum‑likelihood fitting of power‑law, exponential, log‑normal, Weibull, or truncated power‑law models; Kolmogorov–Smirnov goodness‑of‑fit; information‑criterion model selection; and bootstrap confidence intervals — is the responsibility of the dedicated CDF tab, which operates on the same underlying Length_m / TraceLengthM data. Analysts requiring a defensible power‑law exponent or a formal comparison between candidate distributional models should proceed to the CDF tab rather than inferring such conclusions from the equal‑width histogram alone, since bin width and placement are well known to bias the visual appearance of heavy‑tailed distributions.

6.2 Cross‑Plot Regression Diagnostics

The R² value reported alongside the OLS trend line is the squared Pearson product‑moment correlation coefficient between the plotted x and y arrays, computed via numpy.corrcoef. It quantifies only the proportion of variance in y that is linearly associated with x for the specific pair of attributes currently selected; it does not by itself establish causation, and a low R² does not preclude a genuine non‑linear or threshold relationship (many aperture–length relationships in the fracture‑mechanics literature follow power‑law rather than linear scaling, in which case a log–log transformation prior to interpretation — not currently automated by this tab — would be more appropriate).

7. Visualization & Figure Components

7.1 Histogram Figure Components

  • Bars — one bar per bin, centred on the bin midpoint, width equal to the bin step, coloured and outlined per the Element Styling card.
  • Left y‑axis (Frequency) — absolute bin counts, labelled and coloured per the Text & Axis Styling card, with an integer‑only tick locator so fractional count ticks never appear.
  • Right y‑axis (Frequency, %) — the same bar heights rescaled to percentage of n, sharing the x‑axis via a twin axis.
  • Minimum/Maximum reference lines — full‑height vertical lines marking the sample extremes, each with its own colour and width and included in the legend.
  • Title — automatically composed from the selected mode and sample size, e.g. "Segment lengths, n=1,204", with censored‑count annotation appended when applicable; replaced verbatim by the Custom Title field when non‑blank.
  • Z‑order correction — because Matplotlib's twin axis by default stacks the new (right) axis above the original, the tab explicitly reassigns z‑order and patch visibility so the left axis (and therefore its legend) remains clickable and visible in front of the right‑axis background.

7.2 Cross‑Plot Figure Components

  • Scatter markers — either solid‑coloured or continuously colour‑mapped by the selected third attribute.
  • Colourbar — rendered on the requested side (Right/Left/Bottom/Top) only when a colour‑by attribute is active, labelled with that attribute's name.
  • OLS trend line (optional) — a single straight line spanning the observed x‑range, with a legend entry reporting the fitted slope, intercept, and R².
  • Title, axis labels, spines, and gridlines — styled identically in spirit to the histogram mode, using the same Plot Appearance controls.

7.3 Save Image

The "Save Image" toolbar button calls the shared save_plot_image() helper (implemented once in the parent StaticAnalyzerApp and reused by every tab), which writes the current Figure to a timestamped PNG file inside a per‑project "FracSTAT_plot_exports" folder at the export DPI configured in the application Settings page (default 300 dpi), using bbox_inches='tight' so that any externally positioned legend or annotation is not clipped from the saved image.

8. Step‑by‑Step Example Workflows

8.1 Example 1 — Segment‑Length Distribution from an SVG Trace Map

  1. In the Fracture Map tab, load an SVG containing digitised fracture traces and confirm the profile width/length and display unit are correctly calibrated.
  2. Switch to the Histogram Tab. The Data Settings dropdown defaults to "Segment lengths".
  3. Set "Number of bins" to a value appropriate to the sample size — a common heuristic is nBins ≈ √n or nBins ≈ 1 + log₂(n) (Sturges' rule), though the module does not enforce either automatically and leaves the choice to the analyst.
  4. Enable "Censor edge traces" if the mapped area is known to be smaller than the characteristic fracture length, to reduce truncation bias in the reported Mean and Total Length statistics.
  5. Record the Statistics card's n, Minimum, Maximum, Mean, and Total Length values for the project report, and use "Save Image" to export a publication‑ready PNG of the histogram.

8.2 Example 2 — Comparing Segment vs Trace Length

Switching the Data Settings dropdown between "Segment lengths" and "Trace lengths" on the same loaded dataset allows a direct comparison between the length distribution of individual digitised segments (influenced by how finely a curved trace was sampled during digitisation) and the length distribution of complete traces (summed across all of a trace's constituent segments). A pronounced discrepancy between the two — for example, a segment‑length histogram dominated by very short bins while the trace‑length histogram shows a broader spread — can indicate over‑segmentation of curved traces during the SVG tracing step (see the Trace Map tool's "Curve step" parameter) rather than a genuine geological signal.

8.3 Example 3 — Cross‑Plotting Length against Aperture

  1. Load an Excel/CSV fracture‑plane table containing Length (m) and Aperture (mm) columns.
  2. In the Histogram Tab, set Data Settings to "Cross Plot".
  3. Set X axis to "Segment Length" and Y axis to "Aperture (mm)" (these are the pre‑selected defaults whenever Excel/CSV data with these columns is detected).
  4. Enable "Show trend line (OLS)" to obtain a first‑order linear length–aperture scaling estimate and its R² for the loaded population; note the caveat in Section 6.2 regarding non‑linear (e.g. power‑law) length–aperture scaling relationships commonly reported in the fracture‑mechanics literature.
  5. Optionally set "Colour points by" to "Strike (°)" to visually assess whether length–aperture scaling differs systematically between orientation sets.

9. Geological Interpretation

9.1 Fracture Length Distributions

Fracture (and joint, vein, and fault) length populations in nature are widely documented to follow heavy‑tailed distributions — commonly approximated by power‑law, log‑normal, or truncated power‑law forms over some intermediate range of scales — reflecting scale‑invariant growth and coalescence processes during fracture propagation. A histogram that appears strongly right‑skewed, with most bars concentrated at short lengths and a long thin tail of rare, very long traces, is consistent with this expectation and should motivate a follow‑up analysis in the CDF tab rather than being taken as evidence of a log‑normal or exponential process from visual inspection alone, since equal‑width linear binning tends to obscure power‑law tail behaviour that is only clearly visible on logarithmic axes.

9.2 Truncation and Censoring Bias

Because any finite outcrop, borehole image, or remotely sensed map necessarily has edges, fractures that would naturally extend beyond the mapped area are recorded with an artificially shortened apparent length. The edge‑censoring feature provides a first‑order correction by removing affected traces entirely from the length statistics, trading a reduction in sample size for an improvement in the accuracy of the remaining Mean and Total Length values. Analysts working with small mapped areas relative to the expected fracture length scale should treat the uncensored Mean length statistic with particular caution, since it will systematically underestimate the true population mean.

9.3 Segment Angle Histograms and Structural Sets

The Segment angles histogram mode provides a linear‑axis (as opposed to the Rose Plot tab's polar/equal‑area) view of the strike‑angle distribution, folded to the 0–180° axial convention. Multiple distinct peaks (modes) in this histogram are typically diagnostic of multiple structural fracture sets formed under different regional or local stress orientations, or of a single set overprinted by a later, differently oriented set. Because a linear histogram does not correct for the fact that angular data "wraps around" at 0°/180°, a population whose true dominant orientation happens to straddle the 0°/180° boundary can appear artificially bimodal in this view; the Rose Plot tab's polar representation is the more reliable tool for such near‑boundary orientation populations.

9.4 Length–Aperture and Length–Attribute Cross Plots

Positive correlations between fracture length and aperture are frequently reported in the structural geology and reservoir engineering literature and are commonly interpreted in terms of linear elastic fracture mechanics scaling relationships between fracture length and displacement/aperture. The Cross Plot tool's OLS trend line and R² provide a rapid first assessment of such a relationship within a given dataset, but formal scaling‑law exponent estimation (which typically requires a log–log regression and awareness of truncation/censoring effects analogous to those discussed for length distributions) should be pursued using dedicated statistical tools outside FracSTAT if publication‑quality scaling exponents are required.

No formal distributional fittingThe module performs no maximum‑likelihood fitting, goodness‑of‑fit testing, or confidence‑interval estimation; such analysis is deferred entirely to the CDF tab.
Equal‑width binning obscures heavy tailsLinear binning can visually obscure power‑law‑like fracture length distributions, which are frequently better examined on logarithmic axes or via the CDF tab's complementary cumulative distribution plots.

11. Advanced & Related Capabilities

11.1 Legacy Column and Unit Compatibility

The attribute‑resolution logic explicitly strips a stray "Â" character sequence and collapses whitespace before matching labels, guarding against the case where the degree symbol (°) has been mis‑decoded through a legacy Windows/Tk text encoding path. It also recognises both singular and pluralised historical label variants (e.g. "Trace Angle" vs "Trace Angles"), so previously saved project sessions or exported CSVs that used an older label convention continue to resolve correctly.

11.2 Automatic Default Correction for Mixed Sessions

Because the two supported data modalities (SVG vs Excel/CSV) have different natural default Cross Plot axis pairs, the module detects and silently corrects the situation where a Cross Plot was opened before an SVG file had finished loading — in which case the widgets could still hold the Excel‑oriented default pair, Segment Length vs Aperture (mm), which has no meaning for SVG‑only data — and resets them to the SVG‑appropriate default pair, Segment Angle (°) vs Segment Length, immediately before resolving columns for that redraw.

11.3 Performance Notes

  • Constrained layout over manual tight_layout() — recomputes label/legend spacing analytically on every draw, which is more robust than a single post‑hoc pass when the user is actively changing dimensions, legend position, or the secondary percentage axis.
  • Deferred redraw via draw_idle() — schedules a repaint on the UI event loop rather than forcing an immediate synchronous render, letting several rapid successive setting changes coalesce into fewer actual repaints.
  • On‑demand Cross‑Plot geometry reconstruction — the SVG‑geometry fallback path is only invoked when the primary results table is unavailable, incomplete, or the requested axis pair is the special‑cased Segment‑Angle/Segment‑Length combination for SVG data, avoiding redundant per‑redraw recomputation for datasets with many thousands of segments.
  • Defensive parsing instead of exceptions — every user‑editable numeric or colour field is read through a safe‑parsing wrapper that falls back to the last‑known‑good default rather than propagating an error up through the redraw call stack, so an in‑progress, momentarily invalid edit never crashes the plot.

The Histogram Tab shares its underlying segment and trace length values with the CDF tab's distributional fitting, the Rose & Stereonet tabs' orientation analysis, and the Intensity & Density tab's P21/P10/P20 calculations — so nothing computed here is wasted downstream.
