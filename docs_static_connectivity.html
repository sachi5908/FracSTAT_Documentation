Static Connectivity Tab

Located at static_analyzer/tabs/static_connectivity_tab.py, the Connectivity Tab (implemented as the mixin class StaticFluidFlowTab) addresses a fundamentally different and complementary question to FracSTAT's geometry-focused tabs: how are fractures connected to one another, and what does that connectivity imply about the ability of the network to transmit fluid, accommodate strain, or compartmentalise a reservoir?

This module implements the topological node-counting and branch-classification methodology pioneered by Sanderson and co-workers (Sanderson & Nixon, 2015; Manzocchi, 2002) and widely used in reservoir characterisation, tunnelling, groundwater hydrology, and CO₂ storage studies. It operates exclusively on data carrying real map geometry — SVG-digitised trace maps or spatial Excel/CSV imports — and is the only Static Analyzer tab that performs its own expensive O(n²) geometric pre-processing step to detect pairwise segment intersections.

The module performs four closely related tasks: classifies every fracture-trace endpoint and every internal intersection into one of three topological node types (I, Y, X); splits every digitised segment at its crossing points to obtain true topological branches and classifies each branch by the connectedness of its endpoints; computes a battery of scalar connectivity statistics; and renders five complementary diagrams — a spatial node map and four ternary diagrams — that let the analyst see at a glance where the network sits on the spectrum between a poorly connected "scattered" network and a well-connected "lattice" network.

2. Data Input

2.1 Upstream Data Source

The Connectivity tab does not load files directly. It consumes the same per-segment results DataFrame (self.results) produced by the Static Analyzer's shared geometry engine after an SVG or spatial Excel/CSV file is loaded, and reacts passively whenever results change via the refresh_static_fluidflow() hook called from StaticAnalyzerApp.refresh_all_tabs().

2.2 Required Columns

ColumnDescription
TraceInteger or string identifier grouping segments belonging to the same digitised fracture trace.
SegmentInteger identifier of a segment within its trace, used to order segments when extracting trace endpoints.
X1, Y1, X2, Y2Real-world (scaled) coordinates of each segment's two endpoints, in the project's selected display unit (m, cm, mm, or km).

2.3 Data-Source Gating

Before any computation is attempted, the tab checks whether the loaded data actually carries genuine spatial geometry. SVG-digitised trace maps and spatial Excel/CSV imports are accepted. Legacy orientation-only formats (ID/Strike/Dip tables, Trend/Plunge pole tables) are explicitly rejected with an explanatory message — node counting on synthetic orientation-wheel coordinates would be meaningless without real spatial position. The rejection test inspects whether the first loaded Segment object carries a non-null _preset_strike attribute, set only by the Excel loader for orientation-only formats.

Note: when an orientation-only format is detected, all five canvases display "I-Y-X connectivity needs map geometry; load an SVG or spatial CSV/Excel" rather than raising an error.

2.4 Unit Handling

The tolerance entry field and all reported lengths and areas automatically track the project's currently selected display unit (obtained from _static_map_unit_var, defaulting to metres). The node-snap-tolerance label refreshes on every draw to show the active unit in parentheses — e.g. "Node snap tolerance (m)" or "Node snap tolerance (cm)".

4. Mathematical Background

Let S = {s₁, …, s_n} be the set of digitised segments, each defined by two endpoints (x1, y1) and (x2, y2) belonging to a parent trace.

4.1 Node Classes

Following the terminology introduced for fracture network topology (Manzocchi, 2002; Sanderson & Nixon, 2015), every distinct point where the network terminates or crosses itself is classified into exactly one of three mutually exclusive categories:

SymbolNameGeometric DefinitionGeological Meaning
IIsolated nodeA trace endpoint that does not lie within snap tolerance of any other trace or interior crossing.A fracture tip that terminates in intact rock — propagation stopped without linking to another fracture.
YAbutting nodeAn intersection point where one trace terminates against another; the intersection coincides with an endpoint of at least one of the two segments involved.One fracture terminated against an earlier through-going fracture — the classic "T" intersection recording a relative age relationship.
XCrossing nodeAn interior intersection point lying strictly within both intersecting segments, not coincident with either endpoint.Two fractures crossed each other, implying both were open/active at the time of intersection or that a younger fracture propagated straight through an older one.

4.2 Node-Level Statistics

N = N_I + N_Y + N_X (total node count) CF = (N_Y + N_X) / N (connected fraction) C̄ = (N_Y + 2·N_X) / N_B (average connections per branch)

The connected fraction CF expresses what proportion of all identified nodes actually represent a connection between two or more fractures, as opposed to a dead end. It ranges from 0 (a network of entirely isolated fractures) to 1 (every node is a connection). Because a crossing (X) node links four branch ends while an abutting (Y) node links three, the X contribution is weighted by 2 relative to Y in the connections-per-branch statistic.

4.3 Dimensionless Intensity (B₂₂)

B₂₂ = L_total² / A

B₂₂ is the square of the cumulative trace length L_total divided by the sample bounding-box area A. It is a scale-independent measure of overall fracture density that can be compared directly between outcrops of different sizes, following the P₂₂ dimensionless-intensity convention used throughout the structural-geology fracture-intensity literature.

4.4 Branch Classification

A raw digitised segment is not itself a topological branch whenever it is crossed by another segment partway along its length. The module explicitly splits every segment at each of its interior (X-type) crossings before classification. For each resulting sub-segment (branch), both endpoints are tested for proximity to any accepted X-node or Y-node position and labelled C (connected) or I (isolated):

ClassEndpoint PatternSymbolMeaning
I–Iisolated – isolatedn_iiA fully isolated strand that neither terminates against nor is crossed by any other fracture at either end.
I–Cisolated – connectedn_icA branch that dangles freely at one end but connects to the network at the other — a "dead-end spur".
C–Cconnected – connectedn_ccA branch connected to the rest of the network at both ends — the fully connected, fluid-conducting portion of the fracture set.
N_B = N_II + N_IC + N_CC

4.5 Line- and Branch-Density Ratios (C_L and C_B)

N_L = (N_I + N_Y) / 2 C_L = 2·(N_X + N_Y) / N_L N_Br = (N_I + 3·N_Y + 4·N_X) / 2 C_B = (3·N_Y + 4·N_X) / N_Br

N_L is an estimate of the number of discrete "lines" (through-going fracture traces) implied by the node population, and C_L (connections per line) expresses how many connections, on average, each such line participates in. N_Br and C_B serve as independently cross-checkable companions to the connections_per_branch property derived from the explicit branch-split procedure, providing an internal consistency check.

4.6 Ternary (Barycentric) Coordinates

Each of the four ternary diagrams plots a data point inside an equilateral triangle using standard barycentric-to-Cartesian conversion. For any three proportions p_bl, p_br, p_apex that sum to 1, the Cartesian plotting position within a unit-height triangle is:

bx = p_br + 0.5 · p_apex by = (√3 / 2) · p_apex

For the I-Y-X ternary, the vertex assignment is bottom-left = Y fraction, bottom-right = X fraction, apex = I fraction (i.e. p_i = N_I/N, p_y = N_Y/N, p_x = N_X/N). For the Branch Topology ternary, the three proportions are the I-I, I-C, and C-C branch fractions: p_ii at the apex, p_cc at bottom-right, p_ic at bottom-left.

4.7 Theoretical Random-Network Locus (Branch Topology Ternary)

The Branch Topology diagram overlays a reference curve corresponding to the theoretical relationship expected for a network where node connectivity follows a simple binomial (random) model. Parameterising by C_B ∈ [0, 2] and setting p_C = C_B / 2:

p_II = (1 − p_C)² p_CC = p_C² p_IC = 2·p_C·(1 − p_C)

The locus is plotted as a smooth blue curve with tick marks at C_B = 0.0, 0.5, 1.0, 1.5, 2.0, giving the analyst an immediate visual reference for how far the observed network departs from a purely random connectivity structure. Real fracture networks are typically displaced toward the C-C corner relative to this random-model curve.

5. Algorithms

The complete node-classification and branch-typing pipeline is implemented in core/branch_counter.py, function count_iyx_nodes(), and is called from the GUI layer only when a recompute is actually required (Section 10).

5.1 Pre-processing and Validation

If the results table is empty or missing any of the required geometry columns (Trace, Segment, X1, Y1, X2, Y2), the function immediately returns an all-zero ConnectivityResult. All coordinate columns are coerced to numeric (pandas.to_numeric, errors='coerce'); rows that fail conversion are dropped. Segments shorter than 1×10⁻¹² are discarded as degenerate. The tolerance is clamped to be non-negative (max(tolerance, 0.0)). The bounding box of surviving endpoints is computed, anchoring both the boundary-exclusion test and the B₂₂ statistic.

5.2 Pairwise Segment Intersection

For every distinct pair of segments belonging to different traces (segments sharing the same Trace identifier are never tested against each other), the module solves the standard 2-D parametric line intersection system:

P = A₁ + u₁·(A₂ − A₁) = B₁ + u₂·(B₂ − B₁) denominator = dy₂·dx₁ − dx₂·dy₁ If |denominator| < 1×10⁻¹² → segments are parallel; no intersection recorded. Otherwise solve for u₁ (ua) and u₂ (ub) algebraically. If both u₁, u₂ ∈ [0, 1] (±1×10⁻¹⁰ tolerance) → intersection accepted.

The accepted intersection point (x, y) is computed by substitution back into the first segment's parametric equation.

5.3 Classifying the Intersection: Abutment vs Crossing

Each accepted intersection is tested by _near_any_endpoint(): the intersection point is compared, using Euclidean distance and the user-specified tolerance, against all four endpoints of the two participating segments. If the intersection lies within tolerance of any one of those four endpoints, it is classified as an abutting (Y) candidate. Otherwise, provided u₁ and u₂ both lie strictly within the open interval (0, 1), it is classified as a crossing (X) candidate.

5.4 Endpoint (I-node) Extraction and Boundary Exclusion

Independently of the pairwise intersection search, the true endpoints of every trace are extracted by _trace_endpoints(): segments are grouped by Trace and sorted by Segment, and the first coordinate of the first segment and the second coordinate of the last segment in each trace are taken as the two trace-tip candidates — correctly handling multi-segment traces even though the trace is stored as a chain of individual straight segments.

If "Include boundary I-nodes" is off (the default), every candidate tip lying within tolerance of any side of the sample bounding box is discarded by _on_boundary(). This censoring correction reflects the fact that a trace exiting the mapped area is not truly an isolated fracture tip — it is an artefact of the finite digitisation window.

5.5 Point Clustering

Because floating-point intersection arithmetic and independently listed endpoint candidates can produce several near-duplicate points representing the same physical node, all raw candidate points of a given class are merged by _cluster_points() using a greedy single-pass algorithm: each new candidate point is compared, in the order encountered, against the running centroid of every cluster created so far. If it lies within tolerance of an existing cluster centroid, it is merged and the centroid is updated as the incremental running mean; otherwise a new cluster is started.

5.6 Priority-Ordered De-duplication Across Classes

After clustering, three independent point sets exist. Any Y candidate also matching an X cluster is removed via _remove_points_near() (a true crossing takes priority over a coincidental abutment). Any I candidate matching an X or Y cluster is removed (a genuine intersection always takes priority over an apparent near-miss endpoint). This produces the strict, mutually exclusive priority ordering X > Y > I, ensuring every physical location is counted exactly once in its most topologically significant category.

5.7 Branch Splitting and Classification

The final stage (_classify_branches) re-runs the pairwise intersection search, retaining only interior (X-type) crossings, and records for every original segment the ordered list of parametric u-values at which it is crossed. Each segment is split into sub-segments at these u-positions (sorted ascending), and each resulting sub-segment (branch) has both endpoints tested for membership in the final merged X∪Y node set via is_connected(). The branch is assigned to the I-I, I-C, or C-C class according to the connectivity of its two endpoints.

python 
def count_iyx_nodes(results_df, tolerance=0.1, include_boundary_i=False):
    """Main entry point: classify I/Y/X nodes and I-I/I-C/C-C branches."""
    if results_df is None or results_df.empty:
        return _empty_result()

    # Coerce coordinate columns; discard degenerate rows
    for col in ("X1", "Y1", "X2", "Y2"):
        results_df[col] = pd.to_numeric(results_df[col], errors="coerce")
    results_df = results_df.dropna(subset=["X1", "Y1", "X2", "Y2"])
    lengths = np.hypot(results_df.X2 - results_df.X1, results_df.Y2 - results_df.Y1)
    results_df = results_df[lengths > 1e-12]
    tolerance = max(tolerance, 0.0)

    # Pairwise intersection search (O(n^2), cross-trace pairs only)
    x_raw, y_raw = [], []
    for (row_i, row_j) in cross_trace_pairs(results_df):
        pt, ua, ub = _segment_intersection(row_i, row_j)
        if pt is None:
            continue
        if _near_any_endpoint(pt, row_i, row_j, tolerance):
            y_raw.append(pt)            # abutting (Y) candidate
        elif 0 < ua < 1 and 0 < ub < 1:
            x_raw.append(pt)            # crossing (X) candidate

    # Cluster, then de-duplicate with priority X > Y > I
    x_nodes = _cluster_points(x_raw, tolerance)
    y_nodes  = _cluster_points(
        _remove_points_near(y_raw, x_nodes, tolerance), tolerance)
    i_tips   = _trace_endpoints(results_df)
    if not include_boundary_i:
        i_tips = [p for p in i_tips if not _on_boundary(p, results_df, tolerance)]
    i_nodes  = _cluster_points(
        _remove_points_near(i_tips, x_nodes + y_nodes, tolerance), tolerance)

    # Branch split and classify into I-I / I-C / C-C
    n_ii, n_ic, n_cc = _classify_branches(
        results_df, x_nodes, y_nodes, tolerance)

    return ConnectivityResult(
        i_nodes=np.array(i_nodes), y_nodes=np.array(y_nodes),
        x_nodes=np.array(x_nodes), n_ii=n_ii, n_ic=n_ic, n_cc=n_cc,
        total_length=lengths.sum(),
        sample_area=_bounding_box_area(results_df),
    )

5.8 Algorithm Complexity Summary

StageComplexityNotes
Pairwise intersection search (first pass)O(n²)n = number of segments; every cross-trace pair is tested once.
Point clusteringO(k²) per classk = raw candidate count for that node class, typically ≪ n².
Boundary exclusionO(m)m = number of trace-tip candidates (2 per trace).
Branch split & classify (second pass)O(n²)Repeats the interior-crossing search to build the per-segment split table.

The dominant cost is quadratic in the number of segments. For very dense SVG traces (many thousands of short curve-sampled segments), this can become the single most expensive computation in the entire Static Analyzer workspace.

6. Statistical Methods

Unlike the Histogram or CDF tabs, the Connectivity module does not perform inferential statistics. Its statistical content is entirely descriptive and topological: it counts, classifies, and ratios discrete network elements. All reported quantities are deterministic functions of the digitised geometry and the two user-controlled parameters (snap tolerance, boundary-I-node inclusion). No sampling error or estimator variance is associated with these numbers in the classical statistical sense.

QuantityTypeRangeInterpretation
N_I, N_Y, N_XCount (integer)≥ 0Absolute abundance of each node class.
N (total nodes)Count (integer)≥ 0Sum of all node classes.
Connected Fraction CFProportion[0, 1]Higher = more of the network participates in genuine connections rather than dead ends.
Connections / Branch C̄Ratio (dimensionless)≥ 0Average branch-end connectivity, weighted 1× for Y and 2× for X.
B₂₂Dimensionless intensity≥ 0Scale-independent overall fracture density (P₂₂-style intensity metric).
N_II, N_IC, N_CCCount (integer)≥ 0Branch population by connectivity class.
C_LRatio (dimensionless)typically 0–4Connections per fracture line.
C_BRatio (dimensionless)typically 0–2Connections per branch; theoretical random-network bound is exactly 2.
Barycentric ternary positionComposition (proportion triple)simplexVisual, compositional summary of any of the three-part classifications above.

The principal sources of uncertainty are (a) digitisation error in the original trace map, (b) sensitivity to the chosen snap tolerance, and (c) truncation/censoring at the edges of the mapped domain.

7. Visualization & Figure Components

Five complementary Matplotlib diagrams are available, switchable instantly from the Plots navigation card. All share the workspace's global font-size, font-weight, and font-colour settings (matplotlib.rcParams) and honour the Display Settings colour/size overrides described in Section 3.4.

7.1 I-Y-X Connectivity Nodes (Spatial Map)

This is the only diagram drawn in true map (X, Y) coordinates. Every digitised segment is rendered as a thin line (LineCollection, using the theme's spine/border colour), and the three node classes are overlaid as coloured scatter points: red for I, green for Y, blue for X by default. Marker size follows the "Base Node Size" setting. If "Show node labels" is enabled, each node receives a small text label (e.g. "X7") numbering it within its class. A legend with three columns is drawn beneath the plot. The axes are scaled equal (1:1 aspect) and padded by 5% of the bounding-box extent on each side.

7.2 I-Y-X Ternary Connectivity

An equilateral-triangle ternary diagram with vertices labelled Y (bottom-left), X (bottom-right), and I (apex). A single orange marker plots the network's (p_i, p_y, p_x) composition using the barycentric transform of Section 4.6, annotated with a text callout reporting all three fractions to two decimal places (e.g. "I 0.32   Y 0.41   X 0.27"). A point sitting close to the Y-X base edge indicates a well-connected network dominated by terminations and crossings rather than dead ends.

7.3 Connections per Line (C_L)

The same Y/X/I ternary frame, overlaid with a family of iso-C_L contour lines drawn for C_L = 1, 2, 3, 4, 6, 8, 10, 20, plus a distinguished dashed contour at the frequently cited threshold C_L = 3.57 — the value associated with the transition to well-connected, likely- percolating networks in several published topological classifications. The observed data point is plotted as an amber marker with an annotated C_L value.

7.4 Connections per Branch (C_B)

Structurally identical to the C_L diagram but contoured for C_B = 0.5, 1.0, 1.3, 1.5, 1.7, 1.8, 1.9, 1.95, plus the fixed C_B = 2.0 contour, which coincides exactly with the bottom (Y–X) edge of the triangle since C_B = 2 is the theoretical maximum (every branch fully connected at both ends, i.e. zero I-nodes). The observed point is plotted as a cyan marker with an annotated C_B value.

7.5 Branch Topology (I-I · I-C · C-C)

A ternary diagram with vertices I-C (bottom-left), C-C (bottom-right), and I-I (apex), overlaid with the theoretical random-network locus described in Section 4.7 (blue curve with C_B tick labels at 0.0–2.0) and the observed branch-population point plotted as a white marker with a dark outline, annotated with its independently computed C_B value derived from the I-I/I-C/C-C proportions.

7.6 Empty and Loading States

When no data is loaded, when the data source lacks real map geometry, or while a recomputation is pending but the tab is not the active one, all five axes are cleared and a single centred, muted-colour text message is drawn by _fluid_empty_state() explaining the reason. All sidebar statistic labels are simultaneously reset to "–".

7.7 Global Styling Controls

Every diagram observes the same background/foreground/grid/border colour variables and the same border line width and grid line width entries, so the five diagrams present a visually consistent, publication-ready style suite. Corner labels and axis titles automatically scale with the workspace's global font-size control (mpl.rcParams['axes.titlesize']).

8. Step-by-Step Example Workflows

8.1 Assessing Reservoir Connectivity from an Outcrop Trace Map

  1. Load an SVG trace map (or a spatial Excel/CSV) into the Static Analyzer and confirm the Fracture Map tab renders correctly with calibrated scale.
  2. Open the Connectivity tab. Because the data is new, the tab is marked stale and an automatic lazy computation runs immediately with the default 0.1-unit tolerance.
  3. Inspect the I-Y-X Connectivity Nodes map. Visually confirm that crossing (blue) markers sit at genuine X-shaped intersections, and abutting (green) markers at T-shaped terminations. If many genuinely touching fracture tips are coloured red (I), the snap tolerance is too small relative to digitisation precision; if unrelated nearby fractures are merged into a single node, the tolerance is too large.
  4. Adjust "Node snap tolerance" as needed (e.g. from 0.1 m to 0.25 m for coarse digitisation) and press "↻ Recalculate".
  5. Decide on boundary I-nodes: use exclusion (the default) for an arbitrary rectangular crop of a larger fracture field; use inclusion if fracture tips genuinely terminate at a natural geological boundary such as a bounding fault.
  6. Read the I-Y-X Summary card: for example, I = 42, Y = 118, X = 76, Total = 236, Branches = 310, CF = 0.822, C̄ = 0.877, B₂₂ = 4.6.
  7. Switch to the I-Y-X Ternary diagram — a point sitting close to the Y-X edge (away from the I apex) indicates a well-connected network.
  8. Switch to the C_L diagram — a point to the right of the dashed C_L = 3.57 contour indicates a topologically well-connected, likely-percolating regime.
  9. Switch to the Branch Topology diagram — a point displaced toward the C-C corner relative to the random-network locus suggests connectivity in excess of a purely random linking process, consistent with mechanically controlled fracture growth.
  10. Use "⬇ Save Image" on each diagram of interest to export 300 dpi PNGs to the project's FracSTAT_plot_exports folder for inclusion in a report or publication.

8.2 Worked Numerical Example

Consider a network reporting I = 10, Y = 20, X = 10 (N = 40 nodes) and, after branch splitting, N_II = 4, N_IC = 12, N_CC = 14 (N_B = 30 branches). The connectivity statistics are:

CF = (20 + 10) / 40 = 0.75 C̄ = (20 + 2·10) / 30 = 40/30 ≈ 1.33 N_L = (10 + 20) / 2 = 15 C_L = 2·(10 + 20) / 15 = 4.0 N_Br = (10 + 3·20 + 4·10) / 2 = 55 C_B = (3·20 + 4·10) / 55 = 100/55 ≈ 1.82

With C_B = 1.82 exceeding the well-connected reference value of 1.7–1.8 commonly cited in topological classifications, and C_L = 4.0 exceeding the 3.57 percolation-related threshold, this hypothetical network would be characterised as well connected and likely to behave as an effective fluid-flow pathway network.

9. Geological Interpretation

9.1 Connectivity Statistics and Fluid Flow

Fracture connectivity, not fracture density alone, is frequently the controlling factor in whether a naturally fractured reservoir, aquifer, or rock mass behaves as an effective flow or mechanical system. Two intervals with identical fracture intensity (P10, P21, or B₂₂) can have dramatically different bulk permeability if one is dominated by isolated fractures (high I-node fraction, low CF) while the other forms an interconnected lattice (high X/Y-node fraction, high CF).

9.2 Node-Type Ratios and Network Maturity

Increasing abundance of crossing (X) nodes relative to abutting (Y) nodes is often associated with a more mature, multiply-reactivated fracture system where successive fracture sets have propagated through pre-existing fractures rather than systematically terminating against them. Conversely, a network dominated by Y-nodes frequently records a clear relative-age sequence, with younger, shorter fractures terminating against an earlier, more continuous fracture set — a pattern common in systematically jointed sedimentary sequences with a dominant bedding-perpendicular joint set.

9.3 Percolation and Effective Connectivity Thresholds

The C_L = 3.57 and C_B ≈ 2 reference values on the ternary diagrams derive from topological/percolation-theory studies of two-dimensional fracture networks (see Section 13, References), where networks below the threshold are statistically unlikely to form a fully connected percolating cluster spanning the sample domain, while networks above typically do. These are heuristic reference values rather than universal physical constants: the true percolation threshold for any specific network depends on the fracture-length distribution, orientation distribution, and aperture/permeability contrast, quantified separately in other FracSTAT tabs.

9.4 Dead-End Branches (I-C) and Hydraulic Significance

A large I-C (dead-end spur) fraction indicates that a substantial portion of the mapped fracture length — while nominally connected to the main network at one end — terminates in intact rock at the other and cannot itself provide a through-going flow path. Distinguishing the C-C (fully connected, potentially conductive) branch population from the I-C and I-I populations is therefore directly relevant when the Connectivity results are combined with permeability up-scaling or discrete-fracture-network flow simulation performed outside FracSTAT.

9.5 Scale Dependence and Sampling Considerations

Connectivity statistics are not fully scale-invariant: the observed I-node fraction is sensitive to the size of the mapped domain relative to the mean fracture length, because a larger domain captures more complete fracture traces (fewer artificial edge terminations). The boundary-I-node exclusion option partially corrects for this effect, but analysts should still be cautious when comparing connectivity statistics computed from maps of substantially different physical size or aspect ratio.

10. Performance Optimization

Because the underlying topological computation is O(n²) in the number of digitised segments, the Connectivity tab is deliberately engineered to run this computation as rarely as possible while still guaranteeing that displayed statistics are always correct and current.

10.1 Result Caching

The single ConnectivityResult object produced by the most recent successful call to count_iyx_nodes() is retained in self._fluid_connectivity for the lifetime of the workspace session. Every cosmetic redraw (colour change, grid toggle, node-label toggle, pan, zoom, Reset View) reuses this cached object directly rather than recomputing it.

10.2 Lazy, On-Demand Computation

The tab is only ever computed when it is both (a) marked stale by a genuine geometry change and (b) actually the currently visible tab (or explicitly forced via Recalculate or a settings change that mandates recomputation). If the analyst never opens the Connectivity tab, the expensive computation never runs at all, regardless of how many files are loaded or how many times the profile scale is changed elsewhere in the workspace.

10.3 Geometric Fingerprinting

Every time refresh_static_fluidflow() is called, it computes a cheap fingerprint of the current results table — the tuple (row count, rounded total segment length). The fingerprint is O(1) to compare and O(n) to compute, several orders of magnitude cheaper than the O(n²) computation it guards. It changes only when the underlying digitised geometry itself has changed, remaining completely insensitive to purely cosmetic changes such as colour, font, or grid visibility.

10.4 Deferred Execution and UI Responsiveness

Because the algorithm can take perceptible wall-clock time for dense digitisations, every user action that may trigger a recompute is scheduled through Tkinter's after(100, …) mechanism rather than executed synchronously. This 100 ms delay is sufficient for the global loading overlay to be physically painted to the screen before the main thread blocks on the O(n²) search, so the application never appears to freeze without explanation.

10.5 Single-Figure-per-Diagram Construction

All five Matplotlib Figure/Axes/Canvas triples are constructed exactly once at tab-build time and simply shown/hidden thereafter. This avoids the overhead of destroying and rebuilding Matplotlib canvases every time the analyst switches between the five available diagrams, at the modest cost of holding five figures in memory simultaneously.

10.6 Recommended Practice for Large Data Sets

For SVG traces exceeding several thousand segments, consider increasing the Curve step (px) parameter when loading the SVG to reduce the number of straight-line segments used to approximate curved traces. Adjust the snap tolerance thoughtfully before pressing Recalculate rather than repeatedly nudging it — each change re-triggers the full O(n²) computation. Leave "Show node labels" off for very dense networks, as densely overlapping text labels can noticeably slow down canvas rendering and obscure the spatial map.

11. Advanced Features

11.1 Interactive Pan and Zoom

All five canvases share a common pair of mouse-driven navigation handlers connected at tab-build time. The scroll wheel (_on_scroll_zoom) zooms in or out by a fixed 1.2× factor per click, centred exactly on the cursor position so that the data-space coordinate under the cursor is held fixed while the axis limits are rescaled around it. Left-click drag (_on_mouse_press / _on_mouse_drag / _on_mouse_release) pans the active diagram by translating both axis limits by the same data-space displacement as the cursor. "↺ Reset View" restores the default extent without any recomputation.

11.2 Custom Colour Picker

Every colour swatch in the Display Settings sidebar opens the shared FracSTAT colour-picker dialog (_StaticColorButton), which presents a curated 8×5 matrix of 40 professionally chosen preset swatches spanning eight hue families at five lightness levels, plus a free-form hex-code entry field with a live preview patch. Selecting a preset or typing a valid hex code and pressing "Apply" immediately re-fires the connectivity redraw callback bound to that colour variable.

11.3 Calibrated Background Photograph Overlay

The Connectivity tab supports overlaying a real, scale-calibrated photograph of the outcrop behind the spatial node map (the four ternary diagrams do not use a background image). The implementation (_autodetect_fluid_bg_image_path()) searches in priority order: (1) an image path already stored in this tab's own StringVar; (2) the Fracture Map tab's own calibrated background image variable, if set; (3) a sidecar PNG automatically saved alongside a traced SVG by the Trace Map tool (named <svg-basename>_bg.png). The chosen path is cached in memory so repeated redraws do not re-read the file from disk. The image is drawn with imshow, its extent locked to the bounding box of the current fracture geometry, and its opacity controlled by the Image opacity entry.

12. Limitations

LimitationDescription
Requires genuine spatial geometryOrientation-only Excel/CSV formats (legacy Strike/Dip tables, Trend/Plunge pole tables) cannot be analysed, since they carry no real map position.
O(n²) computation costFor very dense digitisations (tens of thousands of segments), computation time can become noticeable even with the caching strategy, since a genuine geometry change still requires a full recomputation.
Greedy point-clusteringThe order-dependent greedy clustering algorithm can produce weakly order-dependent cluster centroids in rare pathological configurations of closely spaced, nearly-collinear candidate points. In practice, at sensible tolerances, this has negligible effect on reported statistics.
Two-dimensional onlyThe module operates purely on the projected trace map and does not account for fractures that appear to intersect in map view but are vertically offset (non-intersecting) in three dimensions — a limitation inherent to any 2-D trace-map analysis.
No uncertainty quantificationNode classification and branch typing are entirely deterministic given the digitised geometry, tolerance, and boundary setting. No Monte Carlo resampling of tolerance or digitisation error is currently implemented within this module.
Heuristic reference contoursThe C_L = 3.57 and C_B ≈ 2 reference contours are values drawn from the published topological fracture-network literature and are not dynamically recalculated for the specific length or orientation distribution of the loaded data set.
Rectangular boundary assumptionBoundary I-node exclusion uses the simple rectangular bounding box of all digitised endpoints as the sample-domain boundary. For irregularly shaped mapped areas, this can over- or under-exclude genuine edge-truncated tips relative to the true panel boundary.
Main-thread computationAll fitting and intersection computation runs on the main UI thread since Matplotlib's TkAgg backend is not safe to drive concurrently from a worker thread. Very large networks will therefore block interaction until the computation completes, mitigated only by the loading overlay.
