Dynamic Map Tab

Located at dynamic_analyzer/tabs/dynamic_map_tab.py, the Map Tab (implemented as the MapTab mixin class) is the primary spatial visualization component of the Dynamic Analyzer workspace within FracSTAT. Whereas the Static Analyzer's Fracture Map tab presents purely geometric information (trace geometry, segment length, orientation), the Dynamic Analyzer's Map Tab overlays the mechanical response of each fracture segment — its resolved slip tendency, dilation tendency, fracture susceptibility, or binary critical-stress state — onto the same spatial or angular reference frame from which the segment was digitized.

This module functions as the geometric and cartographic rendering engine for the SlipTendencyApp workspace (dynamic_app.py). It receives a fully populated pandas DataFrame of per-segment stress-resolution results (produced by core.calculator.run_calculation) and renders it using one of three complementary spatial representations, automatically selected according to the nature of the loaded dataset:

  • A true-to-scale Cartesian map for SVG-digitized trace networks and for Excel/CSV datasets that carry genuine X–Y coordinate geometry;
  • A synthetic unit-circle orientation wheel for legacy Excel/CSV fracture-plane tables (Strike/Dip only, with no real spatial position);
  • A geographic longitude–latitude scatter map, with optional embedded beachball diagrams, for Global-CMT-style focal-mechanism datasets.

The module also provides the calibrated real-world photographic background overlay, interactive pan/zoom, cross-tab segment highlighting, a fully configurable appearance settings panel, and a Move-Elements layout editor that lets the user reposition the plot axes and colour bar for publication-quality figure export.

1. Position within the FracSTAT Architecture

The Map Tab is one of six tabs assembled inside SlipTendencyApp (dynamic_app.py), the others being Rose, Mohr, Stereonet, Table, and Method. All six tabs share a single results DataFrame (self.results) that is recomputed whenever the user presses Calculate, changes a stress or geometry parameter, or loads a new SVG/Excel file. The Map Tab does not perform stress resolution itself; it is a pure consumer of the DataFrame produced by core/calculator.py and, for SVG data, the trace geometry produced by core/svg_loader.py and core/models.py.

1.1 Data Flow

  1. core.svg_loader.read_svg_segments() or core.excel_loader.read_excel_segments() parse the source file into a list of Segment objects.
  2. core.calculator.run_calculation() resolves the stress tensor onto each segment, producing columns such as StrikeDeg, DipDeg, SigmaN, Tau, SlipTendency, SlipTendencyNorm, DilationTendency, FractureSusceptibility, FractureSusceptibilityNorm, CriticalStress, CriticalStressValue, X1/Y1/X2/Y2, RawX1…RawY2, Lat, Lon, EventLabel, PlaneName, BeachballURL, and related fields.
  3. dynamic_app.py stores this table as self.results and calls refresh_plots(), which in turn calls MapTab._draw_map().
  4. MapTab._draw_map() dispatches to one of four internal drawing routines depending on the data source and the geometry actually present, and renders the result onto a Matplotlib Figure embedded in the CustomTkinter window via FigureCanvasTkAgg.
  5. User interaction with the Map Tab (segment selection, pan/zoom, appearance changes) feeds back into shared application state (selected_segment_id, view limits) that the Mohr, Stereonet, and Table tabs also read, enabling synchronized cross-tab highlighting.

2. Interface Overview

The Map Tab occupies the full content area of the "Map" tab within the Dynamic Analyzer's CTkTabview. Its layout is a two-column grid: a resizable plot area on the left (weighted to expand) and a collapsible settings panel on the right (fixed width, hidden by default).

It is organized into four labelled sections — Display Options, Map Elements, Colour Bar, and Background Image — followed by a Reset Map Defaults button. Every control in the panel is wired to immediately redraw the map with preserve_view=True, so pan/zoom position is not lost while the user experiments with appearance settings.

3. Data Input and Prerequisites

The Map Tab does not load files itself; it consumes state populated by the shared workspace header's Load SVG / Load Excel actions and by core.calculator.run_calculation(). Three attributes on the parent SlipTendencyApp instance govern which drawing routine is used:

3.1 Required and Optional Columns

Depending on the branch selected, the following results columns are read:

  • Cartesian / SVG mode: X1, Y1, X2, Y2 (scaled, real-world units) and RawX1, RawY1, RawX2, RawY2 (unscaled SVG pixel coordinates, used only when the profile has not yet been calibrated — see Section 5.3).
  • Orientation-wheel mode: StrikeDeg, DipDeg, ID.
  • Geographic mode: Lat, Lon, PlaneName, EventLabel, Region, BeachballURL.
  • All modes: ID (for selection), plus whichever attribute column is selected for colouring (SlipTendency/SlipTendencyNorm, DilationTendency, FractureSusceptibility/FractureSusceptibilityNorm, or CriticalStressValue).

3.2 Coordinate Calibration Prerequisite

Cartesian rendering assumes the fracture network has been placed within a physically meaningful profile box. If the profile has not yet been calibrated (Width and Length sidebar fields still at their uncalibrated defaults of 260 and 100), the module falls back to plotting the raw, unscaled SVG pixel coordinates so that the original digitized proportions are preserved rather than being artificially stretched to fill a default 260 × 100 box — see the raw-coordinate detection logic described in Section 6.2.

The Reset Map Defaults button restores every Display Options, Map Elements, and Colour Bar control to the factory values listed above and redraws the map with preserve_view=True. By deliberate design, this reset does not clear the Background Image settings (mode, path, or opacity): the code comment in the implementation notes that "resetting plot cosmetics shouldn't silently detach a calibrated reference photo the user deliberately attached." 5. Mathematical Background

5.1 Colour-Mapped Attributes

The Map Tab colours every plotted element (line segment, orientation tick, or event marker) according to one of four mechanical attributes, selected via the Display card's "Map Display" dropdown in the sidebar (outside the Map Tab itself, but consumed here) and the "Color by normalized Tₛ" checkbox. The mapping from the selected mode to a results column, axis label, and title fragment is:

Map Display ModeNormalized columnRaw columnFormula
Slip tendencySlipTendencyNormSlipTendencyTₛ = τ / σₙ
Dilation tendency— (always shown raw)DilationTendencyT_d = (σ₁ − σₙ) / (σ₁ − σ₃)
Fracture susceptibilityFractureSusceptibilityNormFractureSusceptibilityS_f = σₙ − P_f − (τ − C₀) / μ
Critical fracturesCriticalStressValue (binary)—1 if τ ≥ μ(σₙ − P_f) + C₀, else 0

5.1.1 Definition of Terms

  • σₙ — normal stress resolved on the fracture plane (MPa).
  • τ — shear stress resolved on the fracture plane (MPa).
  • σ₁, σ₃ — maximum and minimum principal stresses (MPa).
  • P_f — pore-fluid pressure (MPa).
  • C₀ — cohesion of the fracture surface (MPa).
  • μ — coefficient of friction (dimensionless).
  • Tₛ (Slip Tendency) — the ratio of resolved shear to normal stress; a plane slips when Tₛ approaches or exceeds μ.
  • T_d (Dilation Tendency) — the propensity of a fracture to open (dilate) under the current stress state, normalized between the two principal stress extremes.
  • S_f (Fracture Susceptibility) — a Mohr-Coulomb failure-margin proxy combining effective normal stress and cohesion-corrected shear stress; lower (more negative) values indicate closer proximity to shear failure.

5.1.2 Normalization

When "Color by normalized Tₛ" is enabled, Slip Tendency is divided by a theoretical maximum Tₛ,max computed analytically from σ₁ and σ₃ alone (independent of the sampled fracture set — see core.calculator._theoretical_max_slip_tendency), guaranteeing that SlipTendencyNorm values fall in a stable, comparable [0, ~1] range regardless of which fractures happen to be present in the loaded dataset. FractureSusceptibilityNorm is instead a simple min–max rescaling across the loaded fracture population: (S_f − S_f,min) / (S_f,max − S_f,min). Dilation Tendency and Critical Fractures are dimensionless/binary by construction and are never further normalized.

5.2 Colour Normalization for Rendering

Once the attribute column is selected, MapTab.attribute_colormap_and_norm() (delegating to SlipTendencyApp._attribute_norm in dynamic_app.py) determines the Matplotlib Normalize object used to map data values to the 0–1 colormap domain:

  • Columns ending in "Norm", plus CriticalStressValue and DilationTendency, use a fixed Normalize(vmin=0, vmax=1).
  • All other (raw, unnormalized) columns use the empirical minimum and maximum of the finite values present in the current dataset. If the data is degenerate (min ≈ max), a symmetric padding of 5% of the magnitude (or 1.0 if the value is exactly zero) is applied so the colour scale remains well-defined.
  • CriticalStressValue is treated specially: rather than a continuous colormap, a two-colour ListedColormap(["muted", "failure"]) is combined with a BoundaryNorm([-0.5, 0.5, 1.5], N) so that the binary 0/1 values are rendered as two crisp, discrete colours ("Stable" and "Critical") rather than a misleading continuous gradient. The colour bar in this mode is relabelled with the ticks "Stable" and "Critical" instead of numeric values.

5.3 Cartesian Coordinate Scaling

For SVG-sourced traces, real-world segment endpoints (X1, Y1, X2, Y2) are computed upstream by scaling the raw pixel bounding box of the digitized traces onto the user-specified Profile Width × Profile Length box. Because the module cannot know from the DataFrame alone whether this calibration has actually been performed, it applies a heuristic: if the scaled data's X-span and Y-span are numerically close (relative tolerance 1 × 10⁻³) to the un-calibrated default profile dimensions of 260 and 100 (the StringVar defaults for profile_width_var and profile_height_var in dynamic_app.py), the module infers that no real calibration has occurred and instead renders the RawX1/RawY1/RawX2/RawY2 pixel columns, preserving the original digitized aspect ratio rather than an arbitrary default stretch. Axis labels switch accordingly from "X (unit)"/"Y (unit)" to "X (px)"/"Y (px)".

5.4 Orientation-Wheel Geometry

For legacy Excel/CSV fracture-plane tables with Strike/Dip but no spatial coordinates, each fracture is represented as a short radial tick mark on a unit circle (radius R = 1). Let θ be the strike azimuth in radians (compass convention: 0° = north, increasing clockwise) and δ be the true dip in degrees. The tick's centre point is:

cx = R · sin(θ), cy = R · cos(θ)

and the tick extends symmetrically along the local perpendicular direction, (cos θ, −sin θ), by a half-length of:

half = 0.12 · sin(δ · π/180)

so that a vertical fracture (δ = 90°) is drawn as the longest possible tick, while a horizontal (bedding-parallel, δ = 0°) fracture collapses to a point — an intuitive visual analogue of true dip magnitude. The wheel therefore encodes strike as angular position around the circle and dip as radial tick length, giving a compact one-glance summary of the orientation population without requiring real spatial coordinates.

5.5 Geographic (Focal-Mechanism) Projection

For focal-mechanism datasets, event locations are plotted directly in longitude/latitude space. Because one degree of longitude does not span the same physical distance as one degree of latitude away from the equator, the Axes aspect ratio is corrected by:

aspect = 1 / max(cos(latitude_mean · π/180), 0.15)

where latitude_mean is the arithmetic mean latitude of the plotted events. This keeps circular/point features visually undistorted at the map's characteristic latitude, using a lower bound of 0.15 to avoid an unbounded aspect ratio near the poles.

5.6 Interactive Zoom Mathematics

Mouse-wheel zooming is implemented as a cursor-anchored rescaling of the current axis limits. Given a zoom factor f (0.85 per "scroll up" step, 1.18 per "scroll down" step) and cursor data-coordinates (cx, cy), the fractional position of the cursor within the current view is computed for each axis, e.g. for x:

relx = (cx − x_min) / (x_max − x_min)

The axis span is then scaled by f and the new limits are placed so that the cursor's fractional position within the view is preserved, i.e. the point under the cursor remains stationary on screen while the surrounding view expands or contracts. The same logic is applied independently to the y-axis, and correctly accounts for inverted axis directions (relevant when the SVG Y-axis convention places larger Y values toward the top of the screen).

6. Algorithms

6.1 Drawing Dispatch Algorithm

Every call to _draw_map() begins by determining which of four internal renderers should handle the current dataset. The decision tree, implemented in _draw_map() and mirrored in the table below, is evaluated in this priority order:

StepCondition TestedRoutine Invoked
1data_source == "excel" and segments[0]._has_map_geometry is False_draw_map_no_geometry — explanatory empty-state message
2data_source == "excel", geometry present, and "Lat" column has at least one non-null value_draw_map_geo — geographic focal-mechanism map
3data_source == "excel", geometry present, no Lat data, and "Length" column is not uniformly zero_draw_map_svg — Cartesian map using the genuine X/Y coordinates supplied by the spreadsheet
4data_source == "excel", none of the above (legacy Strike/Dip table)_draw_map_excel — synthetic orientation wheel
5data_source == "svg" (default branch)_draw_map_svg — Cartesian map from digitized trace geometry
python 
def _draw_map(self, preserve_view=False):
    if getattr(self, "data_source", "svg") == "excel":
        segs = getattr(self, "segments", None)
        has_geometry = bool(segs) and getattr(segs[0], "_has_map_geometry", True)

        # 0. No real spatial coordinates and no Lat/Lon — e.g. a legacy
        #    fracture-plane file or a Trend/Plunge pole file. There is no
        #    meaningful map to draw, so show an explanatory empty state.
        if not has_geometry:
            self._draw_map_no_geometry(preserve_view)

        # 1. Geographic Data (Focal Mechanisms)
        elif "Lat" in self.results.columns and self.results["Lat"].notna().any():
            self._draw_map_geo(preserve_view)

        # 2. Raw Coordinate Data (Spatial Map)
        elif "Length" in self.results.columns and not (self.results["Length"] == 0).all():
            self._draw_map_svg(preserve_view)

        # 3. Legacy Strike/Dip (Orientation Wheel)
        else:
            self._draw_map_excel(preserve_view)
    else:
        # Standard SVG
        self._draw_map_svg(preserve_view)

6.2 Cartesian Rendering Algorithm (_draw_map_svg)

  1. Preserve the current axis limits if preserve_view is requested and a home view already exists.
  2. Clear the plot and colour-bar Axes and apply the outer/inner background colours.
  3. Determine whether raw (uncalibrated) or scaled coordinates should be used, per the heuristic in Section 5.3.
  4. Reshape the endpoint columns into an (N, 2, 2) array suitable for a Matplotlib LineCollection.
  5. Compute the data bounding box and draw the calibrated background photograph, if enabled, beneath the traces (zorder = 0).
  6. Apply a 2% padding margin to the bounding box so that boundary segments are not visually clipped.
  7. Resolve the active colouring attribute and its colormap/normalization (Sections 5.1–5.2), build the LineCollection with per-segment colour values, and add it to the Axes.
  8. If a segment is currently selected (selected_segment_id), redraw it with a two-layer halo-and-core highlight (a thick under-stroke in the text colour, a slightly narrower over-stroke in the failure/accent colour) plus endpoint markers, so the selection remains visible against any colormap.
  9. Position and populate the colour bar according to the chosen dock position (Section 8.4), or hide it and expand the plot Axes if "None" is selected.
  10. Apply axis limits (respecting the Y-axis reversal setting), then set the aspect ratio to 'equal' with adjustable='box' — deliberately applied after the limits so the plot area shrinks to fit the data rather than stretching the data to fill the plot area.
  11. Store the resulting (pre-aspect-correction) tight data bounds as the new "home" view for the Reset View button and zoom-percentage calculation.
  12. Apply axis labels, title, gridlines, and boundary spine styling; refresh the zoom label; request an idle canvas redraw.

6.3 Orientation-Wheel Rendering Algorithm (_draw_map_excel)

  1. Draw the unit-circle boundary ring and, if enabled, the four compass labels.
  2. For each fracture, compute its tick centre and half-length per Section 5.4, colour it by the active attribute, and collect all ticks into a single LineCollection for efficient rendering.
  3. If a segment is selected, recompute its tick position and draw the same halo-and-core highlight pattern used in Cartesian mode, plus an "ID n" text annotation offset from the tick.
  4. Fix the axis limits to a symmetric [-1.35, 1.35] box (turning off axis ticks and labels, since the wheel has no physical coordinate system), apply the descriptive title, and finalize styling exactly as in Cartesian mode.

6.4 Geographic Rendering Algorithm (_draw_map_geo)

  1. Reduce the results table to one representative row per event: rows tagged PlaneName == "FP1" are preferred when present (so two-nodal-plane focal mechanisms are not double-plotted); otherwise rows are de-duplicated on the (Lat, Lon) pair.
  2. Resolve the active attribute for colouring and build a single scatter plot of all events, with marker size proportional to the Line width control.
  3. Compute padded longitude/latitude limits (12% of the data range, or a fixed 0.5–1.0° minimum pad for single-point or degenerate datasets) and apply the latitude-corrected aspect ratio from Section 5.5.
  4. If a segment/event is selected, draw a double concentric-ring highlight around it, an event-label annotation, and — if a Beachball URL is present for that event — invoke the beachball inset renderer (Section 9.2).
  5. Position the colour bar and finalize styling as in the other two modes.

6.5 Empty-State Algorithm (_draw_map_no_geometry)

When the loaded Excel/CSV table has neither spatial coordinates nor latitude/longitude columns (e.g. a legacy fracture-plane file lacking even orientation-wheel-compatible fields, or a Trend/Plunge pole file), the module deliberately avoids drawing a misleading synthetic map. Instead it clears the Axes, hides the colour bar, and displays a centred, word-wrapped explanatory message directing the user to the Table, Rose, Mohr, and Stereonet tabs, where orientation and stress results remain fully available.

7. Internal Workflow

The Map Tab participates in the broader Dynamic Analyzer refresh cycle as follows:

  1. The user loads an SVG or Excel/CSV file, or edits a stress/geometry parameter, or clicks Calculate.
  2. SlipTendencyApp.calculate() (dynamic_app.py) invokes core.calculator.run_calculation() and stores the returned DataFrame as self.results.
  3. SlipTendencyApp.refresh_plots() is called, which invokes MapTab._draw_map(preserve_view=...) alongside the Rose, Mohr, and Stereonet tabs' own redraw methods.
  4. _draw_map() runs the dispatch algorithm of Section 6.1 and delegates to the appropriate renderer.
  5. The renderer clears and repopulates the Matplotlib Axes, updates self._map_home_limits, and calls draw_idle() on the FigureCanvasTkAgg for an efficient, non-blocking repaint.
  6. Subsequent user interactions — mouse-wheel zoom, click-drag pan, segment selection in this or another tab, or a settings-panel change — call _draw_map(preserve_view=True) so the current pan/zoom state (rather than the freshly computed home view) is retained across cosmetic or selection-only updates.

7.1 Cross-Tab Segment Selection

Clicking a focal-mechanism event marker (geographic mode only; Cartesian and wheel modes do not currently support direct click-to-select on the map itself) triggers on_map_pick(), which resolves the picked scatter index back to a segment ID via the parallel _map_geo_ids array and calls the shared select_segment() method on the parent application. This, in turn, updates selected_segment_id and triggers a synchronized redraw of the Map, Mohr, and Stereonet tabs, and refreshes the Table tab's active row — so a fracture identified as critically stressed in the Mohr diagram can be located instantly on the map, and vice versa.

7.2 State Persistence

Several pieces of Map Tab state persist across saves/reloads of a project session (see core/project_state.py): the background-image mode flag, image path, and opacity are written into the saved session JSON so a calibrated photograph reattaches automatically when the project is reopened. Pan/zoom position and the settings-panel open/closed state are, by contrast, session-local UI state and are not persisted.

8. Statistical Methods and Colour-Bar Geometry

8.1 Descriptive Range Handling

For raw (non-normalized) attribute columns, the module relies on the empirical minimum and maximum of the finite values in the current results table (Section 5.2) rather than any fixed theoretical range. This is a deliberate choice: because Fracture Susceptibility and raw Slip Tendency have stress-tensor-dependent physical ranges, a fixed colour scale would either clip informative variation or waste dynamic range on datasets with a narrow spread. The 5% padding rule for degenerate (single-value) datasets prevents a divide-by-zero condition in Matplotlib's Normalize and ensures the colour bar always renders a visible, non-degenerate gradient.

8.2 Binary Classification Display

The Critical Fractures mode is a two-class statistical summary rather than a continuous field: each segment is classified as "Critical" (τ at or above the Mohr-Coulomb failure threshold) or "Stable" (below it). The Map Tab renders this classification with a fixed two-colour palette and boundary-normalized colour mapping (Section 5.2) so that the discrete nature of the result is never misrepresented as a continuous gradient — a common source of misinterpretation in geomechanical figures.

8.3 Event De-duplication (Geographic Mode)

Focal-mechanism datasets may contain one or two nodal-plane rows per physical earthquake. The Map Tab's de-duplication step (preferring the FP1-tagged row, or otherwise the first occurrence of each unique Lat/Lon pair) ensures that summary statistics implied by the map — the number of markers, their spatial density — correctly reflect the number of physical events rather than the number of nodal-plane table rows.

8.4 Colour-Bar Docking Geometry

The exact rectangle occupied by the colour bar Axes, and the corresponding resize of the main plot Axes, is computed from the user-selected Position and Length settings as follows (all values in figure-fraction coordinates; L = user-configured colour-bar length fraction):

9. Visualization Components

9.1 Line and Marker Rendering

All fracture segments (or orientation ticks) sharing a common colouring attribute are rendered as a single Matplotlib LineCollection rather than as individually plotted lines. This is both a performance optimization (Section 11) and a visual-consistency measure, since a LineCollection guarantees uniform handling of colour mapping, z-ordering, and anti-aliasing across the entire fracture population.

9.2 Beachball Diagram Insets

For focal-mechanism datasets that supply a Beachball URL, selecting an event renders an inset diagram card near that event's marker using Matplotlib's offsetbox machinery (AnnotationBbox, OffsetImage, VPacker, DrawingArea, TextArea). The inset card displays:

  • A bold title line showing the event label (or "Focal mechanism" if unavailable);
  • The downloaded beachball image, once available, rendered via OffsetImage;
  • A live "Loading…" placeholder — a circular ring with a coloured arc segment mimicking a spinner — while the image is still downloading;
  • An "Image unavailable" placeholder if the download failed.

The card is drawn with a rounded, semi-transparent background box, an arrow connector back to the event's map location, and (where the Matplotlib version supports it) a subtle drop-shadow path effect for visual depth, styled to match the current light/dark appearance mode.

9.3 Selection Highlighting

Across all three rendering modes, the currently selected fracture is emphasized with a consistent two-layer "halo and core" technique: a wider stroke in the workspace text colour is drawn first, followed by a narrower stroke in the failure/accent colour on top, producing a high-contrast outline that remains legible regardless of the underlying colormap. Endpoint or event markers are added in the accent colour with a contrasting edge, and — in geographic and wheel modes — a short text label identifying the segment ID or event label is annotated nearby.

9.4 Background Photograph Overlay

The calibrated real-world photograph, when enabled, is drawn beneath all vector content (zorder = 0) using imshow with an extent tuple of (xmin, xmax, ymax, ymin) rather than the more conventional (xmin, xmax, ymin, ymax). This intentional vertical flip compensates for the SVG coordinate convention (Y increasing downward) so that the photograph aligns pixel-for-pixel with the overlaid trace geometry rather than appearing upside-down relative to it.

10. Advanced Features

10.1 Move Elements Layout Editor

When Move Elements mode is enabled from the shared workspace header (top toolbar, outside this module), clicking and dragging inside the map or colour-bar Axes repositions that Axes' bounding box within the figure, in real time, by translating the mouse displacement in screen pixels into figure-fraction coordinates. This allows the user to manually compose a publication-ready layout — for example, shrinking the plot slightly and relocating the colour bar to an unconventional position — before using Save Image to export a high-resolution figure. The Map Tab intercepts these press/motion/release events itself (_on_layout_press, _on_layout_motion, _on_layout_release) rather than relying solely on the generic cross-tab layout-editing handlers used by the Rose, Mohr, and Stereonet tabs, because the Map Tab's canvas also has ordinary pan/zoom bindings on the same mouse buttons that must be suppressed while editing is active.

10.2 Calibrated Background Photograph Auto-Detection

The module attempts to automatically reattach a background photograph without requiring the user to re-browse for it, using a three-tier fallback search: (1) the path already stored for this session, if the file still exists; (2) a sidecar PNG file named after the source SVG ("<name>_bg.png"), automatically written by the Trace Map tool when a photograph was calibrated there; (3) a photograph already calibrated in the Static Analyzer workspace for the same project, shared live via the launcher's cross-workspace state. This means a single calibration performed anywhere in the application — the Trace Map tool, the Static Analyzer, or the Dynamic Analyzer's own Scale dialog — becomes available to the Map Tab without repetition.

10.3 Asynchronous Beachball Retrieval

Beachball diagram images are fetched over HTTP in a background ThreadPoolExecutor (four worker threads) rather than on the Tkinter main thread, so downloading a diagram from an external URL never freezes the user interface. Each URL is tracked in one of three sets — cached (successfully loaded), pending (download in progress), or failed (download attempted and unsuccessful) — and a completed download is marshalled back onto the main thread via the Tkinter after() scheduling mechanism, which then triggers a targeted redraw of the map only if the just-loaded image belongs to the currently selected event.

10.4 Reference-Image Prefetching

prefetch_beachballs() allows the calling code (invoked when an Excel focal-mechanism file is first loaded) to warm the cache for every event's beachball URL in the dataset up front, so that once the user begins clicking through events the corresponding diagrams are already downloaded or in flight, minimizing perceived latency during interactive exploration.

12. Practical Examples

12.1 Example 1 — Mapping Slip Tendency on a Digitized Fault Network

  1. Load a fracture-trace SVG via the workspace header's Load SVG button.
  2. Calibrate the profile scale (Scale button) against a known reference length in the source image.
  3. Enter appropriate principal stress magnitudes and orientation in the sidebar, then click Calculate.
  4. On the Map Tab, confirm the sidebar's Map Display mode is set to "Slip tendency" and "Color by normalized Tₛ" is checked.
  5. Segments rendered in the colour bar's upper range represent fractures whose resolved shear-to-normal stress ratio approaches the frictional threshold — the most mechanically favourable orientations for reactivation under the current stress state.
  6. Click a segment of interest in the Table tab to see it highlighted with the halo-and-core marker on the Map, and cross-check its exact Tₛ and Mohr-diagram position in the Mohr tab.

12.2 Example 2 — Visualizing a Legacy Fracture-Plane Spreadsheet

  1. Load an Excel/CSV file containing only ID, Strike, and Dip columns (no spatial coordinates).
  2. The Map Tab automatically falls back to the orientation-wheel renderer, since no genuine geometry is present.
  3. Set Map Display to "Fracture susceptibility" to visualize which orientations are closest to the Mohr-Coulomb failure envelope, independent of their (unknown) spatial position.
  4. Enable "Show compass (Wheel Map)" to retain the N/E/S/W frame of reference while interpreting tick azimuths as fracture strike.

12.3 Example 3 — Focal-Mechanism Regional Stress Screening

  1. Load a Global-CMT-style Excel file with Lat, Lon, FP1 Strike/Dip/Slip (and optionally FP2) columns.
  2. The Map Tab automatically renders a geographic scatter map; enter a candidate regional stress tensor and click Calculate.
  3. Set Map Display to "Critical fractures" to immediately identify which historical earthquakes are kinematically consistent with reactivation under the candidate stress field ("Critical", shown in the failure colour) versus which are not ("Stable", shown in the muted colour).
  4. Click an event marker to open its beachball diagram inset and inspect the Rake Misfit reported in the Mohr/Table tabs, which quantifies how well the entered stress tensor explains that earthquake's observed slip direction.

13. Geological Interpretation

13.1 Slip Tendency Maps

Spatial or angular clusters of high slip tendency indicate structural elements that are mechanically well-oriented for frictional reactivation under the assumed stress field. In fault-reactivation risk assessment, induced-seismicity screening, and reservoir-compartmentalization studies, such clusters are frequently used to prioritize which mapped structures warrant closer geomechanical scrutiny or avoidance during operations such as fluid injection.

13.2 Dilation Tendency Maps

High dilation tendency identifies fracture orientations most prone to opening (rather than shearing) under the current stress state. These orientations are of particular interest in fractured-reservoir permeability studies, since dilated fractures typically provide the dominant fluid-flow pathways, whereas fractures with low dilation tendency but high slip tendency are more likely to behave as shear (and potentially seismogenic) structures.

13.3 Fracture Susceptibility Maps

Fracture Susceptibility expresses proximity to the Mohr-Coulomb failure envelope in stress units (MPa) rather than as a dimensionless ratio, making it directly comparable to engineering pressure changes (e.g. planned pore-pressure increases during injection). Areas or orientation sets with low (near-zero or negative) susceptibility values are closest to failure and warrant the greatest operational caution.

13.4 Critical Fracture Maps

The binary Critical/Stable classification provides an unambiguous, decision-oriented summary suitable for risk communication to non-specialist stakeholders: it answers the direct question "is this structure currently at or beyond the failure threshold under the assumed stress state?" without requiring interpretation of a continuous colour gradient.

13.5 Orientation-Wheel Interpretation

Because the orientation wheel discards spatial position entirely, it is best interpreted as a structural-domain summary: clusters of ticks at particular azimuths with consistently high-risk colouring indicate that an entire joint or fault set — irrespective of where individual examples are located — is mechanically unfavourable under the current stress assumptions, which is often the more actionable finding for regional structural-domain characterization than any single mapped trace.

13.6 Focal-Mechanism Maps

Cross-referencing observed earthquake locations and nodal planes against a candidate regional stress tensor via the Critical Fractures colouring and Rake Misfit statistic (Wallace-Bott hypothesis; see the Method tab and the Dynamic Analyzer documentation) provides an independent, seismicity-based test of whether a proposed stress model is consistent with the historical earthquake record in the study area.

14.2 Design Notes and Caveats

  • The orientation-wheel colour bar is constructed against the same mappable artist object used by the geographic-mode scatter plot (self._map_geo_scatter); in practice this mappable carries the correct colormap and normalization for the wheel's own data because attribute_colormap_and_norm() is recomputed immediately beforehand, but users inspecting the implementation should be aware that this is an internal artist-reuse detail rather than a dedicated wheel-specific mappable.
  • Direct click-to-select on the map canvas itself is currently implemented only for geographic (focal-mechanism) mode; in Cartesian and orientation-wheel modes, segment selection is driven from the Table, Mohr, or Stereonet tabs and reflected onto the map via the shared select_segment() mechanism.
  • The helper method _expanded_map_limits(), which computes an aspect-ratio-matched bounding box from the figure and Axes display geometry, is defined on the MapTab class for potential use in aspect-correction workflows but is not invoked by the drawing routines described in this document; it is retained as available shared logic.
  • The Reverse Y-axis behaviour (SVG Y-axis points downward) is controlled by a Display-card checkbox on the shared sidebar rather than within the Map Tab itself, but directly affects the home_ylim orientation computed inside _draw_map_svg().

15. Limitations

  • The orientation wheel discards all spatial information; two fractures with identical strike and dip but located kilometres apart are rendered as coincident ticks.
  • Background-photograph alignment assumes the photograph's full pixel extent corresponds exactly to the calibrated data bounding box; a photograph that does not cover the same physical area as the digitized traces will not register correctly.
  • Beachball diagram rendering depends on network connectivity and the continued availability of the externally hosted image referenced by each event's Beachball URL; unreachable URLs degrade gracefully to an "Image unavailable" placeholder rather than an error.
  • Direct click-to-select is unavailable in Cartesian and orientation-wheel modes; selection in those modes must be initiated from another tab.
  • The raw-vs-scaled coordinate heuristic (Section 5.3) is a tolerance-based inference rather than an explicit calibration flag, and could in principle misclassify a genuinely calibrated profile whose dimensions happen to coincide closely with the uncalibrated 260 × 100 default.
  • Very large fracture populations rendered via a single LineCollection remain efficient to draw, but per-segment tooltip/hover interactivity is not implemented in this module (unlike the Mohr tab's point-hover annotation).
