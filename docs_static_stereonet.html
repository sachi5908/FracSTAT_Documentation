Static Stereonet Tab

Located at static_analyzer/tabs/static_stereonet_tab.py, the Stereonet Plot tab (implemented as the mixin class StaticStereonetTab) renders a lower‑hemisphere, equal‑area (Schmidt‑net) stereographic projection dedicated exclusively to the geometric description of fracture orientation data. Unlike the Dynamic Analyzer's stress‑resolved stereonet, the Static Analyzer's Stereonet Plot performs no mechanical calculation whatsoever: it is a purely descriptive, geometry‑only tool intended to reveal the orientation fabric of a fracture population — preferred strikes, dip magnitudes, clustering, girdle patterns, and conjugate set relationships — independent of any assumed stress tensor. Every fracture segment is rendered as a pole (the point representing the plane's normal vector), optionally overlaid with the corresponding great‑circle trace of the plane itself, and individual poles can be picked interactively to cross‑reference a specific fracture across the map, table, and every other Static Analyzer tab.

1. Interface Overview

The Stereonet Plot tab uses the same two‑column layout as every other Static Analyzer tab: a scrollable control sidebar (fixed width, approximately 290 px) on the left, and a plot card occupying the remaining horizontal space on the right. The stereonet is drawn on a standard Cartesian Matplotlib Axes (not a polar Axes) sized 9 × 7 inches at 100 DPI — the equal‑area transformation is computed manually in Python and the resulting Cartesian (x, y) coordinates are plotted directly, giving the module full control over grid styling, labelling, and the great‑circle plane curves that would be awkward to render on a native polar projection.

1.1 Sidebar Card Structure

The sidebar is subdivided into seven bordered, rounded‑corner "cards", built top‑to‑bottom in the following order:

1.3 Reset Defaults

A single Reset Defaults button at the base of the sidebar restores every styling variable (colours, sizes, toggles, custom title) to its factory value and immediately redraws the diagram. It does not clear loaded fracture data, dip overrides, or the current pole selection.

2. Data Input

2.1 Source of Fracture Data

The Stereonet Plot tab does not load files itself. It reads directly from self.results, the shared pandas DataFrame populated by the Static Analyzer's map‑geometry engine whenever an SVG trace map or an Excel/CSV orientation table is loaded via the workspace header's Load SVG / Load Excel buttons, or whenever the profile scale, Y‑axis azimuth, or reverse‑Y setting is changed and geometry is recalculated. Because every Static Analyzer tab consumes the same results table, the stereonet, the fracture map, the rose diagram, the CDF, and the results table are always mutually consistent for a given dataset.

2.2 Required and Optional Columns

ColumnRequiredDescription
StrikeDegYesFracture strike measured from North, stored as a fully‑directed value in the 0–360° domain. Folded to 0–180° internally for orientation statistics where direction of dip is irrelevant, but the full 0–360° value is retained wherever dip direction matters.
DipDegNo (defaults to 90°)Dip angle of the fracture plane, 0–90°. If absent, every segment is treated as a vertical fracture (90°) — the typical situation for a two‑dimensional SVG trace map with no independently measured dip.
DipDirectionNoOptional dip‑azimuth column present in some legacy Excel fracture‑plane files. When populated for a row, it overrides that row's strike via the right‑hand‑rule relationship strike = (DipDirection − 90°) mod 360°.

Length_mNoReal‑world segment length in the project's active display unit; used only for the "Length colour" plot mode and the Total Length statistic.

2.3 Excel/CSV Orientation Formats Accepted Upstream

Although the Stereonet Plot tab itself is format‑agnostic (it only ever sees the already‑parsed results table), it is useful to record which upstream input formats can populate StrikeDeg/DipDeg: (1) raw segment/results tables carrying explicit X1, Y1, X2, Y2 endpoints and a pre‑computed StrikeDeg; (2) the legacy fracture‑plane format (ID, Strike, Dip, optional Trace, Dip Direction, Length, Aperture); (3) pole‑orientation tables (Trend, Plunge), automatically converted to strike/dip; and (4) focal‑mechanism tables (Lat, Lon, FP1/FP2 Strike/Dip/Slip), used primarily by the Dynamic Analyzer but also readable here. Regardless of the source format, the Stereonet Plot tab treats every row identically once StrikeDeg and DipDeg exist in the results table.

2.4 Row‑Level Validation

On every redraw the module performs the following validation sequence before any geometry is computed:

  1. Strike values are read and wrapped into the 0–360° domain via a modulo operation, guaranteeing a well‑defined azimuth even if upstream data supplied negative or over‑range values.
  2. If a DipDeg column exists, dip values are read directly; otherwise every row is assigned a default dip of 90°.
  3. If a dip‑override dictionary is active (Section 3.3) and the results table carries an ID column, each row's dip is replaced with the corresponding override value wherever a finite override exists for that ID; rows without a matching override retain their original dip.
  4. If a DipDirection column is present and finite for a given row, that row's strike is recomputed from the dip‑azimuth relationship described above, superseding the StrikeDeg value for that row only.
  5. A combined finite‑value mask (numpy.isfinite) is applied to both strike and dip arrays; rows failing either test are dropped from the plotted set (and from the DataFrame passed onward to selection/statistics).
  6. Surviving dip values are clipped to the physically valid 0–90° range with numpy.clip, protecting the projection mathematics from out‑of‑range inputs that could otherwise produce a negative argument to a square root.

If, after this pipeline, zero rows remain, the module displays an explanatory empty‑state message instead of attempting to draw an empty or degenerate diagram (see Section 7.5).

3.3.1 File Parsing and Column Matching

Column headers are matched case‑insensitively and are tolerant of internal spaces, hyphens, and underscores: the header text is lower‑cased and every non‑alphanumeric character is stripped before comparison. The identifier column is matched against id, segmentid, fractureid, or fracture; the dip column is matched against dipdeg, dipdig, dipdegree, dipdegrees, fracturedip, fracturedipdeg, wholerockdip, or dip. If either column cannot be located, the load is rejected with an explanatory error dialog.

3.3.2 Validation Rules

  • Rows with a missing ID or DipDeg value are dropped via pandas.DataFrame.dropna before processing.
  • Every surviving DipDeg value must lie within the closed interval [0°, 90°]; if any row violates this bound, the entire file is rejected with a descriptive error and no partial override is applied.
  • The identifier column is cast to Python int and the dip column to float; the resulting {segment_id: dip_value} dictionary is stored as self.static_stereo_dip_overrides and is also written into the launcher's cross‑workspace shared_Analyzer dictionary so that a later switch to the Dynamic Analyzer or a project reload can, where applicable, see the same override set.
  • The status label beneath the buttons is updated to read "Loaded dips for N IDs." on success.

4. Mathematical Background

4.1 The Lower‑Hemisphere Equal‑Area (Schmidt) Projection

A stereonet is a two‑dimensional projection of the lower half of a reference sphere (the "projection hemisphere") used throughout structural geology to plot the orientation of lines and planes. FracSTAT implements the equal‑area (Schmidt, or Lambert azimuthal equal‑area) variant rather than the equal‑angle (Wulff) variant, because equal‑area projections preserve angular area — a critical property for any subsequent density‑contouring or population‑statistics work, since it guarantees that a uniformly random population of orientations produces a uniform point density on the diagram.

For a line with trend T (measured clockwise from North, 0–360°) and plunge P (measured downward from horizontal, 0–90°, positive), the module computes the projected Cartesian position (x, y) with the following transformation, implemented in the static method _static_stereo_line_point:

colatitude θ = 90° − P radius r = √2 · sin(θ / 2) x = r · sin(T) y = r · cos(T)

This is the standard Lambert azimuthal equal‑area formula expressed in colatitude form. At P = 90° (a vertical line, plunging straight down), θ = 0° and r = 0, placing the point at the centre of the net, exactly as expected for a line piercing the lower hemisphere at its pole. At P = 0° (a horizontal line), θ = 90° and r = √2·sin(45°) = 1, placing the point exactly on the unit‑radius bounding circle.

4.2 From Plane Strike/Dip to Pole Position

Every fracture segment in the results table is characterised by a strike (the compass bearing of the horizontal line of intersection between the fracture plane and a horizontal reference plane) and a dip (the maximum angle of inclination of the plane, measured downward from horizontal, perpendicular to strike). The pole of a plane is the unit vector normal to that plane; plotting poles rather than the planes themselves is the conventional way to visualise large fracture populations, since a cluster of points is far easier to read than a cluster of overlapping great‑circle arcs.

The relationship between a plane's strike/dip and its pole's trend/plunge, using the right‑hand rule convention adopted throughout FracSTAT (dip direction = strike + 90°), is:

pole trend = (strike − 90°) mod 360° pole plunge = 90° − dip

Substituting these into the colatitude form of Section 4.1 gives θ = 90° − pole plunge = dip, so that the pole radius can be computed directly from dip without an intermediate plunge calculation:

pole radius r = √2 · sin(dip / 2) x = r · sin(pole trend) y = r · cos(pole trend)

This is the exact computation performed for every row of the results table inside _draw_static_stereonet, applied as a fully vectorised NumPy operation across the entire (filtered) fracture population in a single pass rather than looping row by row.

4.3 Great‑Circle Trace of a Plane

When "Show planes" (or "Show selected plane only") is enabled, the module draws the complete great‑circle trace of each plane — the projected image of every point on the plane that intersects the reference sphere — rather than only its single pole point. The static method _static_stereo_plane_curve constructs this curve geometrically as follows, for a plane of strike S and dip D:

  1. Two orthonormal in‑plane basis vectors are built in an (East, North, Up) coordinate frame: a strike vector s = (sin S, cos S, 0) lying along the horizontal strike line, and a down‑dip vector d = (sin(S+90°)·cos D, cos(S+90°)·cos D, −sin D) pointing down the dip direction into the plane.
  2. A parameter t is swept across 140 equally spaced values from 0 to π radians, and for each value a unit vector on the plane is formed by v(t) = s·cos(t) + d·sin(t). Sweeping only 0 to π (rather than the full 0 to 2π) is sufficient because the lower‑hemisphere projection is symmetric: t and t+π map to antipodal points that, for a plane's great circle, coincide with the same lower‑hemisphere trace.
  3. Each direction vector v(t) = (v_E, v_N, v_U) is converted to a plunge and trend: plunge = arcsin(−v_U), trend = atan2(v_E, v_N).
  4. Every (trend, plunge) pair is projected with the same equal‑area formula as Section 4.1: radius = √2·sin((90° − plunge)/2), x = radius·sin(trend), y = radius·cos(trend).

The resulting 140‑point polyline, plotted with Matplotlib's standard line‑drawing routine, renders as the smooth, gently curved great‑circle arc familiar from printed stereonet overlays. Because the whole curve is generated with vectorised trigonometric NumPy calls, its cost is negligible even for a moderately sized fracture population.

4.4 Circular (Axial) Mean of Strike

Strike is an axial (180°‑periodic) rather than a directional (360°‑periodic) quantity: a fracture striking 010° is geologically identical to one striking 190°. A naive arithmetic mean of strike angles is therefore meaningless whenever a population straddles the 0°/180° wrap‑around boundary (for example, strikes of 179° and 1° should average to approximately 0°/180°, not 90°). FracSTAT resolves this with the standard doubling‑angle circular‑statistics technique, implemented in _static_stereonet_axial_mean:

ā = 0.5 · atan2( Σ wᵢ sin(2θᵢ) , Σ wᵢ cos(2θᵢ) ) mod 180°

where θᵢ is each fracture's strike folded to the 0–180° domain, and wᵢ is a uniform‑weight contribution factor (the Statistics card currently applies uniform weighting rather than a length‑weighted option). Doubling every angle before applying the standard circular‑mean formula collapses the 180° periodicity into a 360° periodicity, allowing the ordinary vector‑sum trigonometric identity to be applied; halving the resulting angle and folding it back into 0–180° recovers the correct axial mean.

5. Algorithms

5.1 Data Preparation Pipeline

Every invocation of the draw routine (_draw_static_stereonet) begins by re‑deriving the plottable strike/dip arrays from scratch rather than caching a previous computation, ensuring the diagram always reflects the current state of dip overrides, DipDirection substitutions, and the active selection. The pipeline is:

  1. Read StrikeDeg and, if present, DipDeg from self.results; wrap strike into 0–360° via a modulo operation and default missing dip data to a constant array of 90°.
  2. Apply the dip‑override dictionary, matched by integer ID, using numpy.where to substitute only the rows for which a finite override value exists, leaving all other rows untouched.
  3. Apply any DipDirection‑derived strike override, again via numpy.where, only for rows where DipDirection is finite.
  4. Build a combined boolean validity mask with numpy.isfinite on both strike and dip arrays; index both arrays and the results DataFrame with this mask to obtain the final plottable subset (df_valid).
  5. Clip the surviving dip values into [0°, 90°] with numpy.clip as a final safety net against any residual out‑of‑range values.

5.2 Colour Resolution

Colour resolution is a two‑stage lazy lookup designed to avoid repeatedly importing the Dynamic Analyzer's colour‑scale catalogue module on every redraw. The user‑facing dropdown stores a human‑readable display name (e.g. "Viridis"); a companion callback (_on_cmap_change), fired whenever the dropdown selection changes, resolves that display name through the shared colour‑scale definition catalogue into (a) a genuine Matplotlib colormap identifier string and (b) — for the single special case of "Parula", which has no native Matplotlib equivalent — a bespoke LinearSegmentedColormap built from six manually specified hex‑colour control points. The resolved identifier is cached in self.static_stereo_cmap_resolved so that subsequent redraws (_static_stereo_cmap) can retrieve the ready‑made colormap object without re‑parsing the dropdown text or re‑touching the catalogue.

5.3 Pole Colour‑Value Selection

The helper _static_stereo_values(df, strikes, dips) is a small dispatcher that returns a (values, axis_label, Normalize) triple appropriate to the current Plot Mode, implementing the normalisation ranges tabulated in Section 4.5. When the plot mode is Dip colour, it returns the dip array directly. When it is Length colour and a Length_m column exists, it substitutes any non‑finite length with 0.0 (via numpy.where) before computing the data‑driven maximum, preventing a single corrupt or missing value from breaking the colour scale for the entire population. For every other case (including the default Strike colour fallback used whenever Length colour is selected but no length data exists) it returns the strike array folded into the 0–180° domain.

5.4 Selective Plane Rendering

The module draws great‑circle traces in one of three mutually exclusive states, resolved before the main pole scatter is built: (a) if "Show planes" is on and "Show selected plane only" is off, every valid fracture's great circle is drawn in a single loop using the shared Plane Color/width/opacity; (b) if "Show selected plane only" is on and a fracture is currently selected, only that fracture's great circle is drawn, in the Selected Pole colour at an emphasised (1.8×) line width and full 0.95 opacity, as part of the selection‑highlight sequence described below; (c) if neither toggle is active, no plane curves are drawn at all, leaving only the pole scatter.

5.5 Selection Highlight Rendering and Cross‑Tab Synchronisation

When self.selected_segment_id matches an ID present in the filtered dataset, the module renders a four‑layer highlight stack directly on top of the ordinary pole scatter, in ascending z‑order: (1) a large, thick, white‑edged open circle ("halo") of size max(180, pole_size×4.0); (2) a slightly smaller dark‑edged open circle of size max(125, pole_size×2.8); (3) a filled "P"‑shaped (plus/star) marker in the Selected Pole colour, sized max(62, pole_size×1.6); and (4) a text annotation reading "ID {n}" offset ten points to the upper right, rendered inside a rounded, semi‑opaque background box for legibility against any underlying grid or pole cluster. This layered halo approach guarantees the selected fracture remains visually unambiguous even in a dense, highly overlapping pole cloud.

Picking is implemented via Matplotlib's native artist‑picking API rather than a manual hit‑test: the pole scatter artist is created with picker=True and pickradius=6 (a six‑pixel tolerance radius around each rendered marker centre), giving a consistent, predictable click target in screen‑pixel space regardless of how many poles are compressed into the net. Clicking a pole fires Matplotlib's pick_event, handled by _on_static_stereonet_pick, which verifies the picked artist matches the current pick artist reference, reads the ID of the first matched index, and assigns it to self.selected_segment_id — the same shared attribute used by the Fracture Map tab and (for orientation‑only Excel data) the Results tab. Immediately afterward, if the Fracture Map's own draw method exists on the instance, it is invoked so the map's highlight ring updates in lockstep with the stereonet's, and finally the stereonet itself is redrawn. Clear Selection (_clear_static_stereonet_selection) performs the symmetric operation: it nulls the selection, clears the info panel, and triggers the same two‑tab redraw.

5.6 Stereonet Frame Construction

The frame — the fixed scaffold of circles, grid lines, and labels that gives the plot its stereonet appearance — is built by _format_static_stereonet and is invoked twice per redraw: once at the start of _draw_static_stereonet (to establish axis limits before any data is plotted) and once again at the end (so that frame elements such as the bounding circle are guaranteed to sit visually above the pole/plane layers, since Matplotlib's default z‑ordering follows draw order). The construction sequence is:

  1. Fix the Axes aspect ratio to 1:1 and the view limits to [−1.15, 1.15] on both axes, then disable all default Matplotlib axis chrome (axis('off')).
  2. Draw the unit‑radius bounding circle using 361 uniformly sampled angle steps.
  3. If grid rings are enabled, draw three concentric dashed circles at fractional radii 0.25, 0.5, 0.75 and twelve 30°‑spaced spokes from the origin.
  4. If the crosshair toggle is enabled, draw the two orthogonal N‑S and E‑W diameters.
  5. Render either full 30°‑spaced tick/label sets (optionally cardinal‑letter‑substituted at the four principal directions) or, when the 30°‑labels toggle is off but the cardinal toggle remains on, just the four N/E/S/W letters positioned outside the circle.

5.7 Main Draw Routine — Complete Sequence

The complete draw sequence executed by _draw_static_stereonet, in order, is:

  1. Guard clause: if the Matplotlib Axes has not yet been created (tab not yet built), return immediately.
  2. Read all styling variables (colours, widths, sizes) from their CTk StringVar/BooleanVar containers, with defensive float() parsing wrapped in try/except so that a malformed numeric entry silently falls back to its documented default rather than raising an exception that would freeze the UI.
  3. Clear the primary Axes and the colour‑bar Axes; hide the colour‑bar Axes by default (it is only re‑shown later if the active plot mode actually needs one).
  4. Set the Figure and Axes background colours from the Outer BG / Plot BG swatches.
  5. Call _format_static_stereonet once to establish the frame, boundary circle, and optional grid/crosshair/label elements.
  6. Guard clause: if self.results is empty, missing, or lacks a StrikeDeg column, render the empty‑state message and return.
  7. Execute the full data‑preparation pipeline of Section 5.1.
  8. Guard clause: if zero rows survive validation, render an alternative empty‑state message ("No valid strike/dip data found") and return.
  9. Resolve the active colormap via _static_stereo_cmap().
  10. Branch on plot mode: for Solid colour, set a flat pole_colors value and skip the colour‑bar entirely; otherwise call _static_stereo_values to obtain per‑pole colour values, build a ScalarMappable for the colour bar, and colour each pole via cmap(norm(values)).
  11. If plane rendering is requested (case (a) of Section 5.4), loop over every valid row and plot its great‑circle trace before the poles are drawn, so poles visually sit on top of the plane curves.
  12. Compute pole trend, pole radius, and Cartesian (x, y) for the entire valid population in one vectorised block (Section 4.2).
  13. If "Show poles" is enabled, draw the full pole scatter with picker=True and pickradius=6, storing the returned artist for the pick_event handler to recognise; otherwise set that artist reference to None, which also disables picking.
  14. If a selection is active and its ID is present in the valid subset, execute the selection‑highlight and (optionally) selected‑plane‑only rendering described in Section 5.4/5.5, and populate the Selected Pole info panel.
  15. Call _format_static_stereonet a second time, so frame elements are guaranteed to be layered above the data.
  16. Configure the colour bar via _setup_static_stereo_colorbar (Section 7.4).
  17. Set the plot title (custom or auto‑generated, embedding the live fracture count n).
  18. Call _static_stereonet_update_stats to refresh the Statistics card.
  19. Request a canvas redraw via draw_idle(), which defers the actual paint operation to Tkinter's next idle cycle for smoother responsiveness under rapid successive control changes.

The module additionally exposes a public refresh_static_stereonet() hook, called by the Static Analyzer's shared refresh_all_tabs sequencing routine whenever new geometry has been calculated (a new file load, a profile‑scale change, or a global font/appearance change), guaranteeing the stereonet stays synchronised with the rest of the workspace without the tab needing to poll for changes itself.

6. Statistical Methods

6.1 Summary of Implemented Statistics

StatisticTypeFormula / MethodInterpretation
Planes (n)CountRow count of the validated subsetEffective sample size underlying every other statistic and the colour normalisation.
Mean StrikeCircular (axial) mean, 180°‑periodicDoubling‑angle vector‑resultant method, Section 4.4The population's dominant fracture‑strike orientation; robust to the 0°/180° wrap‑around ambiguity inherent to axial (non‑directional) strike data.
Mean DipLinear (arithmetic) meannumpy.nanmean(dip array)Average steepness of the fracture population; a linear mean is statistically appropriate because dip is bounded (0–90°) and non‑circular.
Total LengthSumΣ Length_m across valid rowsAggregate mapped fracture length in the dataset's active display unit; contextualises the population size in physical (as opposed to purely count‑based) terms.

6.2 Why Circular Statistics Are Required for Strike

Ordinary (linear) descriptive statistics assume that the underlying measurement scale is unbounded and that a small numerical difference always corresponds to a small physical difference. Compass bearings and, more specifically, axial strike measurements violate this assumption at the wrap‑around boundary: 179° and 1° are numerically 178° apart but geologically almost coincident (both describe an approximately N–S striking fracture set, once the 180° axial periodicity is accounted for). Using an arithmetic mean on such data can produce a wildly misleading result — in the worst case, a mean strike diametrically opposed to the true dominant orientation. The doubling‑angle technique used here (Section 4.4) is the standard remedy from directional statistics and is used consistently across FracSTAT's Rose Plot, Static Stereonet, and Results‑tab summary statistics.

6.3 Statistics as a Quality‑Control Tool

Because the Statistics card recomputes on every redraw, it functions as a live sanity check for the Dip Override workflow: after loading an override file, a user can immediately verify that the Mean Dip value has shifted in the expected direction and that the Planes count still matches the expected fracture population size, catching upload errors (mismatched IDs, wrong‑unit dip values, or a partially failed match) before the data is carried forward into any downstream stress analysis in the Dynamic Analyzer.

7.5 Title Construction and Empty/Error States

If the Custom Title field is left blank, the module auto‑generates a title of the form "Static Stereonet: planes and poles (n={count})", where {count} is the live number of valid, plotted fracture poles — giving the reader an immediate population‑size context without needing to consult the separate Statistics card.

Note: two distinct empty‑state messages are used depending on the failure point: "Load SVG or Excel fracture data" is shown when self.results is absent, empty, or lacks a StrikeDeg column entirely (i.e. no dataset has been loaded at all); "No valid strike/dip data found" is shown when a dataset is loaded but every row failed the finite‑value validation of Section 2.4. In both cases the colour bar is hidden, the Selected Pole panel is cleared, and every Statistics field is reset to a dash placeholder, so the sidebar never displays stale numbers alongside an empty plot.

7.6 Image Export

The Save Image toolbar button calls the Static Analyzer's shared save_plot_image routine, which writes the current Figure to a timestamped PNG file inside a per‑project FracSTAT_plot_exports subfolder, using the project‑configured export DPI (default 300, adjustable in the application‑wide Settings page). Because the export routine saves the live Matplotlib Figure object directly, the exported image is pixel‑for‑pixel identical to the on‑screen render, including every custom colour, label, and selection highlight active at the moment of export.

8. Step‑by‑Step Example Workflows

8.1 Example 1 — Basic Orientation Review of an SVG Trace Map

  1. Load a digitised fracture trace map via the workspace header's Load SVG button. Every segment initially receives a default dip of 90°.
  2. Open the Stereonet Plot tab. With Plot Mode left at its default Solid colour, every pole appears at the outer boundary circle (since dip = 90° corresponds to pole radius = 1), forming a ring pattern that directly visualises the strike distribution around the compass.
  3. Switch Plot Mode to Strike colour and enable Show grid rings and Show 30° labels to add angular reference context.
  4. Read the Statistics card: the Mean Strike value reports the population's dominant azimuth using the axial circular mean, and the Planes count confirms the number of segments successfully digitised.

8.2 Example 2 — Incorporating Field‑Measured Dips

  1. Prepare a two‑column spreadsheet with headers ID and DipDeg, where ID values match the segment identifiers shown in the Static Analyzer's Table/Results tab, and DipDeg holds field‑measured dip angles in the 0–90° range.
  2. In the Dip Override card, click View Format to confirm the expected layout, then click Load Dip Override and select the prepared file.
  3. Observe that the pole ring on the stereonet collapses inward toward the centre for every overridden segment, now correctly reflecting non‑vertical dip, while segments without a matching override remain on the outer boundary.
  4. Switch Plot Mode to Dip colour to visually audit which segments received an override (moved inward, showing a graded colour) versus which retained the 90° default (fixed at the boundary colour extreme).

8.3 Example 3 — Identifying Conjugate Fracture Sets

  1. With a fully dip‑populated dataset loaded, enable Show planes to overlay every great‑circle trace.
  2. Visually identify two angularly separated pole clusters; enable Show selected plane only and click a representative pole from each cluster in turn (the module keeps only the most recently picked selection, so compare one set at a time) to isolate and inspect each set's individual great circle against the full population.
  3. Record the two clusters' mean orientations (readable from the Selected Pole panel for a representative fracture in each, or by filtering the source data and re‑loading a subset) and compute their angular separation to assess whether the pattern is consistent with a conjugate shear‑fracture pair.

8.4 Example 4 — Publication‑Quality Figure Preparation

  1. Set a Custom title describing the dataset (for example, "Outcrop A — Joint Set Orientation, n = 248").
  2. In the Colours card, select a colour palette consistent with the target publication's style guide (for example, switching Outer BG and Plot BG to white, and choosing a colour‑blind‑friendly colormap such as Viridis or Cividis for Length colour mode).
  3. Set Colorbar position to Right for a compact, journal‑column‑friendly layout, and adjust Pole size and Boundary width for the target print resolution.
  4. Click Save Image to export a high‑resolution PNG at the project's configured export DPI directly into the project's plot‑exports folder.

9. Geological Interpretation

9.1 Reading Pole Clusters

A tight cluster of poles indicates a single, well‑defined fracture set with consistent orientation — the geological signature of a systematic joint set formed under a single, spatially homogeneous stress or strain event. Multiple, angularly separated clusters typically indicate two or more distinct fracture sets, which may represent conjugate shear fractures (commonly separated by roughly 60–90° depending on the internal friction angle of the rock), successive tectonic events with rotated principal stress axes, or a combination of systematic and non‑systematic (cross) joints.

9.2 Girdle (Great‑Circle) Distributions

When poles are dispersed along a great‑circle arc rather than clustered at a point, the population is described as a girdle distribution. Geologically, a girdle commonly indicates that the fracture planes share a common axis of rotation or intersection — for example, fractures related to folding about a single fold axis, where the pole girdle's own great‑circle axis (its π‑pole) closely approximates the plunge and trend of the fold hinge. The Show planes toggle, by drawing the actual great‑circle traces rather than only their poles, allows a user to visually confirm whether a suspected girdle arrangement of poles is indeed produced by planes sharing a common intersection line.

9.3 Interpreting Colour‑Coded Attributes

  • Strike colour (0–180°, cyclic‑appearing but linearly mapped) highlights orientation domains within a single diagram — useful for visually separating conjugate or cross‑cutting sets without needing a second plot.
  • Dip colour (fixed 0–90° scale) is particularly informative for datasets combining vertical, SVG‑derived trace‑map fractures (typically defaulted to 90°) with dip‑override or directly measured fracture‑plane data, letting a user instantly distinguish which poles carry genuinely measured dip information versus assumed‑vertical defaults, since the latter will cluster tightly at the deep‑red or deep‑blue end of the fixed scale depending on the chosen colormap.
  • Length colour reveals whether the longest, most laterally persistent fractures in the network share a common orientation with the shorter, more numerous population — a pattern often associated with a hierarchy of fracture generations (e.g. long master joints versus short cross joints).

9.4 Relationship to the Dynamic Analyzer

The purely geometric orientation fabric characterised here is the natural precursor to the Dynamic Analyzer's stress‑based slip‑tendency and Mohr‑Coulomb failure analysis: a fracture set identified as a tight, well‑defined cluster on this stereonet is a strong candidate for a mechanically significant, potentially reactivatable set once a stress tensor is applied, whereas a poorly clustered or highly scattered population may indicate either genuine geological complexity (multiple superposed events) or a data‑quality issue (digitisation noise, an inappropriate default dip assumption) that should be resolved — for example via the Dip Override mechanism — before committing to a quantitative stress analysis.

9.5 Caution Regarding Default (Assumed) Dips

Note: any fracture segment lacking an explicit DipDeg value in the source data, or lacking an entry in an uploaded Dip Override file, is plotted at the default dip of 90° (a purely vertical plane). This default is a pragmatic necessity for two‑dimensional SVG trace‑map digitisation, where true three‑dimensional dip cannot be measured from a plan‑view photograph alone, but it must not be misread as a geological observation. Any population containing a mixture of measured and default‑vertical dips will show a spurious concentration of poles at the net's outer boundary (dip = 90° ⇒ pole plunge = 0° ⇒ pole radius = 1) that reflects a digitisation convenience rather than fracture mechanics.

10. Performance Optimization

10.1 Vectorised NumPy Computation

Every geometric transformation described in Section 4 — strike wrapping, dip‑override substitution, pole trend/radius computation, and coordinate projection — is expressed as array‑level NumPy operations (modulo, where, clip, sin, cos, sqrt) applied to the entire fracture population simultaneously, rather than iterating row by row in pure Python. For typical FracSTAT datasets (tens to a few thousand fracture segments), this reduces the per‑redraw geometric computation to a negligible fraction of the overall render time, which is instead dominated by Matplotlib's own artist construction and canvas paint operations.

10.2 Great‑Circle Curve Cost Management

The great‑circle plane‑trace algorithm (Section 4.3) samples 140 points per plane, chosen as a balance between visual smoothness (no perceptible faceting on a typical screen or print render) and per‑plane computational and memory cost. Because plane rendering is opt‑in (Show planes defaults to off) and is itself vectorisable per plane, enabling it for populations of a few hundred fractures remains responsive; very large populations (several thousand fractures with planes enabled simultaneously) will show a proportional increase in draw time, and users working with such datasets are advised to use Show selected plane only, or to filter/subset the input data, rather than rendering every great circle at once.

10.3 Deferred Canvas Painting

The module calls the Matplotlib canvas's draw_idle() method rather than the blocking draw() at the end of every redraw. draw_idle() schedules the actual paint operation on Tkinter's event‑loop idle queue instead of forcing it synchronously, which allows several rapid, successive control changes (for example, quickly typing into a numeric entry field or dragging a colour picker) to coalesce into fewer physical repaints, improving perceived UI responsiveness without any explicit debounce logic in this module.

10.4 Colormap Resolution Caching

As described in Section 5.2, the resolved Matplotlib colormap identifier is cached in self.static_stereo_cmap_resolved after the dropdown's on‑change callback fires, so the hot draw path (_static_stereo_cmap) never needs to re‑parse the display name or re‑touch the shared colour‑scale catalogue on every redraw — it performs, at most, a single dictionary‑free conditional check and a call to matplotlib.cm.get_cmap.

10.5 Colour‑Bar Axis Reuse and I/O

Rather than creating and destroying a Matplotlib Axes object every time the colour bar is shown or hidden (an operation with non‑trivial overhead, since Axes creation touches the Figure's internal layout engine), the module creates the colour‑bar Axes exactly once, at tab‑construction time, and thereafter only toggles its visibility and repositions its bounding rectangle — consistent with the same pattern used across every other Static Analyzer plotting tab. The Stereonet Plot tab also performs no file or network I/O during normal redraw operation; the only I/O paths are the explicit, user‑initiated Load Dip Override file dialog and the Save Image export, both one‑shot operations outside the interactive redraw loop.

11. Advanced Features

11.1 Pole Marker Shape Library

Six marker shapes are exposed, mapped internally to Matplotlib single‑character/style codes:

Display NameMatplotlib Marker CodeTypical Use

11.2 Dip‑Direction‑Aware Strike Correction

Certain upstream Excel fracture‑plane files record a plane's orientation via dip azimuth ("Dip Direction") rather than strike. Rather than requiring the user to pre‑convert this column, the Stereonet Plot tab applies the standard right‑hand‑rule inversion strike = (Dip Direction − 90°) mod 360° on a per‑row basis, silently overriding StrikeDeg only for the rows where a finite Dip Direction value is present. This allows a single results table to mix rows originating from strike‑based and dip‑direction‑based source files without any manual pre‑processing.

11.3 Reset Defaults

The reset routine (_static_stereonet_reset_defaults) writes the factory value into every corresponding StringVar/BooleanVar in a single pass — Plot mode to Solid colour, Colorbar position to Bottom, all colour swatches to their documented defaults, Pole shape to Circle and Pole size to 30, Plane width/opacity to 0.85/0.35, Show poles on with planes and selected‑plane off, grid rings off with crosshair/30°‑labels/cardinal‑labels on, and the Custom title cleared — then performs one redraw.

11.4 Shared Colour‑Scale Ecosystem

The colormap dropdown and its live‑swatch preview are implemented via the same CustomColorDropdown widget and colour‑scale catalogue used throughout FracSTAT (Dynamic Analyzer's Map/Rose/Mohr/Stereonet tabs, and the Static Analyzer's own Fracture Map, Intensity & Density, and Rose Plot tabs). This guarantees that a colour scale chosen here has visually identical appearance and semantics wherever else it is used in the application, and that any future addition to the shared catalogue automatically becomes available in this tab's dropdown without any change to this module.

11.5 Cross‑Workspace Dip Override State

Although the Dip Override loaded through this tab is scoped to the Static Analyzer's Stereonet Plot display, the resulting dictionary is also written into the launcher's shared shared_Analyzer['dip_overrides'] state slot. This is the same slot the Dynamic Analyzer's own, independently maintained dip‑override mechanism reads from when hydrating a freshly opened or mode‑switched workspace, meaning a dip override uploaded here can, depending on the exact sequence of workspace switches, become visible as a starting point in the Dynamic Analyzer as well — a convenience for users moving directly from geometric review to stress analysis, though it should not be relied upon as the primary path for setting Dynamic Analyzer dip overrides, which has its own dedicated upload control.

12. Limitations

LimitationDescription
No stress resolutionThe module performs no stress resolution, slip‑tendency, or Mohr‑Coulomb failure calculation; it is strictly a geometric/orientation visualisation tool. Users requiring stress‑based analysis must use the Dynamic Analyzer's own stereonet, which computes and colours poles by resolved mechanical attributes rather than raw geometric properties.
Fixed default vertical dipDip values for fractures digitised purely from a two‑dimensional SVG trace map default to a fixed 90° (vertical) unless explicitly supplied via a DipDirection column in the source data or a Dip Override upload; the module cannot infer true three‑dimensional dip from plan‑view geometry alone.
Single active selectionThe pole‑selection mechanism retains only a single active selection at a time (self.selected_segment_id is a scalar, not a set); comparing more than one fracture's exact numeric attributes side by side requires switching selection back and forth or consulting the Results/Table tab directly.
Unweighted circular statisticsMean Strike and Mean Dip are unweighted (uniform‑weight) statistics; the module does not currently offer a length‑weighted or area‑weighted circular mean option on this tab, in contrast to the Rose Plot tab, which does support length weighting.
Great‑circle rendering cost at scaleGreat‑circle plane rendering for very large fracture populations (several thousand planes rendered simultaneously) can visibly slow the redraw, since each plane's curve requires its own 140‑point trigonometric sampling and a separate Matplotlib line‑drawing call; no automatic thinning or level‑of‑detail reduction is implemented for this case.
Flat Dip Override file formatThe Dip Override file format supports only a flat ID/DipDeg pairing; it cannot express per‑segment strike overrides, per‑segment uncertainty ranges, or multiple candidate dip values for the same segment ID.
Fixed lower‑hemisphere, equal‑area projectionThe projection is fixed to the lower hemisphere, equal‑area convention; there is no user‑facing option to switch to an upper‑hemisphere or equal‑angle (Wulff) projection within this tab.
Fixed dip‑colour normalisationColour normalisation for Dip colour mode is fixed to the full 0–90° range regardless of the actual data spread, which can compress the visible colour contrast for datasets whose dip values occupy only a narrow sub‑range (for example, a population confined to 60–70°); no auto‑scaling toggle is provided for this mode, unlike Length colour.
