Dynamic Table Tab

Located at dynamic_analyzer/tabs/dynamic_table_tab.py, the Table tab (implemented as the mixin class TableTab) is one of six coordinated views within the Dynamic Analyzer workspace of FracSTAT (the others being Map, Rose, Mohr, Stereonet, and Method). Where the Map, Rose, Mohr, and Stereonet tabs render the results of a stress-resolution calculation graphically, the Table Tab presents the identical per-segment result set in a searchable, sortable-by-inspection, row-and-column format. It is mixed into the top-level SlipTendencyApp window alongside the other tab mixins (MapTab, RoseTab, MohrTab, MethodTab, StereonetTab).

The scientific purpose of the Table Tab is threefold. First, it provides an auditable numerical record of every quantity the Dynamic Analyzer computes for every input fracture segment or fracture plane — orientation, resolved stress, and the four principal geomechanical attributes (slip tendency, dilation tendency, fracture susceptibility, and the Coulomb critical-stress flag) — so results can be verified, spot-checked against hand calculations, or cross-referenced with the underlying reference implementation in calculator.py. Second, it acts as the authoritative tabular export source: the "Export Excel" action writes the same DataFrame that populates the table directly to a Microsoft Excel workbook for use in external reporting, statistical post-processing, or inclusion in a technical report. Third, and importantly for the interactive character of the workspace, the Table Tab functions as a selection hub: clicking a row highlights the corresponding fracture segment simultaneously on the Map, in the Mohr diagram, and on the Stereonet, letting the analyst move fluidly between an abstract stress-space representation and its concrete geometric counterpart.

1. Position within the FracSTAT Architecture

FracSTAT is organised around a Launcher/Home screen (launcher.py) that hosts two principal analytical workspaces embedded as CustomTkinter frames: the Static Analyzer (geometric fracture-network characterisation) and the Dynamic Analyzer (stress-based slip-tendency analysis, implemented in dynamic_app.py as the SlipTendencyApp class). A third, Uncertainty Analyzer module is present as an under-construction placeholder. The Table Tab belongs exclusively to the Dynamic Analyzer workspace.

Within SlipTendencyApp, six named CTkTabview tabs are constructed in a fixed order — Map, Rose, Mohr, Stereonet, Table, Method — each built by a dedicated _build_*_tab method contributed by its mixin class. All six tabs read from two shared, workspace-level attributes maintained by SlipTendencyApp:

  • self.results — a pandas DataFrame produced by calculator.run_calculation(), containing one row per fracture segment (SVG-derived) or per fracture plane (Excel/CSV-derived), refreshed every time the user presses Calculate or edits an input that triggers recomputation.
  • self.selected_segment_id — an integer (or None) identifying the segment currently highlighted across every tab, mutated by SlipTendencyApp.select_segment() and SlipTendencyApp.clear_selection().

The Table Tab reads self.results to populate its rows and reads/writes self.selected_segment_id indirectly through the shared select_segment() method, making it functionally symmetrical with the picking/selection logic already present in the Map, Mohr, and Stereonet tabs (each of which also supports row/point selection that calls the same select_segment() entry point).

2. Interface Overview

The Table Tab occupies the full main-content region of the Dynamic Analyzer window beneath the shared workspace header (font/colour controls, Load SVG/Load Excel, Save Project, Move Elements, Scale, Switch Mode, Back) and to the right of the collapsible sidebar (Profile & Units, Stress & Orientation, Display, Actions, Results summary, Selected Segment panel — all shared with the other tabs). The tab's own layout consists of three stacked regions arranged on a two-column, three-row Tk grid:

  • Row 0 — Toolbar: a transparent frame containing the section label "Segment results" (left-aligned, bold) and an "Export Excel" button (right-aligned).
  • Row 2 — Data grid: a ttk.Treeview widget (columns-only view, i.e. no expandable tree hierarchy) occupying column 0, paired with a vertical ttk.Scrollbar in column 1.
  • (Row 1 is intentionally left empty as spacing in the underlying grid definition.)

The Treeview is configured with row weight 1 on grid row 2 and column weight 1 on grid column 0, so the table area expands to fill all available vertical and horizontal space as the window is resized; only the scrollbar column remains fixed-width.

3. Data Input and Underlying Data Model

The Table Tab performs no calculation of its own. It is a pure presentation layer over self.results, the DataFrame assembled by calculator.run_calculation() every time SlipTendencyApp.calculate() executes (triggered by pressing "Calculate", pressing Enter in a sidebar field, loading a new SVG or Excel/CSV file, changing the profile scale, or applying a dip-override file). The DataFrame's construction pathway differs depending on the data source:

  • SVG source — segments are read by svg_loader.read_svg_segments(), converted from SVG pixel coordinates to physical (metre-equivalent) coordinates using the profile width/length calibration, and their strike is derived from segment geometry via models.segment_strike() and north-corrected via models.north_corrected_strike().
  • Excel/CSV source — segments are read by excel_loader.read_excel_segments(), which auto-detects one of five input schemas (raw XY segment coordinates, legacy Strike/Dip fracture planes, Trend/Plunge pole data, focal-mechanism nodal-plane tables, or FracSTAT's own exported results format) and populates the Segment dataclass's _preset_strike/_preset_dip fields directly, bypassing pixel geometry entirely.

Regardless of source, every row passed through calculator.run_calculation() is resolved against the current principal-stress tensor (σ1, σ2, σ3), its orientation mode (Andersonian or fully general 3-D), and the Mohr–Coulomb failure parameters (cohesion C₀, pore pressure Pf, friction coefficient μ) that are set in the sidebar's Stress and Orientation card. The resulting DataFrame therefore couples pure geometry (position, strike, dip, length) with the outcome of a specific stress-field hypothesis; re-running Calculate with different stress inputs regenerates self.results and the Table Tab automatically reflects the new numbers on its next refresh.

4. Settings and Controls

The Table Tab exposes a deliberately minimal control surface: unlike the Map, Rose, Mohr, and Stereonet tabs — each of which carries an extensive colour/geometry/typography settings panel — the Table Tab has no cosmetic settings of its own. Its only interactive controls are the Export Excel button, the column headers (for visual reference; the Treeview widget used here does not implement interactive column sorting), row selection via mouse click, and the vertical scrollbar. This design choice keeps the tab focused on its role as a neutral, complete numerical record rather than a further stylable visualisation.

6. Mathematical Background

Although the Table Tab performs no arithmetic itself, correct interpretation of its columns requires the exact formulas used upstream in calculator.py and models.py. This section reproduces those formulas with variable definitions so that the table's numbers can be manually verified or compared against independent reference calculations (e.g. legacy MATLAB slip-tendency scripts).

6.2 Stress Resolution (σₙ, τ, α)

For a plane of given strike and dip, the outward unit normal is constructed from the pole azimuth (strike + 90°) and the dip angle. In Andersonian mode (σ1 horizontal along the entered azimuth, σ2 vertical, σ3 horizontal-perpendicular), or in the fully general 3-D mode (σ1, σ2, σ3 each independently oriented from trend/plunge inputs and combined via principal_stress_axes_fracstat()), the normal is projected onto each principal axis to obtain direction cosines c0, c1, c2. The resolved normal and shear stresses are then:

σₙ = σ1·c0² + σ2·c1² + σ3·c2² τ = √( σ1²·c0² + σ2²·c1² + σ3²·c2² − σₙ² ) α = arccos( |c0| )

This is the standard Cauchy stress-resolution equation for a plane in a principal-axis coordinate frame: σₙ is the diagonal (normal) component of the traction vector and τ is the magnitude of its orthogonal in-plane component, obtained by subtracting σₙ² from the total squared traction magnitude. σₙ is floored at 1×10⁻¹² MPa in the implementation to avoid division-by-zero downstream.

6.3 Slip Tendency (Ts, TsNorm)

Slip tendency, following Morris, Ferrill & Henderson (1996), expresses how close a plane is to frictional sliding under the resolved stress state, independent of any assumed cohesion or pore pressure:

Ts = τ / σₙ

The normalised column TsNorm divides this raw value by the theoretical maximum slip tendency attainable by any plane orientation under the same stress tensor. For a diagonal (principal-axis) stress state this maximum has the closed-form solution derived from the tangent line from the origin to the outer σ1–σ3 Mohr circle:

Ts,max = (σ1 − σ3) / ( 2·√(σ1·σ3) )

This closed-form expression depends only on σ1 and σ3 (not on σ2, the principal-axis orientation, or which fracture planes happen to be loaded), and reproduces exactly the legacy MATLAB script's TsMax, which is computed there by a dense numerical grid-scan over all φ/θ orientation pairs. TsNorm therefore always lies in the interval [0, 1], with 1 corresponding to the single, optimally oriented plane for the given stress tensor. A superseded implementation normalised instead by the maximum Ts found only among the loaded segments; this silently disagreed with the reference value whenever the input dataset did not happen to include a near-optimal plane, and has been replaced by the closed-form calculation above.

6.4 Dilation Tendency (Td)

Td = (σ1 − σₙ) / (σ1 − σ3)

Dilation tendency (Ferrill et al., 1999) measures the propensity of a fracture to open (dilate) rather than slip, and is highest for planes whose normal is closely aligned with σ3 (minimum compressive stress), where the clamping normal stress is smallest relative to the overall stress differential.

6.5 Fracture Susceptibility (Sf, SfLegacy)

Two historically distinct MATLAB reference formulas exist for fracture susceptibility, differing by exactly the pore-pressure and cohesion terms. The Table Tab reports both simultaneously as separate columns so that either can be matched against whichever legacy script produced a comparison dataset:

Sf (updated / current) = σₙ − Pf − (τ − C₀) / μ SfLegacy (original) = σₙ − τ / μ

Algebraically, SfLegacy = Sf + Pf whenever C₀ = 0, since the only structural difference between the two formulas is the presence of the pore-pressure subtraction and the cohesion offset inside the friction term. A lower Sf value indicates a plane closer to, or past, the failure envelope; Sf can be negative for critically or super-critically stressed planes.

6.6 Coulomb Critical-Stress Flag (CSF)

The Mohr–Coulomb failure criterion used throughout FracSTAT is:

τ ≥ μ·(σₙ − Pf) + C₀ ⇒ plane is critically stressed

The CSF column reports "Yes" when this inequality holds for the row's resolved (σₙ, τ) pair, and "No" otherwise. This is exactly the boolean condition drawn as the shaded Coulomb failure envelope on the Mohr tab and reflected in the map/rose colour scale when "Critical fractures" is the selected attribute.

7. Algorithms and Internal Workflow

7.1 Tab Construction — _build_table_tab()

Executed once, during SlipTendencyApp._build_layout(), after the tab's Tk container frame has been created by the CTkTabview. The method performs, in order:

  1. Grid configuration — row 2 (the data grid row) and column 0 are given weight 1 so the table stretches to fill the tab.
  2. Theme application — configure_table_style() is invoked immediately so the Treeview's ttk style exists before any rows are inserted (see Section 7.3).
  3. Toolbar construction — a transparent CTkFrame with the "Segment results" label and Export Excel button, gridded at row 0.
  4. Column and heading setup — the fifteen-element columns tuple is used both to instantiate the ttk.Treeview(columns=columns, show="headings") (note: show="headings" suppresses the default first "#0" tree column, so the widget behaves as a flat, non-hierarchical grid) and to populate each column's heading text from the headings dictionary and a fixed 90 px, centre-anchored column width.
  5. Scrollbar wiring — a ttk.Scrollbar is created with command=self.table.yview and the Treeview's yscrollcommand is bound back to the scrollbar, establishing the standard bidirectional Tk scroll contract.
  6. Event binding — the Treeview's <<TreeviewSelect>> virtual event (fired by Tk whenever the widget's selection changes, whether by mouse or keyboard) is bound to on_table_select.

No rows are inserted at build time; the table remains empty until the first refresh_table() call, which normally follows the first successful calculation.

7.2 Selection Handling — on_table_select()

This callback fires on every change to the Treeview's internal selection, including selections made programmatically by refresh_table() itself when it restores a previously highlighted row after a redraw. To avoid a feedback loop or redundant recomputation, the handler follows this short-circuit sequence:

  1. Read self.table.selection() — a tuple of currently selected item identifiers (iids). If empty (e.g. the user clicked empty space below the last row), the handler returns immediately.
  2. Parse the first selected iid as an integer with int(selection[0]). The iid was set at insertion time to str(int(row.ID)), so this recovers the numeric segment ID. If parsing fails (defensive — should not occur given how iids are always assigned), the handler returns silently.
  3. Compare the parsed ID against self.selected_segment_id. If they are already equal — for example because refresh_table() just re-selected the same row the user had previously picked — the handler returns without further action, preventing an unnecessary cascade of redraws across the Map, Mohr, and Stereonet tabs.
  4. Otherwise, call self.select_segment(segment_id), the shared cross-tab selection routine defined on SlipTendencyApp itself (see Section 8).
python 
def on_table_select(self, _event=None):
    selection = self.table.selection()
    if not selection:
        return
    try:
        segment_id = int(selection[0])
    except ValueError:
        return
    if segment_id == self.selected_segment_id:
        return
    self.select_segment(segment_id)

on_table_select() in full — the short-circuit guards prevent a redundant cascade of redraws whenever the incoming selection already matches the current cross-tab selection.

7.4 Refresh Algorithm — refresh_table()

refresh_table() is the workhorse of the tab and is called every time self.results changes (after calculate(), after select_segment(), and after clear_selection()). Its steps are:

  1. Clear existing rows: self.table.delete(*self.table.get_children()) removes every currently displayed row unconditionally, regardless of how many there are.
  2. Empty-result guard: if self.results.empty, the method returns immediately, leaving the table blank — this is the state shown before any file has been loaded or calculated.
  3. Dynamic unit relabelling: self.table.heading("Length", text=f"Length ({self.unit_label()})") rewrites the Length column's heading text to reflect whatever display unit (m, cm, mm, km) is currently active in the sidebar, so the heading never becomes stale after a unit-conversion action.
  4. Optional-column detection: has_ts_norm and has_sf_legacy are booleans testing whether "SlipTendencyNorm" and "FractureSusceptibilityLegacy" exist as columns in self.results. Because calculator.run_calculation() always adds both columns in the current implementation, these guards are defensive — they protect the table from raising an AttributeError if it is ever driven by an older or externally supplied results DataFrame that lacks one of the derived columns, substituting the placeholder character "—" (em dash) instead.
  5. Row insertion loop: self.results.head(1000).itertuples(index=False) iterates at most the first 1000 rows of the results DataFrame (see Section 10 for the rationale). For each row, the displayed values are formatted (see Section 5's Format column) and inserted with self.table.insert("", "end", iid=str(int(row.ID)), values=(...)). Using the segment's own ID as the Tk item identifier is what allows on_table_select() to recover the numeric ID directly from the widget's selection state without a separate lookup table.
  6. Selection restoration: if self.selected_segment_id is not None and its string form exists among the freshly inserted children, the method calls self.table.selection_set(selected_iid) followed by self.table.see(selected_iid). This re-applies the previously active selection (which was necessarily cleared by step 1's blanket delete) and scrolls the view so the selected row is visible, giving the appearance of a persistent highlight across recalculations even though the underlying Tk items were fully destroyed and rebuilt.

8. Cross-Tab Selection Synchronisation

The Table Tab does not maintain its own notion of "the selected segment"; it defers entirely to SlipTendencyApp.select_segment(segment_id), the single shared routine also invoked by mouse-picking on the Map (on_map_pick), point-picking on the Mohr diagram (on_mohr_pick), and pole-picking on the Stereonet (on_stereonet_pick). When a Table Tab row is clicked, the following chain executes:

  1. on_table_select() resolves the clicked row's ID and calls self.select_segment(segment_id).
  2. select_segment() looks up the matching row in self.results, sets self.selected_segment_id, updates the sidebar's "Selected Segment" text panel (segment_info_text()) with a full geometric/orientation/stress/tendency summary, updates the Mohr diagram's information panel and coloured status badge (Critical / Near envelope / Stable, based on CoulombMargin and a 5%-of-max-τ tolerance band), and updates the Stereonet's pole-inspector text.
  3. select_segment() then calls self.refresh_table() itself, so the Table Tab's own view re-synchronises its highlighted row even when the selection originated elsewhere (e.g. clicking a point on the Mohr diagram highlights the same segment's row in the Table Tab automatically).
  4. Finally, select_segment() triggers preserve-view redraws of the Map (_draw_map(preserve_view=True)), Mohr (_draw_mohr(preserve_view=True)), and Stereonet (_draw_stereonet()) so the selected segment is visually emphasised (halo/outline markers, annotation labels) everywhere simultaneously.

This design means the Table Tab is both a consumer and a producer of selection state: any tab's picking interaction is reflected in the Table Tab's highlighted row, and clicking a Table Tab row is reflected everywhere else. The optional switch_to_map parameter of select_segment() (used by the Mohr and Stereonet tabs' "Show in Map" buttons) is not exercised from the Table Tab itself, which never forces a tab switch.

Clearing the selection — via the Mohr or Stereonet tabs' "Clear Selection" buttons, which call SlipTendencyApp.clear_selection() — sets self.selected_segment_id back to None and calls refresh_table(), which then finds no matching iid in step 6 of Section 7.4 and leaves the table with no row highlighted.

9. Statistical Content and Relationship to the Sidebar Summary

The Table Tab itself performs no aggregation; the workspace-level statistical summary (traces, segments, Max Tₛ, Mean Tₛ, total length, and critical-fracture count) is computed separately in SlipTendencyApp.refresh_summary() and displayed in the sidebar's "Results" card, not inside the Table Tab. However, because the Table Tab exposes every underlying row value, it is the natural tool for verifying those aggregate statistics manually: for instance, the sidebar's "Max Tₛ" / "Mean Tₛ" tiles are computed over whichever attribute (SlipTendency, SlipTendencyNorm, DilationTendency, FractureSusceptibility(Norm), or CriticalStressValue) is currently selected in the "Map Display" dropdown, and the exact numeric contributors to that maximum or mean can be located, sorted by eye, and cross-checked row-by-row in the Table Tab.

10. Performance Optimisation

The Table Tab includes two deliberate performance safeguards, both aimed at keeping the Tk main loop responsive when self.results contains a very large number of segments (dense SVG trace networks with fine curve-tessellation settings can readily produce several thousand to tens of thousands of rows):

  • Row cap — refresh_table() iterates only self.results.head(1000), i.e. at most the first one thousand rows in DataFrame order, regardless of the true row count. ttk.Treeview inserts are individually expensive Tk operations, and inserting tens of thousands of rows synchronously on the main GUI thread would visibly freeze the interface for a period proportional to row count. Capping at 1000 keeps the insertion loop's wall-clock cost small and bounded irrespective of dataset size. This cap affects only the Table Tab's on-screen display and selection restoration; it has no effect on the Map, Rose, Mohr, Stereonet, or Excel-export pathways, all of which continue to operate on the full, uncapped self.results DataFrame.
  • Full-clear-and-rebuild rather than incremental diffing — every refresh deletes all existing rows before reinserting, rather than attempting to compute a minimal diff against the previous state. This trades a theoretically larger amount of Tk work per refresh for substantially simpler, more robust code with no risk of stale rows, orphaned iids, or column-value drift; given the 1000-row cap, the practical cost of full deletion-and-reinsertion is small.

No caching layer, background thread, or asynchronous loading is used for the Table Tab specifically (contrast with, for example, the Static Analyzer's asynchronous SVG/Excel loaders or the Connectivity tab's lazy, tab-click-triggered recomputation) — the table's own render cost is low enough, given the row cap, that synchronous execution on the main thread does not perceptibly delay the interface.

10.1 Practical Implication of the 1000-Row Cap

When a dataset exceeds 1000 segments, the on-screen table displays only the first 1000 in DataFrame order (which is the order segments were produced during loading/calculation — typically grouped by trace for SVG sources, or by input-file row order for Excel/CSV sources). Analysts working with very dense networks should rely on the Export Excel action to obtain the complete, uncapped dataset for full-population statistical analysis, and should not assume the visible table represents the entire fracture population once the row count exceeds one thousand.

11. Export Functionality

The Export Excel button is the Table Tab's only outward data-transfer control; it delegates to SlipTendencyApp.export_excel(), a method shared at the workspace level (the Table Tab does not implement its own export logic). The behaviour is:

  1. Guard: if self.results.empty, an error dialog ("Nothing to export" / "Run a calculation first.") is shown and the method returns without opening a file dialog.
  2. Default directory resolution: the method attempts to read a configured default export path from the application's persisted settings (pages.settings_page.load_app_settings()); if unavailable or invalid, no initial directory is pre-selected and the OS default is used.
  3. Save dialog: a native "Save As" dialog is presented with a .xlsx default extension and an "Excel workbook" file-type filter.
  4. Write: self.results.to_excel(filename, index=False, sheet_name="Segment results") writes the complete (uncapped) DataFrame — every row and every column, including any columns not rendered in the on-screen table — to a single-sheet workbook. This means the exported file is a strict superset of what is visually displayed, and is the recommended path for any downstream analysis requiring the full dataset.
  5. Confirmation: the workspace status bar is updated with the number of exported rows and file path, and a confirmation dialog displays the saved path.
Tip: a companion CSV export (export_csv(), triggered from the sidebar's "Export CSV" action button rather than from within the Table Tab) is available at the workspace level using the same underlying self.results DataFrame and the same empty-result guard, for workflows that prefer a plain-text tabular format over a native Excel workbook.

12. Practical Examples

12.1 Example 1 — Verifying a Slip-Tendency Calculation by Hand

An analyst loads an SVG fracture-trace map, sets σ1 = 104.7 MPa, σ2 = 60.3 MPa, σ3 = 47.1 MPa in Andersonian mode with azimuth 0°, and presses Calculate. To manually verify the reported Tₛ (norm) value for segment ID 12, the analyst locates row ID = 12 in the Table Tab (using the scrollbar or, for large datasets, exporting to Excel and using a spreadsheet find function), reads off StrikeDeg, DipDeg, σₙ, and τ, and reproduces α, σₙ, τ using the equations of Section 6.2, then Ts and TsNorm using Section 6.3's formulas — including the closed-form Ts,max — confirming the value shown in the Tₛ (norm) column.

12.2 Example 2 — Cross-Referencing a Critical Segment Between Table and Mohr Diagram

After calculation, the analyst scans the CSF column in the Table Tab for rows marked "Yes", identifying the subset of the fracture population predicted to be critically stressed under the current stress hypothesis. Clicking one such row (say ID = 47) immediately re-draws the Mohr tab with segment 47's point highlighted with a bold outline and "ID 47" annotation sitting on or beyond the shaded failure envelope, and simultaneously highlights the same segment on the Map and Stereonet tabs — allowing the analyst to confirm both the stress-space position (Mohr) and the real-world geometric position and orientation (Map/Stereonet) of that specific fracture in one interactive step.

12.3 Example 3 — Comparing Legacy and Corrected Fracture Susceptibility

When importing results intended to reproduce an older MATLAB workflow that used the original Sf = σₙ − τ/μ formula (no pore-pressure correction), the analyst compares the Sf (Pf-corr.) and Sf (legacy) columns directly in the Table Tab. If the two differ by a near-constant offset across all rows and the sidebar's Pore press. (MPa) field is non-zero, this offset is expected and equals approximately Pf (Section 6.5); a non-constant offset instead flags a non-zero cohesion input interacting with the two formulas differently and should prompt the analyst to check the Failure Criteria card's Cohesion (MPa) field.

12.4 Example 4 — Exporting Full Results for a Technical Report

For inclusion of a complete segment-level results appendix in a technical report, the analyst clicks Export Excel, saves the workbook, and opens it in Excel or an equivalent spreadsheet application; every row of self.results — not merely the 1000 rows visible in the on-screen table for a large network — is present, including columns (such as raw pixel coordinates, RawX1/RawY1/RawX2/RawY2, or focal-mechanism metadata for geographic datasets) that are not shown in the Table Tab's fifteen-column view at all.

13. Geological and Geomechanical Interpretation

The Table Tab's numerical content supports several standard structural-geology and reservoir-geomechanics interpretation workflows built on the Wallace–Bott hypothesis (that slip on a pre-existing fracture occurs in the direction of maximum resolved shear traction) and its slip/dilation-tendency derivatives:

  • Fault/fracture reactivation risk assessment — rows with high TsNorm and CSF = "Yes" identify the subset of a mapped fracture network most susceptible to shear reactivation under the assumed in-situ or induced stress state, directly informing induced-seismicity risk screening around injection or production operations.
  • Fluid-flow pathway prediction — high-Td fractures are preferentially interpreted as open, dilated conduits, whereas high-Ts, low-Td fractures are better candidates for shear-enhanced (but not necessarily dilated) permeability; comparing the Td and Ts (norm) columns row-by-row across a population supports discriminating between these two permeability-enhancement mechanisms.
  • Wellbore-stability and completion design — the Sf column's sign and magnitude for fractures intersecting a planned or existing wellbore trajectory is a standard input into whether a given natural fracture is likely to be a fluid-loss zone or a slip hazard during drilling or stimulation.
  • Stress-tensor sensitivity testing — because every column updates immediately after a recalculation, an analyst can iterate the σ1/σ2/σ3 magnitudes or orientation inputs and observe, row-by-row in the Table Tab, how sensitive the critically-stressed subset (CSF) and the slip/dilation-tendency rankings are to the assumed stress state — a standard uncertainty-bracketing exercise given that in-situ stress magnitudes and orientations are rarely known with high precision.

14. Implementation Notes

  • Mixin architecture — TableTab is a bare Python mixin class (it does not inherit from ctk.CTkFrame or any Tk widget); its methods assume the presence of self.tabs, self.results, self.selected_segment_id, self.appearance_var, self.unit_label(), and self.select_segment(), all of which are supplied by SlipTendencyApp at composition time via Python's standard multiple-inheritance MRO (SlipTendencyApp(WorkspaceHeaderMixin, MapTab, RoseTab, MohrTab, TableTab, MethodTab, StereonetTab, ctk.CTkFrame)).
  • Widget technology — the Table Tab is the only tab in the Dynamic Analyzer that uses the standard tkinter.ttk.Treeview/Scrollbar widgets rather than CustomTkinter equivalents (CustomTkinter has no built-in multi-column tabular widget); this is why it requires its own explicit ttk.Style theming rather than relying on CustomTkinter's automatic light/dark colour tuples.
  • Item identifiers (iids) — using str(int(row.ID)) as the Treeview iid, rather than Tk's default auto-incrementing internal iid, is what allows O(1) mapping between a clicked row and its underlying segment ID, and what allows selection restoration (Section 7.4, step 6) to work by simple string containment testing ("in self.table.get_children()") without maintaining a separate lookup dictionary.
  • Row-value construction is done with pandas' itertuples(index=False), which is significantly faster than iterrows() for this row count and, unlike iterrows(), preserves each column's native dtype as an attribute (row.ID, row.StrikeDeg, etc.) rather than boxing every value into a single mixed-type Series per row.
  • No sorting state is persisted or implemented — clicking a column heading does not resort the Treeview in the current implementation; the table always reflects self.results' native row order (segment ID order).
  • Thread safety — refresh_table() and on_table_select() are both expected to run exclusively on the Tk main thread, consistent with the rest of the Dynamic Analyzer; no locking or thread-marshalling is implemented because no other thread ever mutates self.results or self.selected_segment_id concurrently with the Table Tab's own execution.

15. Limitations

LimitationDescription
Row capOnly the first 1000 rows of self.results are ever rendered on screen (Section 10); very large fracture networks require the Export Excel action to inspect the full dataset.
No interactive sorting or filteringThe Table Tab cannot resort by column value, filter by threshold, or search by ID directly within the application; such operations require exporting to Excel/CSV first.
No in-place editingAll fifteen displayed columns are read-only; the Table Tab cannot be used to manually override a segment's dip, strike, or any derived attribute (dip overrides must instead be supplied via a separate CSV/Excel dip-override file loaded through the sidebar).
No multi-row selection semantics for cross-tab highlightingAlthough ttk.Treeview technically supports multi-row selection, on_table_select() only ever reads selection()[0], so highlighting a batch of rows simultaneously across the Map/Mohr/Stereonet tabs is not supported.
Column set is fixedThe fifteen columns cannot be reconfigured, reordered, hidden, or extended with additional derived quantities (e.g. Rake, TauRake, PredictedRake, RakeMisfit for focal-mechanism datasets, or Lat/Lon for geographic data) from within the tab itself, even though these fields exist in self.results and are exported to Excel.
Number formatting is fixedDecimal-place precision (1, 2, or 3 dp depending on column) is hardcoded and cannot be adjusted by the user from within the Table Tab.
Dark-mode styling is implemented but currently unreachable in normal useSince SlipTendencyApp.change_appearance() always forces the Light theme at the application level.
