Dynamic Method Tab

Located at dynamic_analyzer/tabs/dynamic_method_tab.py, the Method tab (implemented as the mixin class MethodTab) is the Dynamic Analyzer's built-in, read-only reference sheet. Unlike every other tab in the Dynamic Analyzer, it consumes no project data and produces no plot — it exists purely to document, in one place, the exact geometric and stress-mechanical formulas that the Map, Rose, Mohr, and Table tabs apply to every loaded fracture segment. It is the single source of truth a user can check to confirm precisely how a reported stress angle, slip tendency, or Coulomb failure state was derived.

1. Interface Overview

The Method tab occupies the full width of its workspace tab with a single scrollable CTkTextbox widget, padded 20 px on every side to give the text a clean, card-like appearance. There is no sidebar, no plot canvas, and no interactive control of any kind — the widget is explicitly locked to a read-only (state="disabled") state immediately after its content is inserted, so the tab behaves purely as documentation rendered inside the application itself rather than as an analysis surface.

2. Purpose and Scope

Every other tab in the Dynamic Analyzer — Map, Rose, Mohr, Stereonet, and Table — resolves stress onto fracture planes and reports derived quantities such as slip tendency, dilation tendency, susceptibility, and Coulomb failure state, but none of them re-explain the underlying formulas at the point of use. The Method tab exists to close that gap: it is a static, in-application reference that a user can open at any time to see, in order, exactly how raw SVG geometry is turned into a scaled fracture network, how that network's strikes are corrected to true North, how the stress angle to each fracture is computed, how normal and shear stress are resolved via the 2-D Mohr transformation, and how each of the tendency and failure metrics reported elsewhere in the workspace is defined.

Because the tab performs no calculation of its own — it only documents calculations performed by other tabs — it has no data-input requirements, no settings, and no redraw logic. Its entire behaviour is determined once, at tab-build time, by the sequence of text.insert() calls in _build_method_tab().

3. Content Structure and Typography

The tab uses Tkinter text-widget tags to give the reference sheet a typographic hierarchy without resorting to a rich-text or HTML rendering engine. Four fonts and five tags are defined once at build time and then applied selectively as each line is inserted.

TagFontRole
titleRoboto, 22 pt, bold, centredThe single page heading, "Calculation Method".
headerRoboto, 16 pt, boldSection headings (Geometry, Profile Scaling, Stress Angle, Resolved Stress, Tendency Metrics, Plots, Focal-Mechanism Mode).
bodyRoboto, 13 pt, regularExplanatory prose beneath each section heading.

Note: because a plain CTkTextbox cannot mix a smaller, baseline-shifted run of characters into the middle of a single inserted string, every subscripted symbol used in the Tendency Metrics section (Ts, Td, Sf) is built from three separate insert() calls — the base letter tagged math, the subscript character tagged with the combined ("math", "sub") tuple, and the remainder of the formula tagged math again — rather than from one call carrying the whole line.

4. Mathematical Background

The reference sheet proceeds through five formula-bearing sections, in the same order every other Dynamic Analyzer tab implicitly relies on: geometry construction, profile scaling and strike correction, the stress angle definition, the resolved-stress transformation, and the tendency/failure metrics computed from that resolved stress.

4.1 Geometry

Every digitised SVG trace is converted into one or more straight fracture segments before any stress calculation begins. A line primitive in the source SVG becomes exactly one segment; curved primitives are sampled into a series of shorter straight segments according to the workspace's Curve step setting, exactly mirroring the tessellation approach used by the Static Analyzer's geometry engine. Every resulting segment — whether from an originally straight or originally curved trace — is treated identically and calculated independently by every downstream stage.

4.2 Profile Scaling and Strike Correction

Raw SVG coordinates exist only in arbitrary digitiser pixel units. FracSTAT rescales the full coordinate extent of the loaded trace map onto the physical Profile Width and Profile Length supplied by the user, in metres, so that every plotted X/Y coordinate and every reported segment length is expressed in the currently selected display unit rather than in raw pixels. Because the horizontal and vertical scale factors are computed independently from the profile width and length respectively, non-square profiles (where width and length differ) are still handled correctly — the rescaling is not assumed to be uniform in both directions.

Segment strike, as measured directly from the SVG geometry, is referenced to the image's own Y axis rather than to true geographic North. The Image Y-axis azimuth setting, θ, records how far that image Y axis is rotated away from true North, and every raw image strike is corrected by this offset before it is used anywhere else in the workspace:

strike_N = image strike + θ

4.3 Stress Angle

The maximum principal stress σ₁ is supplied as an azimuth measured from true North. FracSTAT does not resolve stress directly onto the fracture plane's strike direction; it resolves stress onto the plane's pole (its outward normal), which by definition sits 90° from the corrected strike. The stress angle α used in every subsequent Mohr transformation is therefore the angular difference between that normal direction and the σ₁ azimuth:

α = strike_N + 90° − azimuth_N(σ₁)

4.4 Resolved Stress

With α defined relative to the fracture normal, the normal stress σₙ and shear stress τ acting on the fracture plane follow directly from the standard two-dimensional Mohr stress transformation:

σₙ = ½(σ₁ + σ₂) + ½(σ₁ − σ₂)·cos(2α) τ = | −½(σ₁ − σ₂)·sin(2α) |
Sign convention: the σₙ equation carries a plus sign on its second term because α is measured to the fracture normal, not to the fracture plane itself. Formulas quoted elsewhere in the structural-geology literature that instead define the angle relative to the fracture plane will show this same cosine term with a minus sign — the two conventions are equivalent once the 90° offset between plane and normal is accounted for, but they are not interchangeable term-for-term.

4.5 Tendency Metrics

Four derived quantities, each computed directly from the resolved σₙ and τ of Section 4.4, are reported throughout the Dynamic Analyzer's Map, Rose, and Table tabs:

SymbolNameFormula
TsSlip tendencyTs = τ / σₙ
TdDilation tendencyTd = (σ₁ − σₙ) / (σ₁ − σ₂)
SfSusceptibility / fracture stability marginSf = σₙ − Pᶠ − (τ − C₀) / μ
CSFCritical stress flagTrue where τ ≥ μ(σₙ − Pᶠ) + C₀

Here Pᶠ is pore-fluid pressure, C₀ is cohesion, and μ is the coefficient of friction — the same three Coulomb failure-envelope parameters used to draw the failure line on the Mohr diagram. Slip tendency Ts is the ratio of resolved shear to resolved normal stress and increases toward frictional failure; dilation tendency Td measures how close a fracture's normal stress sits to the least principal stress σ₂, with values near 1 indicating a fracture oriented favourably for opening-mode dilation; the susceptibility/stability margin Sf and the boolean CSF flag both express distance from, or crossing of, the effective-stress Coulomb failure envelope.

5. Implementation: How the Text Is Built

_build_method_tab() runs once, when the Method tab is constructed, and performs no further work afterward — there are no callbacks, no variable bindings, and no redraw path, since nothing on the tab can be edited or reconfigured by the user.

  1. Grid-configure the tab's single row and column to expand, and place one CTkTextbox with 20 px internal padding so the text reads as a padded card rather than touching the tab's edges.
  2. Construct the four named CTkFont objects (title, header, body, math) plus the smaller subscript font, all in the Roboto family for prose and Consolas for formulas.
  3. Register five tags on the textbox's underlying Tkinter Text widget (text._textbox), each carrying its own font, spacing, indentation, and (for title) centre justification.
  4. Insert the page title, then each section heading and its body/formula content in sequence, using the appropriate tag or tag-tuple for every inserted run of text.
  5. For the three subscripted tendency-metric symbols, insert the base letter, the subscript character, and the remainder of the line as three separate calls so the subscript renders at reduced size and lowered baseline.
  6. Call text.configure(state="disabled") as the final step, permanently locking the widget against further editing while leaving it fully scrollable and selectable for copying.
python 
text._textbox.tag_config("math", font=font_math, lmargin1=30, lmargin2=30, spacing1=5, spacing3=15)
text._textbox.tag_config("sub", offset="-4", font=font_math_sub)

# T_s, rendered as three separate inserts so the "s" renders as a
# smaller, baseline-shifted subscript within an otherwise monospaced line
text.insert("end", "T", "math")
text.insert("end", "s", ("math", "sub"))
text.insert("end", " = τ / σₙ\n", "math")

# Lock the widget once every section has been inserted
text.configure(state="disabled")

6. Focal-Mechanism Mode

The final section of the reference sheet documents a distinct data pathway that the rest of the Dynamic Analyzer treats transparently: Excel/CSV files carrying Lat, Lon, FP1 Strike/Dip and, optionally, FP1/FP2 Slip columns are recognised as earthquake focal-mechanism data rather than digitised fracture traces. Each supplied nodal plane becomes its own segment, after which the Map tab plots the event at its true Lat/Lon location and can display the corresponding GlobalCMT-style beachball diagram, fetched from the file's Beachball URL column, for whichever event is currently selected.

When a Slip (rake) value is supplied for a nodal plane, FracSTAT goes beyond simple stress resolution: it projects the resolved shear traction onto the true three-dimensional slip vector constructed from that plane's strike, dip, and rake, following the Wallace–Bott hypothesis that slip occurs parallel to the resolved shear stress on the fault plane. The angular difference between the observed slip direction and the direction the entered stress tensor predicts is reported as the Rake Misfit — a low misfit indicates that the assumed stress state is kinematically consistent with the recorded earthquake, while a high misfit suggests the entered principal stress orientation and/or ratio should be reconsidered.

Backward compatibility: older two-column focal-mechanism files containing only ID, Strike, and Dip continue to load and function exactly as before; the loader auto-detects which of the two schemas a given file uses without requiring any action from the user.

7. Relationship to Other Dynamic Analyzer Tabs

The Method tab's content is descriptive of, and consistent with, exactly what the rest of the workspace computes and displays:

Dynamic Analyzer TabWhat it uses from the Method tab's formulas
Map TabColours every segment by whichever tendency/failure metric (Ts, Td, Sf, or CSF) is currently selected, computed via Sections 4.3–4.5.
Rose TabBuilds a length-weighted, 10° binned, square-root (equal-area) polar histogram of the same corrected strike_N values described in Section 4.2, mirroring 0–180° strikes into the 180–360° half for full-circle display.
Mohr TabPlots total stress (σₙ, τ), effective stress shifted by pore pressure Pᶠ, individual segment points, and the Coulomb failure envelope τ = C₀ + μσₙ′ using exactly the resolved-stress formulas of Section 4.4; Mohr-diagram point numbers correspond directly to each segment's ID in the Table tab. The quantity Δτ = τ − [C₀ + μ(σₙ − Pᶠ)] is positive wherever a segment's Mohr point touches or crosses the failure envelope.
Table TabTabulates the per-segment numeric results (strike_N, α, σₙ, τ, Ts, Td, Sf, CSF) produced by the same pipeline documented here.
