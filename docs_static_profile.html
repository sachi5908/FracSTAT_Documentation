Static Profile Tab

Located at static_analyzer/tabs/static_profile_tab.py, the Profile Cross-section tab (implemented as the mixin class StaticGraphsTab) provides a one-dimensional sampling framework for extracting geometric and statistical fracture attributes along, and perpendicular to, a user-defined reference line ("scan line") superimposed on a digitized fracture-trace map. It lets a user interrogate spatial trends in fracture length, orientation, spacing, and intensity across a mapped outcrop, drill-core photomosaic, or remote-sensing image, in a manner analogous to a geological cross-section or transect survey. This module digitizes the classical field scanline survey: a straight scanline is laid across an outcrop and the fractures crossing it are logged together with their spacing, orientation, and length, yielding a one-dimensional statistical sample of an otherwise two-dimensional fracture network — the basis of fracture-intensity metrics such as P10, P21, and P20. The module operates exclusively within the Static Analyzer workspace, performing purely geometric analysis with no dependency on stress, slip tendency, or any mechanical property computed in the Dynamic Analyzer.

Once a fracture-trace map has been imported into the Static Analyzer (via an SVG trace file or a spatial Excel/CSV coordinate table), the user positions a two-point reference line anywhere across the mapped area — either by typing coordinates or by clicking directly on the map — and the module computes, for every fracture segment or trace that the reference line crosses or lies near:

  • The one-dimensional position ("distance along") at which each fracture intersects the line, together with the trace length, segment length, or segment angle associated with that intersection;
  • The perpendicular offset ("distance from") of trace and segment midpoints relative to the line, again cross-plotted against trace length, segment length, or segment angle;
  • A Mauldon-style circular-scanline estimate of fracture intensity (P21) and density (P20) evaluated at a user-specified number of positions along the line.

The module therefore serves two complementary purposes: it is a spatial-trend diagnostic tool, letting the user determine whether fracture attributes vary systematically across a mapped domain (for example, increasing intensity toward a fault zone), and it is a quantitative intensity/density estimator that is robust to network anisotropy, because the Mauldon circular-window method removes the orientation bias inherent in straight scanline counts.

1. Position within the FracSTAT Architecture

FracSTAT is organised into two principal analytical workspaces — the Static Analyzer (geometric fracture-network characterisation) and the Dynamic Analyzer (stress-based slip/dilation tendency analysis) — plus a placeholder Uncertainty Analyzer workspace. The Static Analyzer itself is a tabbed workspace comprising nine tabs: Fracture Map, Histogram, CDF, Rose Plot, Stereonet Plot, Connectivity, Intensity & Density, Profile Cross-section, and Results.

The Profile Cross-section tab is implemented by the StaticGraphsTab mixin class, combined via multiple inheritance into the StaticAnalyzerApp class alongside the other eight tab mixins and the shared WorkspaceHeaderMixin top toolbar. All tabs share the same underlying data source: a pandas DataFrame stored on the application instance as self.results, which is (re)computed whenever an SVG or Excel/CSV file is loaded or the profile scale is changed.

2. Interface Overview

The Profile Cross-section tab is divided into three regions arranged left to right: a tabbed sidebar ("Info" and "Settings" sub-tabs) hosting all analysis and appearance controls, a plot card containing two side-by-side Matplotlib axes — the reference-line map on the left and the derived output plot on the right — and a toolbar above the plot card offering a descriptive label and a Save Image button.

2.1 Left Sidebar — Info Tab

The Info tab is the primary analysis-configuration panel and is organised into four stacked cards:

  • Distance ALONG the line — five checkboxes selecting which along-line attributes to compute and plot, plus a numeric field controlling the number of circular scan windows used for the intensity/density calculation.
  • Distance FROM the line — three checkboxes selecting which perpendicular-offset attributes to compute and plot.
  • Plot Options — a single checkbox controlling whether output series are connected by a line.
  • Line End Points — four numeric entries (X1, Y1, X2, Y2) defining the reference line, an interactive "Select end points" button for on-map digitizing, and a live status label.

An Update Graphs button at the foot of the Info tab forces an immediate redraw using whatever combination of checkboxes and coordinates is currently set.

2.2 Left Sidebar — Settings Tab

The Settings tab controls the cosmetic appearance of both the map and the output plot: a custom title override, independent legend-position selectors for the map and the output plot, numeric grid- and boundary-line-width fields, a full palette of colour pickers (outer/plot background, grid, plot boundary, scan-line colour, trace colour), a grid-visibility checkbox, and a dedicated Background Image sub-section for overlaying a calibrated real-world photograph beneath the fracture traces. A Reset Defaults button restores every appearance setting in this tab to its factory value.

2.3 Plot Card

The plot card hosts a single Matplotlib Figure containing two subplots created with add_subplot(121) and add_subplot(122): the left subplot (graph_map_ax) always shows the fracture-trace network with the current scan line and, where relevant, the scan-circle windows superimposed; the right subplot (graph_plot_ax) shows whichever combination of along-line and from-line series the user has selected, as a scatter plot with optional connecting lines. A toolbar row above the plot card displays a short descriptive caption and a "Save Image" button that exports the current figure (including any floating legend) as a high-resolution PNG.

5. Mathematical Background

5.1 Reference Line and Parametric Coordinates

The reference (scan) line is defined by its two endpoints P₁ = (x₁, y₁) and P₂ = (x₂, y₂), both expressed in the calibrated real-world display unit. Its Euclidean length is:

L = √[(x₂ − x₁)² + (y₂ − y₁)²]

A line shorter than 10⁻⁹ display units (i.e. two coincident points) is treated as invalid and suppresses computation, since no meaningful direction can be defined.

5.2 Line–Line Intersection (Distance Along)

For each fracture segment S, with endpoints Q₁ = (x₃, y₃) and Q₂ = (x₄, y₄), the intersection with the reference line is found using the standard two-parameter linear system. Writing the reference line as P₁ + u·(P₂ − P₁) and the segment as Q₁ + v·(Q₂ − Q₁), the intersection parameters (u, v) are obtained from:

u = [ (x₃−x₄)(y₁−y₃) − (y₃−y₄)(x₁−x₃) ] / D v = [ (x₂−x₁)(y₁−y₃) − (y₂−y₁)(x₁−x₃) ] / D D = (x₁−x₂)(y₃−y₄) − (y₁−y₂)(x₃−x₄)

where D is the determinant of the direction-vector matrix (the two lines are parallel and non-intersecting whenever |D| < 10⁻¹²). A genuine crossing exists only when both u and v lie within the closed unit interval [0, 1] (subject to a small numerical tolerance of ±10⁻¹⁰ to admit endpoint-touching intersections). The along-line distance of a valid intersection is then:

d_along = L · u

This is the quantity plotted on the horizontal axis for every "…along" series; it represents how far the intersection point lies from the P₁ end of the reference line, in real-world units.

5.3 Signed Perpendicular Distance (Distance From)

For a point (x, y) — typically a segment or trace midpoint — its signed perpendicular distance from the infinite line through P₁ and P₂ is computed via the two-dimensional cross-product form:

d_from = [ (x₂−x₁)(y₁−y) − (x₁−x)(y₂−y₁) ] / L

The sign of d_from indicates which side of the reference line the point falls on (consistent with a right-hand rule relative to the line's direction from P₁ to P₂); its magnitude is the shortest Euclidean distance from the point to the line. Because this formula treats the reference line as infinite, contributions may originate from anywhere in the mapped domain, not only from the finite segment between P₁ and P₂ — appropriate for a "from-the-line" transect statistic, where the line represents an axis of reference rather than a bounded sampling window.

5.4 Trace and Segment Midpoints

For the "Segment length/angle from" series, the representative point of each segment is its simple geometric midpoint, ((x₃+x₄)/2, (y₃+y₄)/2). For the "Trace length from" series, each trace is represented by the length-weighted centroid of its constituent segment midpoints:

x̄ = Σᵢ wᵢxᵢ / Σᵢ wᵢ , ȳ = Σᵢ wᵢyᵢ / Σᵢ wᵢ

where xᵢ, yᵢ are the i-th segment's midpoint coordinates and wᵢ is that segment's length (Length_m). If the total weight for a trace is zero or non-finite (e.g. a degenerate zero-length trace), the module falls back to an unweighted arithmetic mean of the segment midpoints.

5.5 Circular Scan-Window Distribution

For the intensity/density series, N circular sampling windows (N = "Number of scan circles") are distributed evenly along the finite reference line. Denoting the user-supplied count as N, the scan-circle radius r and the along-line positions dₖ are:

r = L / (2N) dₖ are N values linearly spaced between r and L − r, k = 1 … N

Each circle is therefore tangent to its neighbours along the line and does not extend past the two line endpoints, which prevents edge circles from being centred outside the mapped area. The circle centre for scan window k is obtained by projecting distance dₖ along the unit direction vector of the reference line.

5.6 Segment–Circle Intersection Count

For each fracture segment and each scan circle of radius r centred at C = (cₓ, c_y), the module must determine (a) whether either endpoint of the segment lies inside the circle, and (b) how many times the segment's finite extent crosses the circle boundary. Writing the segment as A + t·(B − A) for t ∈ [0, 1], substitution into the circle equation ‖A + t(B−A) − C‖² = r² yields the quadratic:

a·t² + b·t + c = 0 a = ‖B−A‖² , b = 2[(A−C)·(B−A)] , c = ‖A−C‖² − r²

The discriminant Δ = b² − 4ac determines the number of real roots. When Δ ≤ 0 or a ≈ 0 (degenerate segment), the segment does not cross the circle boundary. Otherwise the two roots t₁,₂ = (−b ∓ √Δ)/(2a) are tested against the interval [0, 1]; the number of admissible roots (0, 1, or 2) is the number of true boundary crossings contributed by that segment to that scan circle.

5.7 Fracture Intensity (P21) and Density (P20)

Following the circular-scanline method (after Mauldon, and consistent with the Connectivity and Intensity & Density tabs elsewhere in FracSTAT), each scan circle of radius r accumulates two counts: N_int, the total number of segment–circle boundary crossings (from Section 5.6, counting a fully interior segment once and a partially interior segment once for its single boundary crossing), and N_end, the total number of segment endpoints that fall strictly inside the circle. The two derived statistics at that scan position are:

P21 = N_int / (4r) P20 = N_end / (2πr)

P21 has units of [length⁻¹] and estimates the total fracture-trace length per unit sampled area (i.e., areal fracture intensity); P20 has units of [length⁻²] and estimates the number of fracture terminations/nodes per unit sampled area (areal fracture density). Both are evaluated independently at every one of the N scan-window positions along the reference line, producing a one-dimensional profile of intensity/density as a function of position — directly analogous to logging fracture intensity along a physical scanline traverse in the field.

6. Algorithms

6.1 Reference-Line Resolution and Validation

On every redraw, the four coordinate StringVars (X1, Y1, X2, Y2) are parsed as floating-point numbers. If parsing fails for any field, or if the resulting line length is at or below the 10⁻⁹ tolerance, the module renders an explanatory empty-state view ("Enter or select two end points") instead of a plot, preventing a degenerate or undefined geometry from propagating into the downstream calculations.

6.2 Default Line Placement

If none of the four coordinate fields has ever been populated, a default line is generated automatically by taking the minimum and maximum X and Y values across every segment endpoint in the loaded dataset, i.e. the diagonal of the data's axis-aligned bounding box. This guarantees that opening the tab for the first time on a freshly loaded dataset produces an immediately interpretable plot.

6.3 Along-Line Intersection Sweep

The along-line sweep iterates once over every row (segment) of self.results. For each segment, the parametric line intersection of Section 5.2 is evaluated against the current reference line. Successful intersections are collected into a list of records containing the along-line distance, the parent trace's total length (looked up from a pre-computed per-trace length aggregation), the segment's own length, and the segment's strike; the list is then sorted by ascending along-line distance before being handed to the plotting stage. This sort ensures that any optional connecting line (Join Line) traces a coherent path from one end of the scan line to the other rather than jumping between spatially unrelated points.

6.4 From-Line Aggregation

The perpendicular-distance sweep operates in two independent modes. For trace-level output, segments are grouped by their Trace identifier (using a pandas groupby), the length-weighted centroid of each group is computed as in Section 5.4, and the signed distance of that centroid from the reference line is evaluated once per trace. For segment-level output (length or angle), the signed distance is evaluated directly at every segment's own midpoint, without any grouping.

6.5 Circular-Scan Intensity/Density Sweep

This is the most computationally intensive routine in the module. For each of the N scan-circle centres (Section 5.5), every fracture segment in the dataset is tested for endpoint containment (a simple Euclidean-distance-to-centre comparison against the radius) and, where at least one endpoint is outside the circle, for a full segment–circle boundary intersection via the quadratic of Section 5.6. Segments with both endpoints inside the circle are counted as endpoint-only contributions (no boundary crossing is possible for a fully interior segment). This is an O(N × M) procedure, where M is the number of fracture segments in the dataset, and is the dominant computational cost of the module for large fracture networks (see Section 11).

python 
def _graphs_intensity_density_along(self, df, line):
    """Utilizes circular scan windows along the line to calculate P21 & P20."""
    centres, radius, distances = self._graphs_scan_circles(line)
    segs = df[["X1", "Y1", "X2", "Y2"]].to_numpy(dtype=float)
    intensity, density = [], []

    for cx, cy in centres:
        n_intersections = 0
        n_endpoints = 0

        for x1, y1, x2, y2 in segs:
            inside1 = math.hypot(x1 - cx, y1 - cy) < radius
            inside2 = math.hypot(x2 - cx, y2 - cy) < radius
            n_endpoints += int(inside1) + int(inside2)

            if inside1 and inside2:
                continue
            if inside1 or inside2:
                n_intersections += 1
            else:
                n_intersections += self._graphs_segment_circle_intersections(
                    (x1, y1, x2, y2), (cx, cy), radius)

        intensity.append(n_intersections / max(4.0 * radius, 1e-12))
        density.append(n_endpoints / max(2.0 * math.pi * radius, 1e-12))

    return distances, intensity, density

Simplified excerpt of _graphs_intensity_density_along(), showing the per-circle endpoint-containment test and the fallback to the algebraic segment–circle intersection routine for partially interior segments.

6.6 Interactive Endpoint Picking State Machine

The "Select end points" workflow is implemented as a small state machine attached to mouse-click events on the map canvas. Activating the button sets an internal picking flag and clears any previously buffered click, changes the map cursor to a crosshair, and updates the status label to request the first click. The first accepted click stores its scene coordinates, draws a temporary marker and label directly onto the map axis, and updates the status label to request the second click. The second accepted click completes the pair: both points are written into the four coordinate StringVars, picking mode is deactivated, the cursor is restored, and a full redraw is triggered automatically. Clicks that land outside the map subplot (event.inaxes mismatch) or that occur while picking mode is inactive are ignored.

7. Internal Workflow

The end-to-end data flow through the Profile Cross-section module, from user interaction to rendered output, proceeds through the following ordered stages on every redraw request:

  1. Trigger — a control change (checkbox toggle, Enter/FocusOut on a numeric field, colour selection, map click completing a pair, or the Update Graphs button) invokes the module's central draw routine.
  2. Axis reset — both the map axis and the output-plot axis are cleared and their background colours re-applied from the current colour-picker state.
  3. Data availability check — the shared self.results table is inspected; an empty or missing table, or an Excel/CSV source lacking real spatial coordinates, produces an explanatory empty-state message and the workflow halts.
  4. Default-line seeding — if no coordinates have ever been entered, the bounding-box diagonal default line (Section 6.2) is computed and written into the coordinate fields.
  5. Line resolution — the four coordinate fields are parsed and validated (Section 6.1); failure again halts the workflow with an explanatory message.
  6. Map rendering — the full fracture-trace network is drawn as a LineCollection, the reference line is drawn on top in the selected scan-line colour, endpoint markers are added, and — if the intensity/density checkboxes are active — every scan-circle outline is drawn faintly for visual reference. The optional calibrated background photograph is drawn beneath all vector content.
  7. Output collection — each enabled checkbox invokes its corresponding computation routine (Sections 6.3–6.5); results are assembled into a list of (label, x-values, y-values, colour, marker) tuples.
  8. Output rendering — each collected series is scatter-plotted on the output axis in its assigned colour and marker shape; the intensity/density series are always also line-connected (since they represent a continuous function), while other series are connected only if Join Line is enabled — in which case points are first sorted by X to guarantee a directionally coherent line.
  9. Cosmetic finishing — titles (custom or automatic), axis labels using the active display unit, legends at their configured positions, grid lines, and axis-spine styling are applied to both subplots.
  10. Canvas draw — both subplots are redrawn to the screen via the Matplotlib TkAgg canvas's draw_idle() call, and the parent Figure's tight_layout() is invoked to avoid label clipping.

8. Statistical Methods and Quantitative Outputs

Unlike several other Static Analyzer tabs (Histogram, CDF, Rose Plot), the Profile Cross-section module does not fit theoretical statistical distributions; its statistical content is instead the raw sampled series described in Section 5, together with the derived P21/P20 areal estimators. The following table summarises every output series that can appear on the right-hand plot, together with its physical units and the geological question it is designed to answer.

9. Visualization Components

9.1 Reference-Line Map (Left Subplot)

The map subplot renders the complete fracture-trace network as a LineCollection in the user-selected trace colour, the reference line as a bold coloured segment in the scan-line colour, and square markers at Point 1 (cyan-blue, #0891B2) and Point 2 (green, #16A34A) labelled accordingly in the legend. When either intensity or density is enabled, every scan-circle boundary is drawn as a thin, low-opacity red outline so the user can visually confirm the spatial extent each sample window covers relative to the fracture network. Axis limits are padded by five percent of the data range on each side (with a minimum one-unit pad) so traces are never clipped at the figure edge, and the aspect ratio is fixed to "equal" so that geometric distances and angles are not visually distorted.

9.2 Output Plot (Right Subplot)

The output subplot is a Cartesian scatter plot with an optional connecting line per series, using Matplotlib's automatic colour-differentiated markers (circle, square, triangle, and diamond markers are assigned per series in the implementation) so that multiple simultaneously enabled series remain visually distinguishable. The horizontal axis is always labelled "Distance along/from line (unit)"; the vertical axis is deliberately left as a generic "Value" label, since the module may display several physically distinct quantities (length in display units; angle in degrees; intensity in unit⁻¹; density in unit⁻²) simultaneously on a shared axis — the legend, not the axis label, is the authoritative key to each series' physical meaning in that situation.

9.3 Legends and Titles

The map and output-plot legends are positioned independently via their respective dropdown controls. "Outside Right" places the legend just beyond the right edge of the axis using a bbox_to_anchor offset; "Outside Bottom" places it below the axis in a multi-column row. Both subplots' titles default to descriptive automatic text ("Selected Scan Line" and "Selected Graphs" respectively) but are overridden together by the single Custom Title field when non-empty.

10. Advanced and Interactive Features

10.1 Interactive On-Map Line Digitizing

Rather than requiring manual coordinate entry, the user may click the "Select end points" button and then click twice directly on the fracture map to define the reference line visually. The first click is rendered immediately as a labelled marker ("Point 1") so the user receives instant visual confirmation; the second click completes the line, writes both coordinate pairs into the numeric entry fields (so they remain editable afterward), and triggers a full recalculation. This workflow is particularly useful for aligning the reference line with a specific geological feature visible on a loaded background photograph (see Section 10.2), such as a fault trace or bedding contact, without needing to first read off its coordinates.

10.2 Calibrated Background Photograph Overlay

The Settings tab's Background Image sub-section allows a real-world photograph — typically the same calibrated image used to establish the map's pixel-to-real-world scale via the Scale Calibration tool — to be displayed beneath the fracture-trace network and reference line. The module automatically searches for a suitable image in the following priority order: a path explicitly already stored for this tab, the Fracture Map tab's own calibrated background image path, the Trace Map tool's saved sidecar PNG file (co-located with the source SVG), and whichever background image has already been shared to the Dynamic Analyzer workspace via the launcher's cross-workspace shared state. This means a background image only ever needs to be uploaded once, in any single workspace, for it to become available in the Profile Cross-section map as well.

The overlay respects the same real-world bounding-box extent as the fracture network and is drawn with origin="upper" and an explicit extent mapping so that image row order correctly matches the (frequently inverted) Y-axis convention used by SVG-derived fracture maps. Opacity is independently adjustable via the Image opacity field, allowing the fracture traces to remain legible against a bright or high-contrast photograph.

10.3 Move Elements / Free Layout Mode

Like every other plot-bearing tab in the Static Analyzer, the Profile Cross-section map and plot participate in the workspace-wide "Move Elements" editing mode (toggled from the shared header toolbar). When active, individual Matplotlib axes on the canvas may be freely repositioned by click-and-drag, which is useful when preparing a combined figure for external publication or reporting, since the relative sizing of the map versus the output plot can be adjusted without leaving the application.

10.4 Save Image Export

The Save Image button exports the full two-panel Figure — map and output plot together, with legends, titles, and any manual layout adjustments included — as a PNG raster at the application-wide configured export resolution (Settings page "Plot export DPI", default 300 dpi). The exported filename is automatically constructed from the current project name, a fixed "static_graphs" tag, and a timestamp, and is written into the project's FracSTAT_plot_exports subfolder.

11. Performance Optimization

The along-line intersection sweep (Section 6.3) and the from-line aggregation (Section 6.4) are both O(M) in the number of fracture segments M, since each segment is tested against a single fixed reference line exactly once; these therefore scale comfortably even for networks containing many thousands of segments.

The circular-scan intensity/density sweep (Section 6.5) is inherently O(N × M), because every one of the N scan circles must be tested against every one of the M segments independently — there is no shared geometric pre-filtering between different circles in the current implementation. Because N (the scan-circle count) is user-controlled and bounded by the interface to at most 200, and because this calculation is only performed at all when the intensity or density checkboxes are actively enabled, the practical cost is kept manageable: with a typical N of 12 and networks of a few thousand segments, the calculation completes essentially instantaneously; very large fracture datasets combined with a high scan-circle count will scale linearly in both factors, and the user is advised to reduce the scan-circle count if redraw responsiveness becomes noticeable on very dense datasets.

All heavy geometric loops in this module are implemented at the Python level (rather than being vectorised with NumPy), unlike the analogous circular-scanline computation in the Intensity & Density tab, which is fully vectorised for its grid-based Mauldon calculation. Consequently, the Profile Cross-section module's intensity/density feature is best used as a targeted, along-transect diagnostic rather than as a replacement for the grid-based global P21/P20 mapping already provided by the dedicated Intensity & Density tab.

Note: redraws that do not touch the fracture-segment geometry itself — for example, changing a colour, grid width, or legend position — are inexpensive, since no recomputation of intersections, midpoints, or scan-circle sweeps is required; only the already-collected numeric series need to be re-rendered.

12. Practical Examples

12.1 Worked Example — Along-Line Intensity Profile Across a Fault Damage Zone

Scenario: a fracture-trace SVG has been digitized from an outcrop photograph spanning a mapped fault trace, with the fault oriented roughly northeast–southwest across the centre of the image.

  1. Load the SVG file via the workspace header's Load SVG button, and confirm the profile scale using the Scale tool so that real-world distances are correctly calibrated.
  2. Open the Profile Cross-section tab. A default diagonal reference line appears automatically across the full mapped extent.
  3. In the Info tab, click "Select end points", then click once at the outcrop edge on one side of the fault and once on the opposite edge, so that the new reference line crosses the fault trace roughly perpendicular to its strike.
  4. Enable "Estimated intensity, P21" and set "Number of scan circles" to a value such as 20 for finer spatial resolution along the transect.
  5. Inspect the resulting P21 profile in the output plot: a pronounced local maximum in P21 near the along-line position corresponding to the fault trace is diagnostic of a fracture damage zone, and the profile's decay away from that position can be used to estimate the damage-zone half-width.
  6. Optionally enable "Segment angle along" simultaneously to check whether fracture orientation also changes systematically within the damage zone (e.g. rotation toward parallelism with the fault as the fault core is approached).

12.2 Worked Example — Bedding-Perpendicular Asymmetry Check

Scenario: the user suspects that fracture length differs systematically above versus below a particular bedding contact visible in the mapped image.

  1. Set the reference line to run exactly along the bedding contact, either by typing its two endpoint coordinates directly or by clicking along it on a calibrated background photograph.
  2. Enable "Trace length from" in the Distance FROM the line card.
  3. Read the resulting scatter plot: points with positive perpendicular distance lie on one side of the contact, points with negative distance on the other; a visible asymmetry in the spread or central tendency of trace length between the positive and negative halves of the plot supports a genuine lithological or mechanical-stratigraphic control on fracture length.

13. Geological Interpretation

The Profile Cross-section module reproduces, in digital form, the classical field scanline survey that underpins much of quantitative structural-geology fracture characterisation. Its outputs support several standard interpretive workflows:

  • Damage-zone characterisation — a local increase in P21/P20 intensity and/or a systematic rotation of fracture strike toward the along-line position of a mapped fault is the standard diagnostic signature of a fault damage zone, and the half-width of the elevated-intensity interval provides a first-order estimate of damage-zone width for comparison with regional fault-scaling relationships.
  • Stratigraphic (bed-thickness) control on fracture length — fracture length is frequently observed to correlate with mechanical layer thickness in layered sedimentary sequences. Running the reference line parallel to bedding and inspecting the "Segment/Trace length from" plots on either side of a bed boundary can reveal whether fracture populations terminate systematically at that boundary, consistent with bed-confined ("layer-bound") fracturing.
  • Anisotropy-independent intensity estimation — because straight-scanline (P10) sampling is well known to be biased against fracture sets sub-parallel to the scanline, the circular-window P21/P20 estimator implemented here removes this orientation bias, making the reported intensity/density profile more representative of the true areal fracture population than a simple count of scanline crossings would be.
  • Spatial-trend screening — because the reference line can be placed anywhere and reoriented freely, the module functions as a rapid, exploratory tool for screening a mapped area for spatial trends in any measured fracture attribute before committing to a more elaborate grid-based analysis in the dedicated Intensity & Density tab.
Note: practitioners should bear in mind that all statistics computed by this module are two-dimensional trace-map projections of what is, in reality, a three-dimensional fracture network; length, spacing, and intensity measured on an exposed outcrop face or planform map are subject to the standard suite of sampling biases (truncation, censoring, and orientation bias) recognised in the structural-geology literature, and the module's numerical outputs should always be interpreted alongside the geological context of the mapped surface.

14. Implementation Notes

The module is implemented as the StaticGraphsTab Python mixin class, combined into StaticAnalyzerApp via multiple inheritance together with the other Static Analyzer tab mixins. All interactive controls are built with CustomTkinter widgets (CTkCheckBox, CTkEntry, CTkOptionMenu, CTkButton) bound directly to Tkinter-style Variable objects, so that reading a control's current value anywhere in the drawing routine is simply a matter of calling .get() on the corresponding variable; every control that should trigger a redraw is wired either to its variable-change command callback or to a bound Return/FocusOut keyboard event.

Colour selections are handled by a shared _StaticColorButton widget (used identically across all Static Analyzer tabs) that opens a small swatch-and-hex-entry dialog and invokes a callback with the chosen hex string; this callback both updates the corresponding Python attribute and immediately triggers a redraw, so colour changes are reflected live without a separate "Apply" step.

The module maintains no persistent computed-result cache between redraws; because both the along/from sweeps and the circular-scan sweep are computed freshly from self.results on every redraw, the displayed output is always guaranteed to be consistent with whatever fracture dataset and profile-scale calibration are currently active elsewhere in the Static Analyzer, at the cost of recomputation on every appearance-only change (mitigated, as noted in Section 11, by the low cost of the along/from sweeps).

Background-image loading is itself cached at the instance level (a simple path-keyed cache holding the most recently loaded and decoded image array), so that repeated redraws while only a cosmetic setting (such as opacity) changes do not repeatedly re-read and re-decode the same image file from disk.

15. Limitations

LimitationDescription
Spatial data onlyThe module operates only on genuinely spatial data (SVG traces, or Excel/CSV files carrying real X1/Y1/X2/Y2 coordinates); orientation-only spreadsheet formats (legacy Strike/Dip fracture-plane tables, Trend/Plunge pole tables) cannot be profiled, since they carry no real map position.
Unvectorised circular-scan loopThe circular-scan intensity/density computation is an unvectorised, per-segment Python loop and therefore does not scale as efficiently as the equivalent grid-based Mauldon calculation in the dedicated Intensity & Density tab; very large datasets combined with a high scan-circle count may notice a perceptible delay.
Infinite-line "distance from"The "distance from" calculation treats the reference line as infinite in both directions, so points far outside the finite segment between the two chosen endpoints still contribute a perpendicular-distance value; this is by design (consistent with treating the line as an axis of reference) but should not be mistaken for a bounded corridor sample.
No formal statistical inferenceNo confidence intervals, uncertainty bounds, or theoretical-distribution fitting are provided for any series in this module; all outputs are raw or directly derived geometric measurements. Users requiring rigorous statistical inference on length or intensity distributions should additionally consult the CDF tab's maximum-likelihood fitting tools.
Planar assumptionThe module assumes a two-dimensional, effectively planar fracture-trace map; no correction is applied for outcrop-surface curvature, oblique viewing angle, or true 3-D fracture orientation beyond what is already encoded in the loaded strike values.
